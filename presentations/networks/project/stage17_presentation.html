בדיקות
איך מוכיחים שVaultGuard עובד?

בנינו שרת, הצפנה, ניטור, ממשק. עד כה בדקנו "בעין" - הרצנו ידנית וראינו שזה עובד. בשלב הזה אנחנו בונים בדיקות אוטומטיות שמוכיחות את זה בלחיצת כפתור, כל פעם מחדש.

למה "זה עבד לי פעם אחת" לא מספיק?

  • שיניתם קוד בשלב 12 (ניטור) — איך תדעו שלא שברתם את ההתחברות משלב 3?
  • בהגנה בעל פה, המורה ישנה קלט (סיסמה שגויה, קובץ פגום) — אם לא בדקתם, לא תדעו מה קורה
  • "זה עבד אצלי במחשב" — לא מוכיח שזה יעבוד עם 3 לקוחות במקביל, או עם payload לא תקין

בדיקה אוטומטית = קוד שמרים סוגיה ספציפית, מריץ אותה, ומשווה תוצאה בפועל לתוצאה צפויה — בלי שאתם צריכים לבדוק בעין כל פעם.

ארבעה סוגי בדיקות

🧩

UNIT בדיקת יחידה

בודקת פונקציה אחת בבידוד מהשאר. לדוגמה: encrypt() מחזירה טקסט מוצפן שונה מהמקור.

🔗

INTEGRATION בדיקת אינטגרציה

בודקת שכמה חלקים עובדים נכון יחד. לדוגמה: לקוח אמיתי מתחבר לשרת אמיתי, השרת שומר ומגיב.

📈

LOAD בדיקת עומס בסיסית

בודקת מה קורה כשיש כמה לקוחות במקביל — לא קורס, לא "תופס" אחד ומשאיר אחרים תקועים.

🚫

NEGATIVE בדיקה שלילית

בודקת שקלט פגום/זדוני לא מקריס את המערכת, אלא מטופל בצורה מסודרת (שגיאה, לא Exception לא מטופל).

pytest: מבנה בסיסי

בדיקה ב-pytest היא פונקציה ששמה מתחיל ב-test_, ובסופה assert שמשווה תוצאה בפועל לתוצאה צפויה.

def encrypt(text):
    return "".join(chr(ord(c) + 1) for c in text)

def test_encrypt_changes_text():
    """בדיקת יחידה: ההצפנה לא מחזירה את הטקסט המקורי"""
    result = encrypt("hello")
    assert result != "hello"

def test_encrypt_same_length():
    assert len(encrypt("hello")) == len("hello")

הרצה: pytest test_encrypt.py -v — pytest מאתר את כל פונקציות ה-test_ ומדפיס PASSED/FAILED לכל אחת.

בדיקה שלילית: pytest.raises

לא בודקים רק "מה קורה כשהכל תקין" — גם "מה קורה כשהקלט פגום, ושהשרת לא מקבל אותו".

import pytest

def parse_message(raw):
    if ":" not in raw:
        raise ValueError("הודעה לא תקינה - חסר מפריד")
    return raw.split(":", 1)

def test_invalid_message_raises_error():
    """בדיקה שלילית: הודעה לא תקינה מחזירה שגיאה, לא קורסת בשקט"""
    with pytest.raises(ValueError):
        parse_message("בלי מפריד בכלל")

חוסר בדיקה שלילית = הסיכון האמיתי: קוד שעובד מצוין עם קלט תקין, אבל קורס (Exception לא מטופל) ברגע שמישהו שולח קלט מוזר — וזה בדיוק מה שתוקף עושה.

Fixture: מקימים ומפילים שרת אמיתי לבדיקה

fixture מכין משאב לפני הבדיקה (כאן: מריץ את שרת ה-VaultGuard ב-thread נפרד), ומסיר אותו אחרי (סוגר את ה-socket) — כדי שכל בדיקה תתחיל ממצב נקי.

import pytest, threading, time, socket

@pytest.fixture
def running_server():
    # מריץ את שרת הפרויקט ב-thread נפרד לפני הבדיקה
    server_thread = threading.Thread(target=start_vaultguard_server, args=("localhost", 9999))
    server_thread.daemon = True
    server_thread.start()
    time.sleep(0.3)  # זמן עלייה
    yield ("localhost", 9999)
    # כאן אפשר לסגור חיבורים אחרי הבדיקה
def test_client_connects_and_server_logs_it(running_server):
    """בדיקת אינטגרציה: לקוח מתחבר, השרת מזהה ושומר"""
    host, port = running_server
    client_socket = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
    client_socket.connect((host, port))
    assert client_socket is not None
    client_socket.close()

בדיקת עומס בסיסית: כמה לקוחות במקביל

def test_three_clients_handled_concurrently(running_server):
    """בדיקת עומס: 3 לקוחות במקביל - כולם מטופלים"""
    host, port = running_server
    sockets = []
    for i in range(3):
        s = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
        s.connect((host, port))
        sockets.append(s)

    for s in sockets:
        s.send(b"PING")
        response = s.recv(1024)
        assert response == b"PONG"  # כל לקוח קיבל מענה, אף אחד לא "נשכח"

    for s in sockets:
        s.close()

זו לא בדיקת עומס "אמיתית" (אלפי משתמשים) — אבל היא חושפת בעיות נפוצות: שרת שמטפל בלקוח אחד בלבד ומשאיר אחרים תקועים.
