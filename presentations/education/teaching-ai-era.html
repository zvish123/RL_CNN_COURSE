🎤 אסטרטגיה 3: "הסבר לי את הקוד"

כלל הזהב

אם התלמיד לא יכול להסביר את הקוד שורה שורה, הוא לא קיבל ציון.

תהליך הבדיקה:

  1. הסבר כללי: "מה הקוד הזה עושה?" (ברמה גבוהה)
  2. הסבר מפורט: "הסבר את השורה הזו במילים שלך"
  3. שאלות "מה אם": "מה יקרה אם נשנה את התנאי הזה?"
  4. test cases: "תן דוגמה לקלט שיפעיל את הענף הזה בקוד"
  5. edge cases: "מה יקרה עם רשימה ריקה?"

תרגיל: בדיקת הבנה - Bubble Sort

def bubble_sort(arr):
    n = len(arr)
    for i in range(n):
        for j in range(0, n-i-1):
            if arr[j] > arr[j+1]:
                arr[j], arr[j+1] = arr[j+1], arr[j]
    return arr

שאלות לתלמיד:

  • מה התפקיד של הלולאה החיצונית (עם i)?
  • למה הלולאה הפנימית מסתיימת ב-n-i-1 ולא ב-n?
  • מה קורה בשורה arr[j], arr[j+1] = arr[j+1], arr[j]?
  • מה יקרה אם נריץ את הפונקציה על [1, 2, 3]?
  • מה יקרה עם רשימה ריקה []?
  • איזו מורכבות זמן יש לאלגוריתם? למה?
אם התלמיד לא יכול לענות על השאלות האלה - הוא חוזר לשלב התכנון. אין "אני לא זוכר" או "זה מה שה-AI נתן".

טכניקות נוספות:

  • Rubber Duck Debugging: התלמיד מסביר את הקוד לחפץ דומם (או לחבר)
  • Code Review בזוגות: תלמיד אחד מסביר את הקוד שלו לשני
  • הקלטת וידאו: התלמיד מקליט את עצמו מסביר את הקוד (מעולה לתיק עבודות)

🔍 אסטרטגיה 4: Reverse Engineering

הרעיון

במקום לבקש מהתלמידים לכתוב קוד מאפס, תנו להם קוד מוכן עם בעיות מכוונות. זה מחייב קריאה מעמיקה והבנה של הקוד.

סוגי תרגילים:

1. מציאת Bugs מכוונים

תנו קוד עם 3-5 שגיאות לוגיות ודרשו מהתלמידים למצוא ולתקן:

# קוד עם bugs - מצאו את הבעיות!
def calculate_average(numbers):
    total = 0
    for num in numbers:
        total = total + num
    average = total / len(numbers)  # Bug 1: מה אם הרשימה ריקה?
    return average

def is_prime(n):
    if n < 2:  # Bug 2: מה עם 2?
        return False
    for i in range(2, n):  # Bug 3: לא יעיל! צריך עד שורש n
        if n % i == 0:
            return True  # Bug 4: צריך להחזיר False!
    return False  # Bug 5: צריך True!

הבאגים שצריך למצוא:

  1. חלוקה באפס אם הרשימה ריקה
  2. המספר 2 הוא ראשוני אבל מוחזר False
  3. אין צורך לבדוק עד n, מספיק עד sqrt(n)
  4. מחזיר True כשנמצא מחלק (צריך False)
  5. מחזיר False כשלא נמצא מחלק (צריך True)

2. שיפור קוד לא יעיל

תנו קוד שעובד אבל לא יעיל, דרשו אופטימיזציה:

# הקוד הזה עובד אבל לא יעיל - שפרו אותו!
def find_duplicates(lst):
    duplicates = []
    for i in range(len(lst)):
        for j in range(i+1, len(lst)):
            if lst[i] == lst[j]:
                if lst[i] not in duplicates:
                    duplicates.append(lst[i])
    return duplicates

# מורכבות: O(n³)
# שפרו ל-O(n) עם set!

פתרון מיטבי:

def find_duplicates(lst):
    seen = set()
    duplicates = set()
    for item in lst:
        if item in seen:
            duplicates.add(item)
        else:
            seen.add(item)
    return list(duplicates)

✍️ אסטרטגיה 6: תרגול בזמן אמת ללא AI

def is_palindrome(s):
    s = s.lower()
    reversed_s = s[::-1]
    if s == reversed_s:
        return True
    return False

# גרסה קצרה (1 שורה):
def is_palindrome(s):
    return s.lower() == s[::-1].lower()

💼 אסטרטגיה 7: בניית תיק עבודות אישי

תיק עבודות מקצועי מעיד על התפתחות התלמיד, מעודד ownership על הקוד, ומכין אותו לתעשייה (חובה להשכלה גבוהה ולחיפוש עבודה!).

מרכיבי תיק העבודות:

  • Commit messages ברורים - לא "update", אלא "Add input validation to login function"

🎯 טכניקות Prompting מתקדמות

ארגז הכלים למורה ולתלמיד

רשימת טכניקות שאילת שאלות יעילות ל-AI

קטגוריות של Prompts:

1️⃣ הבנה קונספטואלית

"הסבר לי את הקונספט של [נושא] כאילו אני תלמיד כיתה י'"

"מה ההבדל בין [מושג א'] ל-[מושג ב']? תן דוגמאות"

"למה משתמשים ב-[טכניקה] ולא ב-[טכניקה אחרת]?"

2️⃣ בדיקת הבנה (לא פתרון!)

"האם הגישה הזו לפתרון הבעיה נכונה? [מסביר את הגישה]"

"מה חסר בפסאודו-קוד שכתבתי? [מדביק פסאודו-קוד]"

"האם אני צריך לטפל במקרי קצה נוספים?"

3️⃣ Debugging

"הקוד שלי מחזיר [תוצאה לא נכונה] במקרה הזה: [דוגמה]. למה?"

"יש לי שגיאה: [הודעת שגיאה]. מה המשמעות שלה?"

"איך אני יכול לבדוק אם הפונקציה הזו עובדת נכון?"

4️⃣ Code Review

"בדוק את הקוד שלי למקרי קצה: [קוד]"

"איך אני יכול לשפר את הביצועים של הקוד הזה?"

"האם הקוד שלי עומד ב-best practices?"

5️⃣ למידת כלים

"מה עושה הפונקציה [שם פונקציה] ב-Python?"

"תן דוגמאות לשימוש ב-[ספרייה/מודול]"

"מתי כדאי להשתמש ב-[כלי] ומתי לא?"

⚠️ Prompts שאסור להשתמש בהם:

❌ "פתור לי את התרגיל: [שאלה]"

❌ "כתוב לי קוד ש-[תיאור]"

❌ "תעשה את המטלה שלי"

❌ "מה התשובה ל-[שאלה סגורה]"

תרגול Prompting - Workshop למורים

תרגיל 1: נסחו 3 prompts שונים לאותה בעיה:

הבעיה: תלמיד צריך לבנות פונקציה שבודקת אם מספר ראשוני

Prompt גרוע:

"כתוב פונקציה שבודקת אם מספר ראשוני"

Prompt 1 (הבנה):

"מה זה מספר ראשוני? מה האלגוריתם הבסיסי לבדיקה?"

Prompt 2 (בדיקה):

"אני חושב שצריך לבדוק חלוקה מ-2 עד n-1. זה נכון? יש דרך יעילה יותר?"

Prompt 3 (Review):

"כתבתי את הקוד הזה [קוד]. האם הוא מטפל בכל המקרים? מה עם 0, 1, 2?"
הכינו "כרטיס עזר Prompting" - מסמך A4 עם הנוסחאות הטובות שהתלמידים יכולים להדביק ליד המסך.

🎓 תרגילים ומשימות למורים

תרגילים מוכנים לשימוש בכיתה

תרגיל 1: "Debug המורה"

משך: 20 דקות | רמה: כיתה י'-י"א

מהלך:

  1. המורה כותב קוד על הלוח עם bug מכוון
  2. תלמידים מנסים למצוא את הבעיה (5 דק')
  3. דיון: איך זיהיתם? איך תתקנו?
  4. המורה מריץ את הקוד ומראה את השגיאה
# דוגמה - מה הבעיה כאן?
def calculate_factorial(n):
    result = 0  # Bug!
    for i in range(1, n+1):
        result = result * i
    return result

# אפשרות לדיון:
# 1. מה התוצאה של factorial(5)?
# 2. למה זה לא עובד?
# 3. איך תתקנו?

תרגיל 2: "Prompt Battle"

משך: 30 דקות | רמה: כל הרמות

מהלך:

  1. חלקו את הכיתה ל-4 קבוצות
  2. כל קבוצה מקבלת אותה בעיה
  3. כל קבוצה צריכה לנסח 3 prompts שונים
  4. מריצים את ה-prompts ב-ChatGPT/Claude
  5. משווים תוצאות - איזה prompt הצליח הכי טוב?
  6. דיון: למה prompt אחד טוב יותר?

דוגמת בעיה: "כתבו תוכנית שמנחשת מספר בין 1-100"

תרגיל 3: "Code Golf Championship"

משך: 40 דקות | רמה: י"א-י"ב

כללים:

  • כתבו פונקציה שעובדת במינימום שורות
  • אסור להשתמש ב-AI
  • הקוד חייב להיות קריא
  • כל שורת קוד = נקודה אחת (פחות = יותר טוב)

אתגרים לדוגמה:

  1. פלינדרום (קל)
  2. FizzBuzz (בינוני)
  3. ספירת מילים ייחודיות (קשה)

תרגיל 4: "תיק עבודות Mini"

משך: 90 דקות (2 שיעורים) | רמה: כל הרמות

מטרה: בניית פרויקט קטן עם documentation מלא

שלבים:

  1. שיעור 1 - תכנון (45 דק'):
    • בחירת פרויקט פשוט (מחשבון, משחק ניחושים, TODO list)
    • כתיבת README ראשוני
    • יצירת repository בגיטהאב
    • פסאודו-קוד
  2. שיעור 2 - מימוש (45 דק'):
    • כתיבת הקוד
    • Commits עם messages טובים
    • השלמת README
    • הצגה קצרה (3 דק' לתלמיד)

תרגיל 5: "Explain to a Rubber Duck"

משך: 15 דקות | רמה: כל הרמות

מהלך:

  1. כל תלמיד מביא חפץ מהבית (ברווז גומי, פלאפון, כל דבר)
  2. נותנים בעיה פשוטה
  3. התלמיד מסביר לחפץ בקול איך הוא יפתור
  4. המורה מקשיב ומתערב רק אם יש שגיאה לוגית
זו טכניקת Debugging מפורסמת בתעשייה! לעתים קרובות הסבר בקול רם חושף את הבעיה.

משימות בית

  • משימה 1: פתחו חשבון GitHub ושלחו לי את ה-username
  • משימה 2: העלו פרויקט אחד לפחות עם README מלא
  • משימה 3: עשו Code Review לפרויקט של חבר לכיתה
  • משימה 4: כתבו פוסט blog על "מה למדתי השבוע"
  • משימה 5: פתרו bug בקוד של חבר (אסור לבקש מ-AI!)
