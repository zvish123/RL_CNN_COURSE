💾 Cache - מטמון

  • ישנם דפי אינטרנט שאנחנו מבקשים שוב ושוב

import requests


class HTTPCache:
    def __init__(self):
        self.cache = {}

    def get_page(self, url):
        headers = {}

        # אם יש מידע במטמון – שלח בקשה מותנית
        if url in self.cache:
            cached = self.cache[url]

            if cached.get("etag"):
                headers["If-None-Match"] = cached["etag"]
            elif cached.get("last_modified"):
                headers["If-Modified-Since"] = cached["last_modified"]

            print(f"📡 בקשה מותנית ל־{url}")
        else:
            print(f"📥 בקשה ראשונה ל־{url}")

        response = requests.get(url, headers=headers)

        # שימוש במטמון
        if response.status_code == 304:
            print("✅ 304 Not Modified – שימוש במטמון\n")
            return self.cache[url]["content"]

        # עדכון מטמון
        print(f"🔄 {response.status_code} – עדכון מטמון\n")
        self.cache[url] = {
            "content": response.text,
            "etag": response.headers.get("ETag"),
            "last_modified": response.headers.get("Last-Modified")
        }

        return response.text


cache = HTTPCache()

url = "http://httpbin.org/cache/60"
url = "http://httpbin.org/etag/abc123"

page3 = cache.get_page(url)  # 304

❓ שאלות הבנה - Cache

שאלות להבנת החומר

1. מהי המטרה העיקרית של מנגנון ה-Cache?
תשובה: המטרה העיקרית היא חיסכון בזמן טעינה ובנפח נתונים. במקום להוריד שוב ושוב את אותו תוכן שלא השתנה, אנחנו משתמשים בעותק המקומי ששמרנו.
2. מה ההבדל בין בקשת GET רגילה לבין Conditional GET?
תשובה: בבקשת GET רגילה, השרת תמיד שולח את התוכן המלא. ב-Conditional GET, הלקוח שולח את תאריך העותק שיש לו (If-Modified-Since), והשרת שולח את התוכן רק אם הוא השתנה מאז. אם לא - מחזיר 304 Not Modified.
3. מה קורה כאשר השרת מחזיר קוד 304?
תשובה: קוד 304 (Not Modified) אומר שהתוכן לא השתנה מאז התאריך ש-הלקוח ציין. הלקוח יכול להשתמש בעותק המקומי שיש לו במטמון. התגובה לא כוללה את התוכן עצמו, מה שחוסך נתונים.

❓ שאלות הבנה - Cookies

שאלות להבנת החומר

1. מדוע פרוטוקול HTTP נחשב ל-Stateless ומה המשמעות של זה?
תשובה: HTTP הוא Stateless כי הוא לא שומר מידע על בקשות קודמות. כל בקשה מטופלת באופן עצמאי, ללא זיכרון של מה קרה קודם. המשמעות היא שהשרת לא יודע אם אותו לקוח ביקר אתמול, לפני דקה, או שזו ביקורו הראשון.
2. הסבירו את תהליך העבודה עם Cookies (שלב אחר שלב)
תשובה:
1. הלקוח שולח בקשה ראשונה לשרת
2. השרת מחזיר תגובה עם השדה Set-Cookie שמכיל מזהה ייחודי
3. הדפדפן שומר את ה-Cookie במחשב המקומי
4. בכל בקשה הבאה, הדפדפן שולח את ה-Cookie בשדה Cookie
5. השרת משתמש ב-Cookie כדי לזהות את המשתמש ולקשר בין הבקשות
3. מהם שלושה שימושים נפוצים ל-Cookies באתרי אינטרנט?
תשובה:
1. אימות ו-Session Management: שמירה על מצב התחברות של המשתמש
2. עגלת קניות: שמירת מוצרים שהמשתמש הוסיף לעגלה
3. העדפות אישיות: שמירת שפה, ערכת צבעים, הגדרות תצוגה
(אפשרויות נוספות: מעקב אנליטי, פרסומות ממוקדות, וכו')
4. מה ההבדל בין Cookie רגיל ל-HttpOnly Cookie?
תשובה: Cookie רגיל ניתן לגישה דרך JavaScript (document.cookie), מה שעלול להיות סיכון אבטחה. HttpOnly Cookie לא נגיש ל-JavaScript ונשלח רק בבקשות HTTP, מה שמקשה על תוקפים לגנוב אותו באמצעות XSS (Cross-Site Scripting).

🔐 Authentication - אימות וזיהוי

מושגי יסוד

מושג הגדרה דוגמה
זיהוי (Identification) לדעת מי המשתמש שם מ שתמש: "student123"
אימות (Authentication) לוודא שהמשתמש הוא באמת מי שהוא אומר שהוא בדיקת סיסמה, טביעת אצבע
הרשאה (Authorization) מה המשתמש מורשה לעשות תלמיד יכול לראות ציונים, מורה יכול לערוך

🔒 למה צריך Authentication?

  • 📧 Gmail - הגנה על הודעות האימייל שלכם
  • 👥 Facebook - גישה לפרופיל אישי
  • 🏦 בנקאות - הגנה על חשבון הבנק
  • 🎓 מערכות לימוד - גישה לציונים וחומרי לימוד
⚠️ חשוב לדעת: פרוטוקול HTTP כולל מנגנון אותנטיקציה בסיסי (Basic Authentication) שהוא קל יחסית לפריצה. לכן פותחו מנגנונים מתקדמים יותר כמו OAuth, JWT, ו-2FA.

תרגיל 4.13: פענוח סיסמה מוצפנת

import base64

def decode_basic_auth(auth_header):
    """
    פענוח Authorization header של Basic Authentication
    
    Args:
        auth_header: המחרוזת מהשדה Authorization
        דוגמה: "Basic dXNlcjpwYXNzd29yZA=="
    
    Returns:
        tuple: (username, password)
    """
    # הסרת המילה "Basic "
    if auth_header.startswith('Basic '):
        encoded = auth_header[6:]  # מדלג על "Basic "
    else:
        encoded = auth_header
    
    # פענוח Base64
    decoded_bytes = base64.b64decode(encoded)
    decoded_str = decoded_bytes.decode('utf-8')
    
    # הפרדה לשם משתמש וסיסמה
    username, password = decoded_str.split(':', 1)
    
    return username, password

# דוגמת שימוש
auth_header = "Basic c3R1ZGVudDpjeWJlcjIwMjU="
username, password = decode_basic_auth(auth_header)

print(f"שם משתמש: {username}")
print(f"סיסמה: {password}")

# יצירת Authorization header
def create_basic_auth(username, password):
    """יצירת Authorization header"""
    credentials = f"{username}:{password}"
    encoded = base64.b64encode(credentials.encode('utf-8')).decode('utf-8')
    return f"Basic {encoded}"

auth = create_basic_auth('student', 'cyber2025')
print(f"\nAuthorization Header: {auth}")

❓ שאלות הבנה - Authentication

שאלות להבנת החומר

1. מה ההבדל בין זיהוי (Identification) לבין אימות (Authentication)?
תשובה: זיהוי הוא התהליך של הצהרת זהות - "אני student123". אימות הוא התהליך של הוכחת הזהות - "הנה הסיסמה שמוכיחה שאני באמת student123". ניתן לחשוב על זה כמו תעודת זהות: השם הוא הזיהוי, התמונה והחתימה הם האימות.
2. מדוע HTTP Basic Authentication נחשב לא מאובטח?
תשובה: Base64 הוא קידוד ולא הצפנה - כל אחד יכול לפענח אותו בקלות. הסיסמה נשלחת בכל בקשה בפורמט שניתן לפענח. אם מישהו מאזין לתעבורה (Sniffing), הוא יכול לגנוב את הסיסמה. לכן חובה להשתמש ב-HTTPS שמצפין את כל התקשורת, או במנגנוני אימות מתקדמים יותר.
3. מה המשמעות של קוד התשובה 401 Unauthorized?
תשובה: קוד 401 אומר שהמשתמש לא התחבר או שהאימות נכשל. השרת דורש אימות ומחזיר גם את השדה WWW-Authenticate שמציין איזה סוג אימות נדרש (למשל Basic). הדפדפן מציג חלון התחברות למשתמש.
4. מהם שלושה מנגנוני אימות מתקדמים יותר מ-Basic Authentication?
תשובה:
1. OAuth 2.0: פרוטוקול הרשאה שמאפשר לאפליקציות גישה מוגבלת למשאבים
2. JWT (JSON Web Tokens): טוקנים מוצפנים שמכילים מידע על המשתמש
3. Two-Factor Authentication (2FA): אימות דו-שלבי (סיסמה + קוד מהטלפון)
(נוספים: אימות ביומטרי, certif icates, SAML)

💡 שאלות יכולת יישום

שאלות מתקדמות

1. תארו מצב בו Cache עלול לגרום לבעיה במקום לעזור. איך אפשר למנוע את הבעיה?
תשובה: דוגמה: אתר חדשות שמתעדכן כל הזמן. אם הדפדפן שומר את העמוד הראשי במטמון לזמן רב, המשתמש לא יראה חדשות עדכניות.

פתרונות:
• הגדרת Cache-Control: max-age=0 (אל תשמור במטמון)
• שימוש ב-Conditional GET עם If-Modified-Since
• הוספת version/timestamp לכתובת הקובץ (style.css?v=1.2.3)
• שימוש בכותרת Expires עם תאריך בעבר
2. איך Cookie עוזר לממש עגלת קניות באתר מסחר? תארו את התהליך המלא.
תשובה:
1. משתמש נכנס לאתר - השרת יוצר session_id ייחודי ושולח כ-Cookie
2. במסד הנתונים, השרת יוצר רשומה ל-session_id עם עגלה ריקה
3. כשמשתמש מוסיף מוצר - שולח בקשה עם ה-Cookie
4. השרת מזהה את המשתמש לפי ה-Cookie ומעדכן את העגלה במסד הנתונים
5. בכל עמוד, השרת יכול לטעון את תוכן העגלה לפי ה-session_id
6. גם אם המשתמש סוגר את הדפדפן - ה-Cookie נשמר והעגלה נשארת!
3. האם אפשר לשלב Cookies ו-Authentication? תנו דוגמה לשימוש משולב.
תשובה: כן! זה בדיוק מה שקורה ברוב האתרים:

התהליך:
1. משתמש מתחבר עם שם משתמש וסיסמה (Authentication)
2. אחרי אימות מוצלח, השרת יוצר session_id ושולח כ-Cookie
3. בכל בקשה הבאה, הדפדפן שולח את ה-Cookie
4. השרת בודק את ה-Cookie כדי לדעת מי המשתמש (בלי לבקש סיסמה שוב!)

דוגמה: Facebook - מתחברים פעם אחת, ואז ה-Cookie שומר את המצב המחובר למשך שבועות.
4. מדוע שילוב של HTTP (ללא S) ו-Cookies יכול להיות מסוכן?
תשובה: HTTP שולח הכל בטקסט פשוט ללא הצפנה. תוקף שמאזין לרשת (Man-in-the-Middle) יכול:

1. לגנוב Cookies: לראות את ה-session_id ולהתחזות למשתמש
2. Session Hijacking: להשתמש ב-Cookie הגנוב כדי לגשת לחשבון
3. Cookie Injection: להחדיר Cookies מזויפים

פתרון: תמיד להשתמש ב-HTTPS שמצפין את כל התקשורת! בנוסף, להגדיר Cookies עם הדגלים Secure (רק HTTPS) ו-HttpOnly (לא נגיש ל-JavaScript).

  • שימושים: התחברות, עגלת קניות, העדפות משתמש

  • זיהוי = מי אתה | אימות = הוכחת זהות
  • Basic Authentication - פשוט אבל לא מאובטח (Base64)
  • חשוב להשתמש ב-HTTPS!

🔑 נקודות מפתח לזכור

  • Cookies הם הבסיס לאפליקציות web מודרניות
  • אבטחה היא קריטית - תמיד HTTPS!
