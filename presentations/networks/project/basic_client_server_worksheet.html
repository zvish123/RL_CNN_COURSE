תזכורת תאורטיתחובה לקרוא לפני שמתחילים לכתוב קוד

socket הוא נקודת קצה של חיבור בין שני רכיבים. כדי שהוא ישמש להעברת מידע, צריך להגדיר את נקודות הקצה שלו: כתובת IP (איזה מחשב) ו-מספר Port (איזו תוכנה על אותו מחשב). חיבור TCP הוא זרם בתים (stream) — לא הודעות בדידות — מה שאומר שrecv בודד לא מבטיח קבלת "הודעה שלמה אחת".

פעולהמי קורא להתפקיד
socket()שרת + לקוחיוצר socket חדש (TCP: SOCK_STREAM)
bind()שרתקושר את ה-socket לכתובת IP ולפורט מקומיים
listen()שרתמתחיל להאזין לחיבורים נכנסים, וקובע כמה לקוחות יכולים להמתין בתור
accept()שרתחוסם (blocking) וממתין לחיבור לקוח; מחזיר socket חדש שמייצג את הלקוח הספציפי + הכתובת שלו
connect()לקוחמתחבר לשרת בכתובת IP+Port מסוימים
send()שרת + לקוחשולח בתים בצד השני של ה-socket
recv(n)שרת + לקוחחוסם וממתין לקבלת עד n בתים (יכול להחזיר פחות!)
close()שרת + לקוחסוגר את ה-socket ומשחרר את הפורט
socket()→bind()→listen()→accept()

כתבו את השרת הבסיסיקל

משימה 1.1 — server.py

שרת שמאזין, מקבל הודעת HELLO, ועונה WELCOME

השלימו את הקוד: שרת שנפתח על פורט 5050, מקבל לקוח אחד, קורא הודעה ממנו, ואם זו ההודעה HELLO — עונה WELCOME. כל הודעה אחרת — עונה UNKNOWN.

▶ קוד זה דורש socket אמיתי — להריץ ב-PyCharm/IDE מקומי, לא בדף זה
צ'ק-ליסט לבדיקה עצמית (לאחר הרצה):
  • השרת עולה בלי שגיאה ומדפיס "השרת מאזין על..."
  • אם מנסים להריץ שרת שני באותו פורט בלי לסגור את הראשון — מתקבלת שגיאת Address already in use (זה תקין ומלמד למה צריך close()!)
  • השרת ממתין (לא מסיים) עד שמתחבר לקוח — זו ההתנהגות הצפויה מ-accept() חוסם

משימה 2.1 — client.py

לקוח שמתחבר לשרת, שולח HELLO, ומדפיס את התשובה

▶ קוד זה דורש socket אמיתי — להריץ ב-PyCharm/IDE מקומי, לא בדף זה

צ'ק-ליסט לבדיקה עצמית:

  • הלקוח מדפיס "השרת ענה: WELCOME"
  • חלון השרת מדפיס "התחבר לקוח מ-(...)" ו"התקבל מהלקוח: HELLO"
3

תכנון פרוטוקול ראשוניהצעד הראשון לפני שמתרחבים לפרוטוקול המלא של הפרויקט

בשלב הזה הפרוטוקול הוא הכי פשוט שיכול להיות: הודעה אחת קבועה (HELLO) ותשובה אחת קבועה (WELCOME). זה לא הפרוטוקול הסופי של הפרויקט (כספת + ניטור) — אבל הוא מבסס את התשתית שעליה תבנו בהמשך (פקודות LOGIN, UPLOAD, DOWNLOAD ועוד).

משימה 3.1

מה הבעיה הבאה שצריך לפתור, ולמה?

כרגע ההודעה "HELLO" קצרה וקבועה — אין סיכוי שתיקטע ב-recv(1024). אבל בשלבים הבאים תרצו לשלוח תוכן קובץ שלם (יכול להיות אלפי בתים). מה הסיכון, ואיזה רעיון תציעו לפתור אותו?

משימה 3.2

הרחיבו את הפרוטוקול: הוסיפו הודעה שלישית

מלבד HELLO/WELCOME ו-UNKNOWN, הוסיפו הודעה אחת חדשה משלכם שתרצו שהשרת הזה הבסיסי יידע לטפל בה (לדוגמה: PING שמחזיר PONG, או BYE שמסיים את החיבור בצורה מסודרת). כתבו: שם ההודעה, והתשובה הצפויה.

משימה 3.3 — אתגר (לא חובה)

הוסיפו בפועל את ההודעה שתכננתם לקוד

חזרו ל-server.py ו-client.py שכתבתם, והוסיפו תמיכה בהודעה החדשה שהגדרתם במשימה 3.2.
