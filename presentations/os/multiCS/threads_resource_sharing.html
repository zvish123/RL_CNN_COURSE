01 / THREADS — הבסיס

מה זה Thread?

Thread = זרם ביצוע עצמאי בתוך אותו תהליך (process)

כל תהליך (process) בעל מרחב זיכרון משלו. אבל בתוך תהליך, אפשר ליצור מספר threads שחולקים את אותו זיכרון.

דוגמה מהחיים: מסעדה = process.
כל מלצר = thread. כולם עובדים באותו מטבח (זיכרון משותף).
אתגר: שני מלצרים לוקחים את הביצה האחרונה ב-זמן.
זה בדיוק Race Condition. 
# דוגמה — שלושה threads במקביל
import threading, time

def worker(name, delay):
    for i in range(3):
        time.sleep(delay)
        print(f"[{name}] step {i}")

# יוצרים 3 threads
t1 = threading.Thread(
    target=worker, args=("A", 0.1))
t2 = threading.Thread(
    target=worker, args=("B", 0.15))

t1.start(); t2.start()
t1.join();  t2.join()
# הפלט מעורבב — A ו-B רצים במקביל!

02 / RACE CONDITION — הבעיה

Race Condition — תחרות על משאב

מה קורה כש-2 threads קוראים ומשנים אותו משתנה בו-זמנית?

📦 דוגמה: מחסן מוצרים

שני עובדים (threads) בודקים את מלאי המחסן ורושמים מכירות בו-זמנית:

# ❌ קוד עם Race Condition
stock = 100   # מלאי משותף

def sell_items(worker_name, amount):
    global stock
    for _ in range(amount):
        # שלב 1: קרא ערך
        current = stock
        # ← Thread אחר עלול לשנות כאן!
        # שלב 2: חשב
        new_val = current - 1
        # שלב 3: כתוב בחזרה
        stock = new_val
⚠️ הבעיה: בין "קרא" ל"כתוב" — thread אחר יכול גם לקרוא ולכתוב.
שניהם קוראים stock=50, שניהם כותבים stock=49 — אבדה מכירה!

🔬 מה קורה בפועל

Thread A Thread B stock ───────────────────────────────────── read stock 50 compute 50-1=49 read stock 50 ← עדיין 50! write stock=49 49 compute 50-1=49 write stock=49 49 ← במקום 48! אבדה מכירה שלמה! stock=49 במקום 48
למה זה קורה? Python GIL מגן על בעיות memory כלליות, אבל לא על רצפים של read-modify-write!

03 / LOCK — הפתרון

threading.Lock — מנגנון ההגנה

Lock מבטיח שרק thread אחד נמצא ב-Critical Section בכל רגע נתון

# ✅ קוד עם Lock — בטוח
import threading

stock = 100
stock_lock = threading.Lock()  # יוצרים Lock

def sell_items_safe(worker, amount):
    global stock
    for _ in range(amount):
        # ← Lock: רק thread אחד נכנס
        with stock_lock:
            current = stock
            stock = current - 1
        # ← Lock שוחרר אוטומטית

# שני threads — stock יהיה בדיוק 80
t1 = threading.Thread(
    target=sell_items_safe, args=("A", 10))
t2 = threading.Thread(
    target=sell_items_safe, args=("B", 10))
t1.start(); t2.start()
t1.join();  t2.join()
print(stock)  # תמיד 80 ✓

04 / CRITICAL SECTION — גישה נכונה

Critical Section — עיקרון הגישה המינימלית

Lock הוא בקבוק צוואר — נעל כמה שפחות, שחרר כמה שיותר מהר

# ❌ Lock רחב מדי — בזבוז!
def process_order(order):
    with warehouse_lock:  # ← נועל הכל
        # שלב 1: חישוב מחיר (לא צריך lock!)
        price = calculate_price(order)
        # שלב 2: עיבוד תשלום (לא צריך lock!)
        process_payment(price)
        # שלב 3: עדכון מלאי (כן צריך lock)
        warehouse[item] -= 1
    # ← שחרר רק כאן — כל הזמן חסום!
כל threads אחרים חסומים גם בזמן החישוב ועיבוד התשלום — ביצועים גרועים! 
# ✅ Lock צר — רק מה שצריך
def process_order(order):
    # שלב 1: חישוב — ללא lock
    price = calculate_price(order)

    # שלב 2: תשלום — ללא lock
    process_payment(price)

    # שלב 3: עדכון מלאי — רק כאן lock!
    with warehouse_lock:
        warehouse[item] -= 1
    # ← שוחרר מיד אחרי העדכון
threads אחרים יכולים לחשב ולעבד תשלומים במקביל — רק העדכון הסופי ממתין.

חוקי Critical Section

01 — כניסה

רק thread אחד יכול להיות ב-Critical Section בכל רגע נתון

02 — קצר

הקוד בתוך ה-Lock צריך להיות קצר ומהיר ככל האפשר

03 — שחרור

תמיד שחרר את ה-Lock — השתמש ב-with כדי להבטיח זאת

05 / DEADLOCK — המלכודת

Deadlock — עצירה מוחלטת

שני threads ממתינים אחד לשני — לנצח

🚦 אנלוגיה: שתי מכוניות בצומת צר — כל אחת מחכה שהאחרת תזוז. אף אחת לא תזוז לעולם. 
# ❌ Deadlock — קוד מסוכן!
lockA = threading.Lock()
lockB = threading.Lock()

def thread_one():
    with lockA:         # נועל A
        time.sleep(0.1)   # ← B מנסה לנעול A
        with lockB:     # ממתין ל-B ← DEADLOCK!
            ...

def thread_two():
    with lockB:         # נועל B
        time.sleep(0.1)   # ← A מנסה לנעול B
        with lockA:     # ממתין ל-A ← DEADLOCK!
            ...
Thread 1 Thread 2 ────────────────────────────────────── acquire(lockA) ✓ acquire(lockB) ✓ trying acquire(lockB)... trying acquire(lockA)... ⏸ WAITING... ⏸ WAITING... ⏸ WAITING... ⏸ WAITING... ⏸ WAITING... ⏸ WAITING... 💀 DEADLOCK! 💀
# ✅ פתרון: סדר נעילה אחיד תמיד!
def thread_one():
    with lockA:     # A לפני B — תמיד
        with lockB:
            ...

def thread_two():
    with lockA:     # A לפני B — גם כאן!
        with lockB:
            ...
# Thread 2 ממתין ל-A — לא נועל B קודם
# ← אין מעגל המתנה → אין Deadlock ✓
06 / דוגמה מלאה — מערכת בנק

מערכת בנק — threads מרובים

דוגמה מלאה: חשבון בנק עם הפקדות ומשיכות מקביליות

# bank.py — מערכת בנק thread-safe
import threading, logging

logging.basicConfig(
    level=logging.DEBUG,
    format="%(asctime)s [%(name)s] %(message)s"
)

class BankAccount:
    def __init__(self, owner, balance):
        self.owner = owner
        self.balance = balance
        self._lock = threading.Lock()
        self.logger = logging.getLogger(owner)

    def deposit(self, amount, by=""):
        with self._lock:
            old = self.balance
            self.balance += amount
            self.logger.info(
                f"[{by}] הפקדה {amount} | "
                f"{old}→{self.balance}"
            )

    def withdraw(self, amount, by=""):
        with self._lock:
            if self.balance >= amount:
                old = self.balance
                self.balance -= amount
                self.logger.info(
                    f"[{by}] משיכה {amount} | "
                    f"{old}→{self.balance}"
                )
                return True
            self.logger.warning(
                f"[{by}] יתרה לא מספיקה!")
            return False
      
# המשך — יצירת threads
account = BankAccount("MyBank", 1000)

def atm_worker(name, ops):
    """מדמה לקוח ATM"""
    for op, amt in ops:
        if op == "dep":
            account.deposit(amt, by=name)
        else:
            account.withdraw(amt, by=name)

# 3 לקוחות בו-זמנית
threads = [
    threading.Thread(target=atm_worker, args=(
        "Dana", [("dep",200),("wd",50)])),
    threading.Thread(target=atm_worker, args=(
        "Roni", [("wd",300),("dep",100)])),
    threading.Thread(target=atm_worker, args=(
        "Amir", [("dep",500),("wd",200)])),
]

for t in threads: t.start()
for t in threads: t.join()

print(f"יתרה סופית: {account.balance}")
# תמיד נכון: 1000+200-50-300+100+500-200 = 1250
📌 עיקרון OOP: ה-Lock מוסתר בתוך המחלקה. המשתמשים של BankAccount לא צריכים לדעת על הנעילה — היא חלק מה-interface.
07 / RLOCK + PATTERNS

RLock ו-Patterns מתקדמים

RLock = Reentrant Lock — thread שכבר מחזיק את ה-Lock יכול לנעול שוב

RLock — מתי צריך?

# ❌ בעיה עם Lock רגיל
lock = threading.Lock()

def outer():
    with lock:
        inner()  # ← DEADLOCK!
                 # inner מנסה לנעול lock שכבר נעול!

def inner():
    with lock:   # ← ממתין לנצח
        ...
      
# ✅ פתרון עם RLock
rlock = threading.RLock()  # Reentrant

def outer():
    with rlock:
        inner()  # ✓ — אותו thread, מותר!

def inner():
    with rlock:  # נועל שוב — בסדר
        ...
# rlock.release() נקרא פעמיים

מתי להשתמש במה?

threading.Lock

ברוב המקרים — פשוט, יעיל, מהיר. Thread שמחזיק לא יכול לנעול שוב (Deadlock!).

threading.RLock

כאשר פונקציות קוראות אחת לשנייה ושתיהן צריכות Lock על אותו משאב.

threading.Semaphore(n)

כמו Lock, אבל מאפשר n threads בו-זמנית. שימושי להגבלת גישה (למשל — מקסימום 5 חיבורים למסד).

# מקסימום 3 threads בו-זמנית
sem = threading.Semaphore(3)
with sem:
    ...  # עד 3 בו-זמנית

09 / סיכום

🧠 סיכום — מה למדנו

Race Condition
  • שני threads קוראים אותו ערך
  • שניהם מחשבים ומשנים
  • כתיבה אחת נעלמת
  • תוצאה: נתונים שגויים
Lock / threading.Lock
  • מבטיח Mutual Exclusion
  • with lock: — מומלץ תמיד
  • נעל כמה שפחות (Critical Section)
  • RLock לפונקציות רקורסיביות
Deadlock
  • מעגל המתנה בין threads
  • פתרון: סדר נעילה אחיד תמיד
  • or: acquire(timeout=...)
  • or: ארכיטקטורה ללא מעגל
📌 כלל אצבע לעבודה: כל גישה ל-clients ול-users.json מתוך thread צריכה להיות מוגנת ב-Lock. השתמשו ב-with lock: תמיד. לפני iteration על dict — עשו snapshot עם dict(clients). 
# Template לשרת צ'אט thread-safe
clients_lock = threading.Lock()
users_lock   = threading.Lock()

# הוספת לקוח (בעת login)
with clients_lock:
    clients[username] = conn

# broadcast — snapshot לפני iteration!
def broadcast(msg, exclude=None):
    with clients_lock:
        snap = dict(clients)          # ← snapshot!
    for user, sock in snap.items():
        if sock != exclude:
            sock.send(msg.encode())    # ← מחוץ ל-Lock

# שמירה ל-JSON
with users_lock:
    data = load_users()
    data[username] = password
    save_users(data)
