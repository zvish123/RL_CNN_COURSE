הצפנה ואבטחת התקשורת

בשלב 10 ראינו תרגיל "הסנפת סיסמה גלויה" — סיסמה שעברה ב-HTTP בטקסט פתוח, וכל מי שהסניף יכול לקרוא אותה. השלב הזה סוגר את הפער הזה: לוקחים את כל מה שבנינו, ומוודאים שמה שעובר ברשת לא חשוף.

הצפנה סימטרית

אותו מפתח משמש גם להצפנה וגם לפענוח. מהיר, ומתאים להצפנת payload בתקשורת בין לקוח לשרת שכבר "מכירים" זה את זה.

plaintext = "הודעה רגישה"
key = generate_key()
ciphertext = encrypt(plaintext, key)
# ... שולחים את ciphertext ברשת ...
decrypted = decrypt(ciphertext, key)  # אותו מפתח בדיוק

האתגר המרכזי: איך שני הצדדים מסכימים על המפתח בלי שמישהו שמסניף את הרשת יתפוס אותו?

תזכורת: Hash

  • פונקציית Hash היא חד-כיוונית — אפשר לחשב hash מטקסט, אבל לא "להפוך" hash בחזרה לטקסט
  • אורך הפלט קבוע תמיד, לא משנה כמה ארוך הקלט
  • שינוי קטנטן בקלט ➜ hash שונה בתכלית

בדיוק בגלל זה משתמשים ב-hash לאחסון סיסמאות: השרת לא "צריך לדעת" את הסיסמה, רק להשוות hash-ים.

שמירת סיסמאות — Hash, לא טקסט גלוי

# הרשמה - שומרים רק hash, לא את הסיסמה עצמה
password_hash = bcrypt.hashpw(password.encode(), bcrypt.gensalt())
db.save(username, password_hash)

# התחברות - משווים hash, לא משווים סיסמאות
if bcrypt.checkpw(entered_password.encode(), stored_hash):
    login_success()

gensalt() מוסיף "מלח" אקראי לכל סיסמה, כך שאפילו שתי סיסמאות זהות מקבלות hash שונה — מונע התקפת "מילון hash-ים מוכנים מראש".

תזכורת: JWT ככלי אימות

JWT הוא טוקן חתום שמעיד שהלקוח "מי שהוא אומר שהוא" — שלושה חלקים מופרדים בנקודה:

header
.
payload
.
signature
  • Header — איזה אלגוריתם שימש לחתימה (לדוגמה HS256)
  • Payload — המידע עצמו (user_id, role, exp - זמן תפוגה)
  • Signature — HMAC על ה-header+payload עם מפתח סודי - מבטיח שלא שינו את התוכן

ה-payload לא מוצפן, רק חתום! כל מי שיתפוס את הטוקן יכול לקרוא אותו (לפענח base64). החתימה מונעת שינוי בתוכן, אבל לא מסתירה אותו.

ההודעה נחתמה ע"י מי שמחזיק במפתח הסודי הנכון.

התקפת Replay

מה קורה אם תוקף לא מנסה לשנות הודעה, אלא פשוט לוכד הודעה חוקית ושולח אותה שוב? אם ההודעה חתומה כראוי — היא תתקבל, כי החתימה תקינה!

# בלי הגנת replay - הודעה חוקית, אבל ישנה, מתקבלת בלי בעיה
{"action": "transfer", "amount": 100, "signature": "..."}

הפתרון: timestamp (לבדוק שההודעה לא ישנה מדי) ו/או nonce (מספר חד-פעמי שהשרת זוכר ולא מקבל פעמיים).

הגנת Replay — בקוד

used_nonces = set()

def validate_message(msg, max_age_seconds=30):
    if time.time() - msg["timestamp"] > max_age_seconds:
        return "⚠ הודעה ישנה מדי"
    if msg["nonce"] in used_nonces:
        return "⚠ נוסה שימוש חזרתי בהודעה (replay)"
    used_nonces.add(msg["nonce"])
    return "✓ הודעה תקינה"

בדיוק כמו חלון הזמן והסף שהגדרתם בשלב 10 — רק שכאן זה בתוך השרת עצמו, לא בניתוח תעבורה חיצונית.

התוצר: SecurityManager + פרק "חולשות והגנות"

🔧 מחלקת SecurityManager

encrypt/decrypt payload, hash_password/verify_password, sign/verify_signature, validate_message (anti-replay).

📄 פרק "חולשות והגנות"

תיעוד כתוב: אילו חולשות זיהינו בפרויקט שלנו, ואיך כל הגנה (הצפנה/hash/אימות/anti-replay) פותרת אחת מהן.
