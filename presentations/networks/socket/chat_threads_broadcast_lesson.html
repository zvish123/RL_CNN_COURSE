Threads + Broadcast בצ'אט

דף אינטראקטיבי לתלמידים: תרשים זרימה של התהליכונים בשרת, דיאגרמת sequence של login ו־message, ושקף לימודי שמסביר למה צריך clients_lock.

thread per client broadcast lock recv / sendall

1. תרשים זרימה של התהליכונים בשרת

בשרת יש thread ראשי שמחכה לחיבורים חדשים, ולכל לקוח שנכנס נפתח thread נפרד שמטפל רק בו. כך אפשר לשרת כמה לקוחות במקביל.

Thread ראשי

srv.listen(5)
accept()
↓
Thread(target=handle_client)
↓
accept() שוב
ה־thread הראשי לא מטפל בהודעות chat. הוא רק מקבל חיבורים חדשים ופותח thread עבור כל לקוח.

Thread של לקוח A

recv()
↓
json.loads(...)
↓
action == 'login'
↓
clients[conn] = username
↓
broadcast(status)

Thread של לקוח B

recv()
↓
json.loads(...)
↓
action == 'message'
↓
broadcast(message)
ה־broadcast לא פותח thread חדש. הוא רץ בתוך ה־thread של הלקוח ששלח את ההודעה.

משאב משותף

clients = {sock: username}
clients_lock
↓
with clients_lock:
for sock in list(clients):
sock.sendall(...)
בלי lock שני threads יכולים במקביל להוסיף, למחוק ולעבור על clients — וזה race condition.

שאלה לתלמידים

למה השרת לא נתקע כולו אם לקוח אחד יושב ב־recv() ולא שולח כלום?

כי recv() חוסם רק את ה־thread של אותו לקוח. שאר ה־threads ממשיכים לעבוד, וה־thread הראשי עדיין יכול לקבל לקוחות חדשים.

2. דיאגרמת Sequence של login ו־message

הדיאגרמה מחולקת לשלושה משתתפים: לקוח, thread של השרת עבור אותו לקוח, וכל הלקוחות המחוברים שאליהם נשלח ה־broadcast.

לקוח
/login alice 1234
⇢
מחכה לתשובת info
⇠
/msg hello class
⇢
recv_loop מקבל הודעות
Thread של השרת
recv() + json.loads()
action == 'login'
בודק DB עם Session()
clients[conn] = 'alice'
send_json(info)
broadcast(status)
recv() שוב
action == 'message'
broadcast({action:'message', username:'alice', message:'hello class'})
כל הלקוחות המחוברים
מקבלים status: *** alice connected ***
Alice עצמה יכולה לקבל info
Bob / Carol / Alice מקבלים את הודעת הצ'אט
ה־recv_loop של כל לקוח מדפיס למסך

Sequence של login

client -> server_thread: {"action":"login", ...}
server_thread -> DB: query(User)
server_thread -> clients: add connection
server_thread -> client: {"action":"info", ...}
server_thread -> all clients: {"action":"status", ...}

Sequence של message

client -> server_thread: {"action":"message", ...}
server_thread -> clients_lock: acquire
server_thread -> all clients: broadcast(message)
all clients -> recv_loop: print(username > text)

שאלה לתלמידים

למה אחרי login יש גם send_json(info) וגם broadcast(status)?

כי הלקוח שביצע login צריך תשובה אישית כמו Welcome alice, אבל שאר הלקוחות צריכים לדעת שמשתמש חדש התחבר. לכן יש הודעה פרטית אחת והודעת broadcast נפרדת.

3. שקף HTML אינטראקטיבי: איך עובד broadcast

החלק הכי חשוב להבנה: ה־broadcast משתמש ב־thread קיים, ולא יוצר אחד חדש. מה שחייבים להגן עליו הוא לא הקוד עצמו, אלא clients — המשאב המשותף.

מה קורה כשלקוח שולח הודעה?

  1. ה־thread של הלקוח קורא recv().
  2. הוא מפענח JSON.
  3. הוא מזהה action == 'message'.
  4. הוא קורא ל־broadcast(...).
  5. אותו thread בדיוק עובר על כל הלקוחות ושולח להם הודעה.

למה צריך lock?

  • כי כמה threads יכולים לגעת במקביל ב־clients.
  • אחד יכול לעשות broadcast.
  • אחר יכול להסיר לקוח שהתנתק.
  • אחר יכול להוסיף לקוח חדש אחרי login.
def broadcast(data_dict):
    payload = (json.dumps(data_dict) + '\n').encode()

    with clients_lock:
        for sock in list(clients):
            try:
                sock.sendall(payload)
            except Exception:
                clients.pop(sock, None)
החלק הנכון כאן: יש הגנה עם clients_lock, ולכן רק thread אחד בכל פעם ניגש לרשימת הלקוחות.
החיסרון כאן: אם sendall() ללקוח מסוים איטי, ה־lock נשאר תפוס יותר זמן. במערכת גדולה עדיף להעתיק snapshot של sockets תחת lock, לשחרר lock, ורק אז לשלוח.

שאלה לתלמידים

מה הבעיה האפשרית אם נעשה for sock in clients: בלי clients_lock?

בזמן שהלולאה רצה, thread אחר יכול למחוק או להוסיף לקוח. זה עלול לגרום ל־race condition, לשגיאות כמו שינוי גודל dictionary בזמן איטרציה, או למצב לא עקבי של שידור.

עוד שאלה

האם broadcast פותח thread חדש?

לא. הוא רץ בתוך ה־thread של הלקוח ששלח את ההודעה או שגרם לאירוע כמו login/logout.
