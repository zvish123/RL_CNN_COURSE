Wireshark וניתוח תעבורה:
בונים את PacketAnalyzer

עד כה ה-AnomalyDetector שלכם הסתכל על קבצים ופעולות מערכת. השלב הזה מוסיף לו "עיניים" על הרשת — מודול שקורא תעבורה אמיתית (קובץ pcap או הסנפה חיה) ומזהה דפוסים חשודים, בדיוק כמו שעשינו ב-Wireshark וב-Scapy בתרגול.

למה זה הולך ביחד? תאוריה ← מודול

📖

מה למדנו (תרגול Wireshark/Scapy)

הסנפה, פילטרים, שכבות התקשורת (Ethernet/IP/TCP/HTTP), DNS, ARP, ובניית פילטר תיכנותי ב-Scapy.

🛠️

מה בונים עכשיו

מודול PacketAnalyzer שקורא פקטות (מ-pcap או מהסנפה), ומחזיר רשימת אנומליות לפי כלל שהצוות בוחר.

תזכורת: קובץ pcap

כל הסנפה ב-Wireshark אפשר לשמור לקובץ .pcap — "תמלול" מלא של כל הפקטות שעברו, עם הזמן, הכתובות, והתוכן הגולמי בייטים.

  • זה מקור הנתונים של PacketAnalyzer — בדיוק כמו שקובץ לוג הוא מקור הנתונים של מודולים אחרים בפרויקט
  • אפשר לפתוח אותו גם ב-Wireshark (לבדיקה ויזואלית) וגם ב-Scapy בפייתון (rdpcap("file.pcap")) — לקריאה תיכנותית
  • לצוות מומלץ לשמור 2-3 קבצי pcap לדוגמה: תעבורה "תקינה" ותעבורה עם דפוס חשוד שיצרתם בכוונה

פילטרים — איך מוצאים את מה שמחפשים

פילטר טוב הוא הצעד הראשון של כל אנומליה: לפני שכותבים קוד, בודקים ב-Wireshark שהדפוס באמת נראה בתעבורה.

http # רק HTTP dns # רק שאילתות/תשובות DNS ip.addr == 10.0.0.5 # תעבורה מ/אל כתובת מסוימת frame contains "txtPa" # חיפוש מחרוזת בתוך הפקטה tcp.flags.syn == 1 && tcp.flags.ack == 0 # רק SYN ראשוני
arp # רק פקטות ARP — לבדיקת ARP spoofing tcp.port == 80 || tcp.port == 443 # תעבורת אינטרנט (HTTP/HTTPS) dns.qry.name contains "evil" # שאילתת DNS לדומיין חשוד

DNS — בסיס ל-DNS Watchdog

כל שם אתר (לדוגמה example.com) מתורגם לכתובת IP בשאילתת DNS. בפקטת DNS אפשר לראות את qry.name (השם שמבקשים) ואת סוג השאילתה.

  • תוכנה זדונית (malware) פעמים רבות שולחת הרבה שאילתות לאותו דומיין חשוד, או לדומיינים אקראיים-נראים (DGA)
  • דפוס לזיהוי: דומיין שחוזר הרבה בזמן קצר, או דומיין באורך/תווים חריגים
  • זה הבסיס למודול הדוגמה DNS Watchdog שנפרט בהמשך

TCP — ה-handshake שמתחיל כל חיבור

לפני שמועברים נתונים, שני הצדדים "לוחצים יד" בשלושה שלבים — ה-3-way handshake:

לקוח
שרת
1. לקוח → שרת: SYN
2. שרת → לקוח: SYN, ACK
3. לקוח → שרת: ACK

הרבה SYN שנשלחים לאותו שרת בלי SYN-ACK שמושלם ל-ACK — סימן אפשרי לסריקת פורטים או ניסיון DoS (כמות חיבורים חריגה). זה הבסיס לדוגמה NetGuard.

ARP — בסיס ל-ARP Guardian

ARP מתרגם כתובת IP לכתובת MAC ברשת מקומית. כל מחשב "זוכר" טבלה של IP↔MAC.

  • ARP spoofing: תוקף משדר שהוא "הבעלים" של IP מסוים, עם כתובת MAC שלו — וכך מסיט תעבורה אליו (התקפת MITM)
  • הסימן ברשת: לאותו IP מופיעות בתוך זמן קצר כמה כתובות MAC שונות
  • זה הדפוס שמודול ARP Guardian מחפש

מתאוריה לקוד — מ-Wireshark ל-Scapy

PacketAnalyzer לא ירוץ בתוך Wireshark — הוא יקרא פקטות בפייתון. הבסיס שראינו בתרגול ה-Scapy:

from scapy.all import rdpcap packets = rdpcap("capture.pcap") # קריאת קובץ ההסנפה for pkt in packets: if pkt.haslayer("DNS"): print(pkt["DNS"].qd.qname) # שם הדומיין שנשאל

בדיוק כמו packets[DNS] בתרגול — רק שכאן הקוד שלכם הוא חלק מהמודול, לא תרגיל עומד בפני עצמו.

ארבע דוגמאות מודול (לבחירת הצוות)

🔍 DNS WATCHDOG

דומיינים שחוזרים הרבה

סופר שאילתות DNS לפי דומיין, ומתריע כשדומיין חוצה סף חזרות בחלון זמן.

🛡️ ARP GUARDIAN

שינוי MAC לאותו IP

בונה טבלת IP↔MAC ומתריע כשמופיע MAC שני לאותו IP — סימן ל-spoofing.

📈 NETGUARD

כמות חיבורים חריגה

סופר חיבורי TCP (SYN) לכתובת/פורט יעד בחלון זמן, ומתריע מעבר לסף.

💬 CHAT SECURITY

הודעות רבות בזמן קצר

בתעבורת אפליקציית הצ'אט של הפרויקט — מזהה ניסיון "הצפה" של הודעות מאותו משתמש.

העקרון המשותף — מה זו "תבנית חריגה"?

בכל ארבעת הדוגמאות, הזיהוי בנוי משלושה מרכיבים קבועים:

  • מה סופרים — דומיין, MAC, חיבור, הודעה (תלוי במודול)
  • חלון זמן — לא "אי פעם" אלא "כמה פעמים בתוך X שניות"
  • סף (threshold) — מספר שמעליו זה כבר לא "נורמלי" — ובחירת הסף היא החלטה הנדסית של הצוות, לא נכון/לא נכון מוחלט
⏸ תחנת עצירה

מה עושים השבוע?

פותחים את דף העבודה — בניית PacketAnalyzer, וביחד עם הצוות:

  • מנתחים קובץ הסנפה לדוגמה ומזהים את הפרוטוקולים שבו
  • בוחרים תבנית חריגה אחת (מתוך הדוגמאות, או רעיון משלכם) ומגדירים חלון זמן וסף
  • מתכננים את שלד הקוד של PacketAnalyzer ומגישים קובץ JSON עם התשובות
