למה השרת שכתבנו בשלב 5 לא מספיק?

השרת הבסיסי קרא ל-accept() פעם אחת, טיפל בלקוח אחד, וסגר את עצמו. הבעיה: accept() ו-recv() הן פעולות חוסמות (blocking) — כל עוד השרת מחובר ללקוח אחד וממתין למידע ממנו, הוא לא יכול לטפל בלקוח שני שמנסה להתחבר. צריך לפתוח "מסלול" נפרד לכל לקוח.

Process מול Thread

Process (תהליך)Thread (תהליכון)
זיכרוןלכל תהליך זיכרון נפרדכל ה-threads בתוך תהליך אחד חולקים את אותו הזיכרון
משקל"כבד" — יצירה איטית יחסית"קל" — יצירה מהירה, יעיל לריבוי לקוחות
שימוש כאן—thread נפרד לכל לקוח שמתחבר לשרת

יצירת Thread בפייתון

import threading

def handle_client(client_socket, address):
    ...  # כל הקוד שמטפל בלקוח הזה בלבד

t = threading.Thread(target=handle_client, args=(client_socket, address))
t.start()   # מריץ את handle_client במקביל, לא חוסם את הלולאה הראשית

⚠️ Race Condition ולמה צריך Lock

כל ה-threads חולקים זיכרון — כולל כל רשימה משותפת שתחזיקו (לדוגמה רשימת הלקוחות המחוברים). אם שני threads מנסים לשנות את אותה רשימה בדיוק באותו רגע (אחד מוסיף לקוח, אחד מסיר לקוח שהתנתק) — יכולה להיווצר race condition: התנהגות לא צפויה שתלויה ב"מי הספיק קודם", ולעיתים גם קריסה.

הפתרון: threading.Lock() — "מנעול" שמבטיח שרק thread אחד בכל רגע נתון יכול לגשת לקוד הרגיש (לדוגמה לשנות את רשימת הלקוחות): 

clients_lock = threading.Lock()

with clients_lock:
    active_clients.append(client_socket)   # רק thread אחד בפנים בכל רגע
שאלת הבנה 0.1 (בדיקה אוטומטית)

מה עלול לקרות בלי Lock על רשימת הלקוחות המשותפת?

✅ נכון. רשימות (כמו מבני נתונים רבים אחרים) אינן "בטוחות אוטומטית" לעדכון מכמה threads בו-זמנית. בלי Lock, פעולות כמו append ו-remove שמתבצעות "בערך באותו זמן" משני threads יכולות להתנגש ולגרום לתוצאה שגויה או לשגיאה. זו הסיבה שעוטפים אותן ב-with clients_lock:.

צעד 1 — Thread לכל לקוחקל-בינוני

משימה 1.1

שדרגו את לולאת ה-accept בשרת כך שתפעיל thread לכל לקוח

השרת ממשיך להאזין ולקבל לקוחות חדשים בלי לחכות שהלקוח הקודם יתנתק. כל לקוח מטופל בפונקציה handle_client שרצה ב-thread נפרד.

▶ קוד זה דורש socket ו-threading אמיתיים — להריץ ב-PyCharm/IDE מקומי
💡 שאלה למחשבה: מדוע הלולאה הראשית בשרת (while True שקוראת ל-accept) חייבת להישאר פשוטה ומהירה, ולא לבצע בעצמה את כל הטיפול בלקוח? (רמז: מה קורה ללקוחות אחרים בזמן שהיא "תקועה"?)

צעד 2 — ניהול רשימת לקוחות פעיליםבינוני

משימה 2.1

הוסיפו את הלקוח לרשימה בתחילת handle_client, והסירו אותו בסיומה

כל גישה לרשימה המשותפת active_clients חייבת לעבור דרך clients_lock — גם הוספה וגם הסרה, כדי למנוע את ה-race condition שדיברנו עליו בשאלה 0.1.

▶ דורש הרצה אמיתית ב-IDE
with clients_lock:
    active_clients.append(client_socket)
שימוש ב-try/finally מבטיח שהלקוח יוסר מהרשימה וייסגר תמיד — גם אם קרתה שגיאה באמצע הטיפול בו, וגם אם הלקוח התנתק "בצורה לא מסודרת".
צ'ק-ליסט לבדיקה עצמית:
  • בכל הוספה/הסרה מ-active_clients משתמשים ב-with clients_lock:
  • אחרי שלקוח מתנתק, מספר הלקוחות הפעילים שמודפס באמת קטן ב-1
  • אם מריצים 2 לקוחות ומנתקים אחד, השני נשאר מחובר ופעיל (זה ה"מבחן" האמיתי ל-thread-per-client)
3

צעד 3 — ניתוק לקוח בלי להפיל את השרתבינוני-מאתגר

כשלקוח נסגר (למשל סוגרים את חלון ה-client, או קוראים ל-close() בצד שלו), הקריאה הבאה ל- recv() בצד השרת לא תזרוק שגיאה — היא תחזיר b'' (בתים ריקים). זה הסימן שהלקוח התנתק. בנוסף, כל קריאה ל-socket עלולה לזרוק ConnectionResetError אם הניתוק היה "פתאומי" — לכן עוטפים גם ב-try/except.

משימה 4.1

לקוח (client.py) ללא שינוי מהותי משלב 5

צ'ק-ליסט לתוצר העובד של השלב (כל הסעיפים חייבים לעבור):
  • הרצתם server.py בחלון אחד, ו-3 מופעים נפרדים של client.py בשלושה חלונות נוספים
  • שלושת הלקוחות מחוברים בו-זמנית — ראיתם בקונסולת השרת 3 הודעות "לקוח התחבר"

  • קונסולת השרת הדפיסה את מספר הלקוחות הפעילים הנכון לפני ואחרי הניתוק
