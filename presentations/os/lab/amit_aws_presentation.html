02 / הבעיה
מוטיבציה
למה P2P?
∞ שטח לא מנוצל ברשת
💾 משתמשים שצריכים אחסון
🔗 חיבור ביניהם = ערך

הבעיה שרציתי לפתור

  • אצל חלק מהאנשים — דיסק מלא, אין מקום לקבצים חשובים
  • אצל אחרים — עשרות ג'יגה-בייט לא מנוצלים "מתבזבזים"
  • פתרונות קיימים (Google Drive, Dropbox) דורשים שרת מרכזי שאפשר להאקר אותו
  • השוק: Filecoin, Arweave, BitTorrent File System — אך אף אחד לא "בניתי אני"

הרעיון המרכזי:

כל משתמש שמוריד את התוכנה הופך בו-זמנית ללקוח (מאחסן קבצים) ולתורם (נותן מקום לאחרים). כמו BitTorrent — אבל עם אחסון ארוך-טווח, הצפנה, וחשבון משתמש אמיתי.

04 / שיקולי עיצוב
החלטות טכנולוגיות
למה בחרתי פתרון 1
פתרון תיאור יתרונות חסרונות סטטוס
פתרון 1 שרת מרכזי מקשר לקוחות
Client → Server → Donor		 אימות זהות, עקיבות, עובד בין רשתות SPOF — שרת נופל, הכל נופל נבחר
פתרון 2 UDP Broadcast
"לצעוק" ברשת המקומית		 ללא שרת, עצמאי לחלוטין לא עובד בין רשתות, אין אימות, קשה לדעת אם לקוח מנותק נדחה
פתרון 3 UDP Broadcast + שרת לעקיבה
היברידי		 שילוב של 1 ו-2 "אם יש שרת — למה UDP?" — כפילות מיותרת נדחה

הנימוק שלי:
ללא שרת מרכזי — משתמש יכול להתחזות למשתמש אחר ולגנוב קבצים.
בנוסף, UDP Broadcast לא עובד בין רשתות שונות — מה שמגביל מאוד את הסקיילינג. עם שרת — אני מקבל אימות זהות, היסטוריה, ובסיס נתונים חינם.

Diffie-Hellman
חלופת מפתחות בכל חיבור חדש — שרת↔לקוח וגם לקוח↔לקוח. גורם שלישי לא יכול לפצח בלי חישובים כבדים.
🔒
AES / Fernet
כל הודעה ברשת מוצפנת. כל קובץ שמועלה מוצפן עם מפתח פרטי של המשתמש — התורם לא יכול לקרוא!
🛡️
Argon2
סיסמאות נשמרות כ-hash בלבד. גם אם מישהו פורץ לDB — הסיסמאות מוגנות.

def generate_key_pair(self):
    self.private_key = bytes_to_long(get_random_bytes(32))
    self.public_key  = pow(self.G, self.private_key, self.P)

def derive_shared_secret(self, public_key):
    shared = pow(public_key, self.private_key, self.P)
    self.key = long_to_bytes(shared, 16)[:32]  # 256-bit AES key

למה עיצוב כזה?
אורך קבוע לכותרת (22 תווים) מאפשר ל-recv() לדעת בדיוק כמה לקרוא — parsing פשוט וחסין לתנאי מרוץ.

כל ההודעות במערכת

LOGIN/REGISTER L → S הרשמה / כניסה
FIND_DONOR L → S בקשת תורם לפי גודל
DONOR_INFO S → L כתובת תורם פנוי
TRANSFER L ↔ L התחלת העברת קובץ
FILE_FIND L → S מציאת קובץ להורדה
SETTINGS L → S עדכון הגדרות תרומה
GET_FILES L → S חיפוש קבצים
07 / אלגוריתם מרכזי
חדשנות
מציאת תורם — האלגוריתם
def get_available_donors(size, username):
    user_free_space = db.get_all_users_free_space()
    user_count      = len(user_free_space)

    # מטריצה: user_space_division[i] = משתמשים
    # שיחד יש להם size/i מקום
    user_space_division = [[] for _ in range(user_count+1)]

    for user in user_free_space:
        if user in logged_users and user != username:
            if user_free_space[user] > 0:
                frac = math.ceil(size / user_free_space[user])
                if frac <= user_count:
                    user_space_division[frac].append(user)

    donors = []
    for i, division in enumerate(user_space_division):
        donors += division
        if len(donors) >= i and i != 0:
            return donors[:i]  # מספיק תורמים!
    return []

איך זה עובד?

  • שלב 1: בודק אם תורם אחד יכול לאחסן את כל הקובץ
  • שלב 2: אם לא — מחפש זוג תורמים שביחד יש להם מספיק
  • שלב 3: מוסיף תורמים עד שמצאנו קבוצה שעובדת
  • העתיד: Sharding — הקובץ מתחלק ומאוחסן במקביל אצל כמה תורמים

הייחודיות:
זה לא סתם "מצא תורם עם מקום" — האלגוריתם חושב על חלוקת קובץ בין מספר תורמים, בדיוק כמו שBitTorrent מחלק קטעים (chunks). בסיס מצוין להרחבה עם Sharding אמיתי.
