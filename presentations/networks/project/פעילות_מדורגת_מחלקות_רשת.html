קריאה לפני התחלה

בהתבסס על דרישות הפרויקט (כספת קבצים מאומתת ומוצפנת + ניטור אנומליות) — אלו הפקודות שעל הצוות לתכנן ולממש בפרוטוקול שבין NetworkClient ל-ClientHandler. לכל פקודה: תבנית הבקשה, תבנית התשובה, והשכבה שבה היא "חיה" (כל הפקודות הללו הן שכבת אפליקציה — הן רצות מעל ה-socket של שכבת התעבורה).

פקודה בקשת הלקוח תשובת השרת קשר לדרישת הפרויקט
LOGIN LOGIN|username|password OK|token או ERROR|invalid_credentials אימות משתמש — בסיס ל"מאומת" בתיאור הפרויקט
UPLOAD UPLOAD|filename|content OK|stored או ERROR|... שמירת קובץ מרחוק (מוצפן בצד השרת/לקוח)
DOWNLOAD DOWNLOAD|filename FILE|filename|content או ERROR|not_found שליפת קובץ אישי מהכספת
LIST LIST FILES|f1,f2,f3 הצגת תוכן הכספת של המשתמש המחובר
DELETE DELETE|filename OK|deleted או ERROR|not_found ניהול תוכן הכספת
LOGOUT LOGOUT OK|bye (ואז סגירת socket) סיום סשן מסודר
ALERT 🚨 — (נשלח מהשרת ללקוח, לא להפך) ALERT|reason|client_address רכיב ניטור האנומליות: השרת מזהה דפוס חשוד (למשל 3+ נסיונות LOGIN כושלים ברצף מאותו לקוח) ושולח התרעה
💡 חשוב להבין: ALERT שונה מכל שאר הפקודות — הוא לא תגובה לבקשה של הלקוח, אלא הודעה שהשרת יוזם. זה אומר שה-ClientHandler (או thread נפרד שמנטר את הלוג) צריך להיות מסוגל לשלוח הודעה ב-socket גם בלי שקדמה לה בקשה — תכננו את זה מראש בפרוטוקול, כדי שהלקוח (או קונסולת ניהול נפרדת) ידע להבדיל בין "תשובה לבקשה שלי" לבין "התרעה שהשרת שולח מיוזמתו".

ארכיטקטורת המחלקות

שלב 1 — מחלקת Protocol 🤖 (הרצה אוטומטית)

קל

תפקיד המחלקה: להגדיר את "שפת השיחה" בשכבת האפליקציה — איך הופכים הודעת טקסט לרצף בתים שאפשר לשלוח ב-socket, ואיך מזהים בצד המקבל היכן ההודעה מתחילה ונגמרת (כדי לא להסתמך על "ריפוד" כמו בתרגילים הקודמים).

השיטה: כל הודעה נשלחת כ"כותרת" (4 בתים שמייצגים את אורך ההודעה) ולאחריה תוכן ההודעה עצמו.

  • encode_message(message: str) -> bytes — מקבלת מחרוזת, מחזירה header (4 בתים, big-endian) + תוכן מקודד ב-UTF-8.
  • decode_header(header_bytes: bytes) -> int — מקבלת 4 בתים, מחזירה את האורך (int) שהם מייצגים.
  • HEADER_SIZE = 4 — קבוע מחלקה.
לחצו על "הריצו את הקוד" כדי לראות פלט...
data = message.encode('utf-8')
header = len(data).to_bytes(Protocol.HEADER_SIZE, 'big')
return header + data

לפענוח: int.from_bytes(header_bytes, 'big')

שלב 2 — מחלקת NetworkClient

בינוני (להרצה ב-IDE)
דורש socket אמיתי — להריץ ב-PyCharm/IDE מקומי. לא ניתן להרצה בדפדפן.

תפקיד המחלקה: לעטוף את כל הפעולות שלקוח מבצע — התחברות, שליחת הודעה (באמצעות Protocol), קבלת תשובה, וסגירה — כך שקוד הלקוח הראשי יקצר ל-3-4 שורות בלבד.

  • __init__(self, host, port) — שומר כתובת ופורט, יוצר socket (עדיין לא מתחבר).
  • connect(self) — מתחבר לשרת (connect()).
  • send_message(self, message: str) — משתמש ב-Protocol.encode_message ושולח את כל הבתים (שימו לב: send לא מבטיח שליחה מלאה בקריאה אחת — שקלו sendall).
  • receive_message(self) -> str — קורא בדיוק Protocol.HEADER_SIZE בתים (כותרת), מפענח את האורך, ואז קורא בדיוק את מספר הבתים הזה לתוכן (שימו לב: recv בודד לא מבטיח קבלת כל הבתים בבת אחת!).
  • close(self) — סוגר את ה-socket.
⚠️ שגיאה נפוצה: socket.recv(n) עשוי להחזיר פחות מ-n בתים גם אם עוד מידע בדרך (במיוחד בהודעות גדולות/רשת עמוסה). חובה לכתוב לולאה שממשיכה לקרוא עד שמתקבלים כל הבתים שה-header הצהיר עליהם.

שלב 3 — מחלקת NetworkServer

בינוני (להרצה ב-IDE)
דורש socket אמיתי — להריץ ב-PyCharm/IDE מקומי.

תפקיד המחלקה: לפתוח, להאזין ולקבל לקוחות — ולכל לקוח שמתחבר, ליצור ClientHandler נפרד (כדי שכמה לקוחות יוכלו להתחבר במקביל בלי לחכות זה לזה).

  • __init__(self, host, port, max_clients=5)
  • start(self) — bind + listen.
  • accept_clients(self) — לולאת while True שקוראת ל-accept(), ועם כל לקוח חדש יוצרת ClientHandler חדש ב-thread נפרד ומפעילה אותו (threading.Thread).
  • stop(self) — סוגר את ה-socket הראשי.
client_socket, address = self.server_socket.accept()
handler = ClientHandler(client_socket, address)
thread = threading.Thread(target=handler.run)
thread.start()

שלב 4 — מחלקת ClientHandler

מאתגר (להרצה ב-IDE)
דורש socket אמיתי — להריץ ב-PyCharm/IDE מקומי.

תפקיד המחלקה: לטפל בלקוח בודד מתחילתו ועד סופו — לקבל ממנו פקודות (באמצעות Protocol), לבצע אותן, ולשלוח תשובה. כאן יושב ה"מוח" של הפרוטוקול — ממשו את שש הפקודות ממאגר הפקודות שלמעלה: LOGIN, UPLOAD, DOWNLOAD, LIST, DELETE, LOGOUT, ובנוסף — מנגנון ספירת נסיונות LOGIN כושלים שמהווה את הבסיס לזיהוי האנומליה.

  • __init__(self, client_socket, address) — שמרו גם self.logged_in_user = None ו-self.failed_logins = 0.
  • run(self) — לולאה שמקבלת הודעה (Protocol + recv), שולחת ל-handle_command, שולחת חזרה את התשובה, וממשיכה עד LOGOUT או ניתוק.
  • handle_command(self, command: str) -> str — מפרקת לפי "COMMAND|param1|param2" ומנתבת לdict של פונקציות-טיפול (אחת לכל פקודה), כדי לא לכתוב if/elif ארוך.
  • בדיקת הרשאה: UPLOAD/DOWNLOAD/LIST/DELETE מורשות רק אם self.logged_in_user אינו None — אחרת להחזיר ERROR|not_logged_in.
  • זיהוי אנומליה: כל LOGIN כושל מעלה את self.failed_logins ב-1; אם הוא מגיע לסף שהצוות קבע (לדוגמה 3) — שלחו ALERT (להדפסה בקונסולת השרת לפחות, או שליחה בפועל ל-socket ניהול אם תכננתם כזה).
  • טיפול בשגיאות: חובה try/except סביב כל פעולת קובץ (UPLOAD/DOWNLOAD/DELETE) — ראו תרגיל 2.7 בספר.
בדיקת סף ההתרעה יכולה להיראות כך:
if self.failed_logins >= self.FAILED_LOGIN_ALERT_THRESHOLD:
    print(f"ALERT|too_many_failed_logins|{self.address}")
🛡️ תזכורת מהספר (תרגיל 2.7): אם הלקוח מבקש פעולה על קובץ שלא קיים, אל תיתנו לתוכנית לקרוס — עטפו ב-try/except והחזירו הודעת שגיאה ידידותית ללקוח, כדי שה-thread של הלקוח הזה ימשיך לרוץ (ושאר הלקוחות לא יושפעו).
