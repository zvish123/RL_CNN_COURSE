למה אנחנו צריכים את ExperimentConfig?

ניהול פרויקט AI דורש מאות הרצות של ניסויים. בלי סדר, קל ללכת לאיבוד בין "מספרי קסם" (Magic Numbers) שקבורים בתוך הקוד.

המטרה: יצירת "מקור אמת יחיד" (Single Source of Truth) לכל הפרמטרים של הניסוי.
  • ניהול מרכזי: שינוי פרמטר במקום אחד משפיע על כל המערכת.
  • תיעוד: לכל ניסוי יש שם, תיאור והגדרות שניתן לשחזר.
  • מחקר השוואתי: היכולת להשוות בין גרסאות AI שונות בצורה מדעית.

ה"כפתורים" של הניסוי

אלו המשתנים שאנו "מכווננים" כדי להשפיע על התנהגות הלמידה של ה-AI:

@dataclass class ExperimentConfig: name: str # שם הניסוי לזיהוי num_games: int # כמה פעמים ה-AI יתאמן exploration_rate: float # אחוז המהלכים האקראיים (Epsilon) gamma: float # חשיבות העבר מול העתיד (Discount Factor) win_reward: float # הפרס על ניצחון loss_penalty: float # העונש על הפסד (חיזוק שלילי) tie_reward: float # הפרס על תיקו

הדקורטור @dataclass

כלי מודרני (מגרסה 3.7) שהופך מחלקות ל-"מיכלי נתונים" חכמים במינימום קוד.

מה זה חוסך לנו?

פייתון בונה עבורנו אוטומטית את הפונקציות הבאות:

  • __init__: אתחול המשתנים ללא צורך בכתיבה ידנית.
  • __repr__: ייצוג טקסטואלי - כשמדפיסים את האובייקט, מקבלים פלט קריא.
  • __eq__: השוואה - ניתן להשוות בין שני קונפיגים עם ==.

יכולות העל בפעולה

ה-Dataclass מאפשר לנו לעבוד עם אובייקטי התצורה בצורה טבעית ונקייה (Pythonic):

config = ExperimentConfig(name="AI_Test", gamma=0.9) # 1. הדפסה קריאה (repr) print(config) # פלט: ExperimentConfig(name='AI_Test', gamma=0.9, ...) # 2. גישה ישירה (Getters אוטומטיים) if config.gamma > 0.5: print("Long-term memory enabled") # 3. השוואה (eq) config_copy = ExperimentConfig(name="AI_Test", gamma=0.9) print(config == config_copy) # פלט: True

תבניות מוכנות עם @staticmethod

אנחנו משתמשים ב-@staticmethod כדי ליצור "מפעל" (Factory) שמייצר ניסויים מוכנים מראש:

@staticmethod def baseline_experiment(): """יוצר תצורת בסיס סטנדרטית בלחיצת כפתור""" return ExperimentConfig( name="baseline", num_games=5000, gamma=0.9, exploration_rate=0.1 ) # שימוש ללא צורך ביצירת מופע של המחלקה: config = ExperimentConfig.baseline_experiment()
