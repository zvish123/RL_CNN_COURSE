שכבת האפליקציה: HTTP / DNS / SMTP

בשלבים 10-11 בנינו מודול שמזהה דפוסים בתעבורה. עכשיו צוללים פנימה לתוך תוכן שכבת האפליקציה עצמה — מה בדיוק כתוב בתוך הבקשה/תשובה — כדי שהמודול יוכל להציג ניתוח אמיתי, לא רק "יש אנומליה / אין אנומליה".

מה זה פרוטוקול אפליקציה?

פרוטוקול שכבת האפליקציה הוא ה"שפה" שבה תוכנות מדברות זו עם זו — לא הביטים של הרשת, אלא המשמעות: בקשה, תשובה, פקודה.

  • כל פרוטוקול בנוי לפי מבנה קבוע — מי שלא עומד בו, מקבל שגיאה
  • שלושת הפרוטוקולים שנעבור עליהם נבדלים בתבנית: בקשה-תשובה (HTTP, DNS) מול שיחה מבוססת פקודות (SMTP)
  • בכל אחד יש header (מטא-דאטה על ההודעה) ולעיתים גם data/body (התוכן עצמו)

HTTP — בקשה ותשובה

לקוח שולח בקשה, שרת מחזיר תשובה. שורת הבקשה הראשונה כוללת את סוג הבקשה והמשאב המבוקש:

GET /index.html HTTP/1.1
Host: www.example.com

לתשובה מבנה דומה, עם status code בשורה הראשונה:

HTTP/1.1 200 OK
קודמשמעות
200OK — הבקשה הצליחה
301Moved Permanently — הועבר לכתובת אחרת
403Forbidden — אין הרשאה
404Not Found — הדף לא קיים
503Service Unavailable — השרת לא זמין

GET מול POST, ו-Headers

📥

GET

בקשת מידע מהשרת. פרמטרים מועברים ב-URL (?key=value).

📤

POST

שליחת מידע לשמירה בשרת (לדוגמה: העלאת תמונה, שליחת טופס).

בנוסף לשורה הראשונה, הבקשה/תשובה מכילה Headers — שדות מטא-דאטה כמו Host, User-Agent, Cookie, Content-Type — ולפעמים, אחרי שורה ריקה, data (תוכן הדף עצמו).

תזכורת: DNS Header

12 בתים ראשונים בכל הודעת DNS — שאלה או תשובה, באותו מבנה קבוע:

Identification (16 bit)   # מזהה שאילתה - מקושר בין שאלה לתשובה
QR (1 bit)               # 0=שאילתה, 1=תשובה
Opcode (4 bit)           # 0=שאילתה רגילה
AA / TC / RD / RA / Z      # דגלים (סמכותיות, קיצור, רקורסיה...)
RCODE (4 bit)            # קוד תשובה - 0=ללא שגיאה, 3=שם לא קיים
Total Questions / Answers / Authority / Additional

ב-Wireshark: dns.flags.response, dns.flags.rcode, dns.qry.name — בדיוק השדות שתרגלתם בשלב 10.

חשוב למודולים שלכם: MX הוא הרשומה הרלוונטית ל-Phishing Analyzer — היא חושפת באיזה שרת דואר באמת משתמש דומיין.

SMTP — שיחה מבוססת פקודות

בניגוד ל-HTTP/DNS (בקשה-תשובה בודדת), SMTP הוא שיחה רציפה של פקודות טקסט בין לקוח לשרת דואר, בפורט 25.

EHLO client.example.com\r\n
# השרת מגיב עם רשימת יכולות, ואז:
AUTH LOGIN\r\n
<username בקידוד base64>\r\n
<password בקידוד base64>\r\n
  • כל פקודה מסתיימת ב-\r\n — חובה, אחרת השרת לא מבין שהפקודה הסתיימה
  • שם המשתמש והסיסמה מקודדים base64, לא מוצפנים — Wireshark יראה אותם בקלות אם לא משתמשים ב-TLS

⏸ תחנת עצירה

מה עושים השבוע?

פותחים את דף העבודה — ניתוח שכבת האפליקציה, וביחד עם הצוות:

  • בוחרים פרוטוקול אחד (HTTP / DNS / SMTP) לניתוח עומק, בהתאם למודול שבחרתם
  • מאתרים בקובץ הסנפה את שדות הפרוטוקול הרלוונטיים ורושמים אותם
  • מתכננים את מסך ה-UI שיציג את הניתוח, ומגישים קובץ JSON עם התשובות

בדיקת הבנה

מה ההבדל המרכזי בין SMTP לבין HTTP ו-DNS?

SMTP לא משתמש בכלל בפורט קבוע
SMTP הוא שיחת פקודות רציפה, בעוד HTTP ו-DNS הם בקשה-תשובה בודדת
SMTP לא כולל header בכלל
