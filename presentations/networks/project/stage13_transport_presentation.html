שכבת התעבורה: TCP מול UDP

בשלבים הקודמים ניתחתם תעבורה קיימת. בשלב הזה אתם מחליטים — איזה פרוטוקול תעבורה הפרויקט שלכם עצמו ישתמש בו, ולמה. ההחלטה הזו, עם ההסברים, היא פרק שלם בתיק הפרויקט שלכם.

תזכורת: פורטים ושכבת התעבורה

שכבת התעבורה מאפשרת לכמה תוכניות על אותו מחשב "לשתף" את חיבור הרשת — בעזרת פורטים. כל פקטה נכנסת נמסרת לתוכנית הנכונה לפי הפורט.

  • סוקט TCP מזוהה לפי 4 ערכים: IP מקור, פורט מקור, IP יעד, פורט יעד
  • סוקט UDP מזוהה רק לפי פורט היעד — אין "חיבור" שצריך לזהות
  • זה הבסיס לכך שהשרת של הפרויקט שלכם יכול לשרת כמה לקוחות במקביל

UDP — פשוט, מהיר, לא אמין

  • Connectionless — אין handshake לפני שליחה, פשוט שולחים
  • Best-effort — אין הבטחה שהפקטה תגיע, ואין הבטחה לסדר הגעה
  • אין flow control ואין congestion control מובנים — האפליקציה צריכה לטפל בזה אם רוצה
  • מתאים ל: וידאו/אודיו בזמן אמת, DNS, מצבים שבהם פקטה אבודה פחות גרועה מהמתנה לה

TCP — אמין, מסודר, עם תקורה

  • Connection-oriented — לפני שליחת מידע, מתבצע 3-way handshake
  • אמינות: כל בית מקבל מספר סידורי (sequence number), והנמען שולח ACK על מה שקיבל
  • אם ACK לא מתקבל בזמן (timeout) — השולח שולח מחדש
  • מתאים ל: כל מקרה שבו אובדן מידע הוא בעיה — קבצים, הודעות, פקודות אימות

תזכורת: 3-Way Handshake

לקוח
שרת
1. לקוח → שרת: SYN
2. שרת → לקוח: SYN, ACK
3. לקוח → שרת: ACK

רק אחרי שלושת השלבים האלה — שני הצדדים "מסכימים" שיש חיבור, ומתחילים להעביר מידע.

מה קורה כשלקוח מתנתק?

שני סוגי "התנתקות" שצריך להבחין ביניהם:

👋

התנתקות מסודרת

הלקוח שולח FIN — "סיימתי לשלוח". השרת מגיב ב-ACK, וסוגר את הסוקט בצורה נקייה.

⚡

התנתקות פתאומית

קריסת תוכנית / ניתוק רשת — אין FIN. השרת מגלה את זה רק כשrecv() מחזיר נתונים ריקים, או כשפעולת שליחה נכשלת.

לכל שרת בפרויקט חייבת להיות תשובה ברורה לשתי המקרים — מה קורה למשאבים (סוקט, thread, נעילות) שהוקצו לאותו לקוח?

בעיית ההודעה החלקית

TCP מעביר זרם בייטים, לא הודעות. אם השרת שולח שתי הודעות, הלקוח עלול לקבל אותן מפוצלות או מאוחדות בקריאת recv() אחת.

# בלי טיפול נכון - בעיה:
data = client_socket.recv(1024)
message = json.loads(data)  # עלול להיכשל אם ההודעה לא הגיעה במלואה!

שני פתרונות נפוצים:

  • תו מפריד (delimiter) — לדוגמה \n בסוף כל הודעה, וקריאה עד שמוצאים אותו
  • קידומת אורך (length-prefix) — שולחים תחילה את אורך ההודעה, ואז קוראים בדיוק כמות בייטים זו

זיהוי הצפה / קצב חריג

זה בדיוק העקרון מ-NetGuard ששלב 10: סופרים אירוע (כאן: הודעות/חיבורים מאותו לקוח) בתוך חלון זמן, ומתריעים מעבר לסף.

# שלד מינימלי בצד השרת
message_counts = {}  # client_id -> list of timestamps

def check_flood(client_id, now, window=10, threshold=20):
    timestamps = message_counts.setdefault(client_id, [])
    timestamps.append(now)
    timestamps[:] = [t for t in timestamps if now - t <= window]
    return len(timestamps) > threshold

התוצר: פרק בתיק הפרויקט

"בחירת פרוטוקול התעבורה" — פרק כתוב, לא קוד. צריך להסביר ולהדגים ארבע נקודות:

  • 1. למה בחרנו TCP או UDP (לאיזה רכיב בפרויקט, ולמה זה מתאים לו)
  • 2. מה קורה כשלקוח מתנתק (מסודר ופתאומי)
  • 3. איך מטפלים בהודעה חלקית (הפתרון שבחרנו, עם קוד לדוגמה)
  • 4. איך מזהים הצפה או קצב הודעות חריג (חלון זמן + סף)
⏸ תחנת עצירה

מה עושים השבוע?

פותחים את דף העבודה — בחירת פרוטוקול התעבורה, וביחד עם הצוות:

  • מחליטים TCP או UDP לרכיב הרלוונטי בפרויקט, עם נימוק מנומק
  • כותבים בקצרה איך מטפלים בניתוק לקוח ובהודעה חלקית
  • מגדירים חלון זמן וסף לזיהוי הצפה, ומגישים את כל זה כטיוטה לפרק בתיק

בדיקת הבנה

למה TCP לבדו לא מבטיח שכל קריאת recv() תכיל הודעה שלמה אחת?

כי TCP מאבד הודעות בדרך
כי TCP מעביר זרם בייטים רציף, בלי גבולות הודעה מובנים
כי TCP לא תומך בהודעות גדולות מ-1024 בייט

קדימה לבנייה

מהחלטה הנדסית לפרק מתועד

ההחלטה על TCP/UDP, על טיפול בניתוקים, ועל זיהוי הצפה — היא חלק מהארכיטקטורה של הפרויקט שלכם. הפרק הזה בתיק הוא ההוכחה שחשבתם על זה, ולא רק "זה עבד אז המשכנו".
