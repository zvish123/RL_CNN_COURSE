למדנו שרת-לקוח עם Sockets.
מה היה חסר בקוד שמימשנו?
א
חסר bind() ו-listen() — בלי זה השרת לא יכול לקבל חיבורים
ב
השרת יכול לשרת רק לקוח אחד בכל פעם — חסר threading לטיפול מקבילי
ג
חסר הצפנה — הנתונים חשופים לכולם ברשת
ד
חסר try/except — הקוד יתרסק מיד עם כל שגיאה ברשת

✅ תשובה נכונה — ב

הקוד עבד מעולה — אבל רק ללקוח אחד. כשהלקוח הראשון מחובר, השרת חסום ב-recv() ולא יכול לקבל לקוח שני. הפתרון: ליצור Thread חדש לכל לקוח שמתחבר, כך accept() ממשיך לחכות לחיבורים חדשים. ג' ו-ד' הם שיפורים חשובים, אך לא חסרים — הקוד עובד בלעדיהם.

📐 ארכיטקטורה

למה צריך Threads בשרת?

😴

שרת חסום

שרת רגיל שמחכה ל-recv() של לקוח A — לא יכול לקבל לקוח B. הוא פשוט נתקע.

🧵

Thread לכל לקוח

לכל לקוח חדש — פותחים thread נפרד שמטפל בו, בזמן ש-thread הראשי ממשיך לקבל חיבורים.

👥

מקבילות אמיתית

10 משתמשים שולחים בו-זמנית. כל אחד "חי" ב-thread שלו ולא חוסם את השאר.

🔒

Lock — מגן משאב

כשThreads מרובים ניגשים לאותו מידע (רשימת לקוחות), צריך Lock שמבטיח גישה בטוחה.

מתכנת המשמרת: ה-thread הראשי = שוטר על entrance שמחלק מפתחות. כל לקוח מקבל מפתח (thread) ונכנס לחדר שלו — בלי לחסום אחרים.

מה יקרה אם שני threads יכתבו
לאותו dict בו-זמנית — בלי Lock?
א
לא יקרה כלום — Python ה-GIL מגן על כל המשתנים אוטומטית
ב
Python יחכה שה-thread השני יסיים לפני שה-thread הראשון יתחיל
ג
עלול להתרחש Race Condition — נתונים יינזקו, יאבדו, או תהיה קריסה
ד
Python יזרוק ThreadingError מיד עם הגישה

✅ תשובה נכונה — ג

זהו Race Condition — מצב מסוכן שקשה לאתר כי הוא לא מתרחש תמיד. ה-GIL של Python מגן על bytecode operations בסיסיות, אך לא על רצף של פעולות כמו dict[key] = val בזמן שאחר קורא. הפתרון: with threading.Lock() מסביב לכל גישה למשתנה משותף.

למה Lock?

בלי Lock — Thread 1 עשוי לקרוא מ-clients בזמן ש-Thread 2 מוחק ממנו. תוצאה: קריסה בלתי צפויה. ה-Lock מבטיח גישה בטוחה.

📋

list(clients)

יוצרים עותק של ה-dict לפני הלולאה — כי אנחנו עשויים למחוק socket תוך כדי ה-for, מה שגורם ל-RuntimeError.

📡

sendall vs send

send() עלול לשלוח רק חלק מהנתונים. sendall() מבטיח שכל ה-payload נשלח — קריטי בתקשורת רשת.

⚡ משימה #1 · עבודה עצמאית

🎯 ממשו את handle_client()

כתבו את הפונקציה handle_client(conn, addr) שמטפלת בלקוח בודד. השתמשו ב-clients_lock ו-broadcast שכבר קיימים.
1
קבלו את שם המשתמש: name = conn.recv(1024).decode()
2
הוסיפו ל-clients עם Lock: with clients_lock: clients[conn] = name
3
שלחו broadcast: f"*** {name} joined ***"
4
לולאת recv() — קבלו הודעות ו-broadcast לכולם
5
ב-finally: הסירו מ-clients, שלחו "disconnected", סגרו socket

def handle_client(conn, addr): try: name = conn.recv(1024).decode() with clients_lock: clients[conn] = name broadcast(f"*** {name} joined ***") print(f"[+] {name} connected from {addr}") while True: msg = conn.recv(1024).decode() if not msg: break broadcast(f"{name}: {msg}") finally: with clients_lock: clients.pop(conn, None) broadcast(f"*** {name} disconnected ***"); conn.close()

🗳️ Poll #3 · פרוטוקול תקשורת
למה מוסיפים \n בסוף כל הודעה ב-TCP?
א
TCP דורש שכל packet יסתיים ב-newline — זהו חלק מהמבנה הפנימי שלו
ב
TCP הוא זרם בלי גבולות — ה-\n הוא הסכם בין שרת ללקוח לסיום הודעה
ג
כדי לחסוך בזיכרון — \n גורם ל-Python לשחרר את buffer האחסון
ד
כדי להודיע לשרת שניתן לסגור את החיבור אחרי ההודעה

✅ תשובה נכונה — ב

TCP הוא stream protocol — הוא לא יודע איפה הודעה אחת נגמרת ואחרת מתחילה. שני packets יכולים להגיע יחד, או packet אחד יחולק לשניים. ה-\n הוא delimiter — סימן גבול שמוסכם מראש בין שרת ללקוח. ה-buffer מצבר נתונים עד שמוצא \n, ואז מפרק הודעה שלמה.

⚡ משימה #2 · CommunicationProtocol

📨 ממשו את מחלקת CommunicationProtocol

המחלקה תוסיף header לכל הודעה ותאמת הודעות נכנסות.
מבנה ה-header: 0000000005||hello (10 ספרות + || + הודעה)
1
build_message(command) — בנה string עם header: f"{len(command):010d}||{command}"
2
parse_message(msg) — פרק: length = int(msg[:10]), parts = msg[12:].split("||")
3
is_valid(command, params) — בדוק: פקודה חוקית? מספר פרמטרים נכון? אורך תואם?
4
השתמשו ב-VALID_COMMANDS = {'hello': 0, 'animal': 1, 'close': 0} מ-constants.py
בונוס: הוסיפו בדיקה שאורך ה-payload תואם את מה שה-header מצהיר.

📚 סיכום · Cheat Sheet

מה בנינו היום

1

SERVER bind + listen → accept() loop

Thread ראשי: מחכה לחיבורים ופותח thread חדש לכל לקוח.

2

THREAD handle_client(conn, addr)

recv → פירוק הודעה → business logic → תגובה. בסיום: ניקוי + disconnect.

3

LOCK clients_lock + broadcast()

גישה לכל משתנה משותף בתוך with clients_lock. broadcast = לולאה על list(clients).

4

CLIENT 2 Threads: send + recv

Thread ראשי: input. Thread שני (daemon): recv_loop שמאזין לשרת.
