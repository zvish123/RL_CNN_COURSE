הגדרה: מהו Protocol?

פרוטוקול הוא אוסף חוקים מוסכם מראש, שמאפשר לשני צדדים (או יותר) להעביר ביניהם מידע ולהבין אחד את השנייה — בלי תלות בכך שמישהו "יסביר" באותו רגע מה קורה.

שני הצדדים חייבים להסכים על הפרוטוקול לפני שמתחילים לדבר — לא ניתן "להמציא" אותו תוך כדי תנועה.

שיחת טלפון היא גם פרוטוקול: מי מתקשר אומר "הלו", המקבל עונה, יש "סליחה?" כשלא שומעים, ו"להתחדש"/"שלום" כדי לסגור את השיחה. אם תדלגו על השלבים — השיחה "תישבר" גם אם שני הצדדים מבינים עברית מצוין.

מה פרוטוקול תקשורת צריך לכלול?

  • שם הפעולה (Command) — מה בעצם רוצים לעשות? (לדוגמה LOGIN, GET)
  • פרמטרים — איזה מידע נוסף צריך לשלוח עם הפעולה? (לדוגמה שם משתמש, שם קובץ)
  • פורמט תשובה — איך נראית תשובת ההצלחה? ואיך תשובת שגיאה?
  • גבולות הודעה — איך יודעים שהודעה הסתיימה? (שלא תיקטע באמצע)
  • טיפול בשגיאות — מה קורה אם הצד השני שלח משהו לא תקין?
  • סדר פעולות — האם יש דברים שמוכרחים לקרות לפני אחרים? (למשל: התחברות לפני בקשת קובץ)
GET /index.html HTTP/1.1
Host: example.com
Accept: text/html

→ תשובה:
HTTP/1.1 200 OK
Content-Length: 1024

<html>...</html>

שימו לב: שם הפעולה (GET), פרמטר (כתובת), header עם metadata, ותשובה עם קוד סטטוס (200) ואורך תוכן מדויק — כל מה שמנינו ברשימה!

איך מחליטים על פרוטוקול? 4 שאלות מפתח

1. טקסט או בינארי?

טקסט (כמו JSON) — קל לדבג, פחות יעיל. בינארי — מהיר וקומפקטי, קשה יותר לקרוא "בעין".

2. איך מסמנים סוף הודעה?

תו מפריד (כמו \n)? אורך קבוע? כותרת (header) שמצהירה על האורך מראש?

3. עם/בלי מצב (state)?

חסר-מצב: כל בקשה עומדת בפני עצמה. עם-מצב: השרת "זוכר" מי מחובר ובאיזה שלב (כמו LOGIN לפני UPLOAD).

4. מה קורה בשגיאה?

קוד שגיאה מובנה? חיתוך החיבור? ניסיון חזרה (retry)? צריך להגדיר את זה מראש, לא "כשזה יקרה".

בדיקת הבנה

שאלה: שרת מקבל הודעות מלקוח באמצעות sock.recv(1024) בלבד, ומניח שכל קריאה אחת = הודעה אחת שלמה. מה הבעיה המרכזית בעיצוב הזה?

✅ נכון! TCP הוא זרם בתים (stream) ולא הודעות בדידות. recv() יכול להחזיר חלק מההודעה, הודעה שלמה, או כמה הודעות ביחד — תלוי בתזמון הרשת. בלי מנגנון שמסמן איפה הודעה מתחילה ונגמרת (header עם אורך, או תו מפריד) — אי אפשר לדעת מתי "לעצור ולקרוא".

כאוס: 3 טעויות נפוצות בעיצוב פרוטוקול

❌ "נשלח טקסט חופשי ונבין"

"please send file vacation.jpg now" — איך מפרקים את זה אוטומטית? כל ניסוח שונה דורש קוד חדש.

❌ "נסמוך על recv אחד = הודעה אחת"

כמו שראינו בשאלה הקודמת — לא מובטח. צריך framing מפורש.

❌ "לא נגדיר מה קורה בשגיאה"

לקוח מבקש קובץ שלא קיים — השרת קורס? שולח משהו לא צפוי? בלי הגדרה — כל מימוש יתנהג אחרת.

מבנה מוצע לפרוטוקול שלנו

שתי שכבות פתרון, אחת מעל השנייה:

שכבה 1 — Framing (גבולות ההודעה)

HEADER (4 בתים = אורך) + BODY (תוכן ההודעה)

פותר את בעיית ה-recv: הצד המקבל קורא תמיד 4 בתים ראשונים, לומד כמה בתים עוד מגיעים, וקורא בדיוק את הכמות הזו.

שכבה 2 — תוכן ההודעה (Command Format)

COMMAND|param1|param2|...

שם הפעולה ופרמטרים, מופרדים בתו מוסכם (|). פשוט לפענוח עם str.split("|").

⏸ עכשיו אתם — תחנת עצירה

בזוגות: רשמו על דף נייר/מחברת — אילו פקודות תזדקקו להן בפרוטוקול של הפרויקט שלכם (כספת קבצים + ניטור), ולכל פקודה: שם, פרמטרים, ותשובת הצלחה/שגיאה. אני אעבור בין הקבוצות. 5 דקות.

למה לעטוף את כל זה במחלקה אחת — Protocol?

  • גם הלקוח וגם השרת צריכים את אותו קוד קידוד/פענוח — אם כל צד יכתוב את זה בנפרד, באג קטן באחד מהם ישבור את כל התקשורת.
  • מחלקה אחת = מקור אמת יחיד (single source of truth) — מתקנים פעם אחת, שני הצדדים מיובאים ממנה.
  • קל לבדוק אותה בנפרד מהקוד של socket — אין צורך בחיבור רשת אמיתי כדי לבדוק שקידוד+פענוח עובדים.

מחלקת Protocol — הקוד

class Protocol:
    HEADER_SIZE = 4

    @staticmethod
    def encode_message(message: str) -> bytes:
        data = message.encode('utf-8')
        header = len(data).to_bytes(Protocol.HEADER_SIZE, 'big')
        return header + data

    @staticmethod
    def decode_header(header_bytes: bytes) -> int:
        return int.from_bytes(header_bytes, 'big')
header = sock.recv(Protocol.HEADER_SIZE)
length = Protocol.decode_header(header)

body = b""
while len(body) < length:          # לולאה! recv בודד לא מבטיח הכל
    body += sock.recv(length - len(body))

message = body.decode('utf-8')

סיכום

  • פרוטוקול = חוקים מוסכמים מראש, לא משהו שמבינים תוך כדי תנועה.
  • צריך להגדיר: שם פעולה, פרמטרים, פורמט תשובה, גבולות הודעה, וטיפול בשגיאות.
  • 4 שאלות עיצוב: טקסט/בינארי, סימון סוף הודעה, עם/בלי מצב, טיפול בשגיאות.
  • המבנה שבחרנו: header (אורך) + COMMAND|param1|param2.
  • מחלקת Protocol משותפת לשרת ולקוח = פחות באגים, יותר סדר.
