עבודת צוות עם Git ו-GitHub

איך חמישה אנשים בונים פרויקט אחד בלי לדרוך זה על הקוד של זה

מסמך זה מניח שאתם כבר מכירים את היסודות של Git — commit, push, pull, branch — מהמצגת. הוא מתמקד במה שמיוחד לעבודת צוות: איך חמישה אנשים עובדים על אותו repository בלי לשבור אחד לשני את הקוד.

למה לא כולם כותבים ישר ל-main?

אם כל חברי הצוות כותבים ישירות ל-branch הראשי (main), כל שינוי לא-בדוק עלול לשבור את הפרויקט עבור כולם בבת אחת. בעבודת צוות, ה-main חייב להיות תמיד עובד — מי שמריץ אותו (כולל המורה, כשהוא בודק) צריך לסמוך שזה לא יקרוס.

הפתרון: branch לכל תפקיד

כל חבר צוות עובד על branch נפרד בשם תחום האחריות שלו, ומשלב אותו ל-main רק כשהוא מוכן ועבר בדיקה.

feature/server

אחראי שרת

VaultServer, ClientHandler, ניהול threads

feature/client-agent

אחראי לקוח/Agent

VaultClient, לוגיקת התנהגות הלקוח

feature/protocol-security

אחראי פרוטוקול ואבטחה

ProtocolMessage, AuthManager, CryptoService

feature/ui

אחראי UI

מסכי login, דשבורד מנהל

feature/tests-docs

אחראי בדיקות ותיעוד

unit tests, README, תיק הפרויקט

שמות ה-branch הם הצעה — העיקר שיהיה שם עקבי וברור שכל הצוות מבין מיד מה נמצא בו, לפני שמסתכלים על הקוד.

Pull Request — איך branch חוזר אל main

Pull Request (PR) הוא לא עוד פעולת Git טכנית — הוא בקשה מחבר הצוות: "סיימתי לעבוד על ה-branch שלי, הקוד עובד, אני מבקש שתבדקו ותאשרו שילוב אותו ל-main". זה הרגע שבו הצוות בודק קוד אחד של השני לפני שהוא הופך לחלק מהפרויקט המשותף.

1. פיתוחעל feature/branch
←
2. פתיחת PRתיאור השינוי
←
3. סקירהחבר צוות בודק
←
4. אישור + Mergeנכנס ל-main

מה כותבים בתיאור PR טוב

  • מה השתנה (לא רק "תיקנתי באג" — אילו קבצים, אילו פעולות)
  • איך בדקתם שזה עובד (הרצה ידנית? בדיקה אוטומטית?)
  • אם יש תלות בעבודה של מישהו אחר — לציין זאת במפורש
כלל צוות מומלץ: אף אחד לא מאשר Pull Request של עצמו. תמיד חבר צוות אחר עובר על הקוד לפני שהוא נכנס ל-main — זו ה"בדיקת קוד" (code review) שגם מקצועני תוכנה אמיתיים עושים.

Issues — איך עוקבים אחרי מה צריך לעשות

Issue הוא תיאור של משימה, באג, או רעיון שצריך להתייחס אליו — לפני שמתחילים לכתוב קוד. במקום ש"מישהו יזכור בעל פה" שצריך להוסיף בדיקת path traversal, פותחים Issue, מקצים אותו לאחראי הרלוונטי, ומסמנים label.

איך כותבים Issue טוב

שדהמה כותבים בו
כותרתמשפט קצר וברור — "להוסיף בדיקת path traversal ב-VaultFileManager", לא "באג בקבצים"
תיאורמה הבעיה/המשימה, ואיך תדעו שהיא הושלמה (criteria for done)
Labelתיוג לפי תחום: server / security / ui / docs / bug
Assigneeמי מהצוות אחראי על הביצוע

Kanban Board — מבט-על על כל המשימות

לוח Kanban (ב-GitHub: Projects) מציג את כל ה-Issues במבנה עמודות, כך שכל הצוות רואה במבט אחד מי עובד על מה ומה עוד נשאר.

TO DO
ui
תכנון מסך login
docs
כתיבת README ראשוני
IN PROGRESS
server
מימוש thread-per-client
security
הגדרת ProtocolMessage
DONE
server
echo server בסיסי
לוח שמתעדכן בזמן אמת (לא רק בתחילת השלב) הוא הסימן הראשון שצוות עובד נכון — אם אף אחד לא מזיז כרטיסים בשבועיים, סימן שמשהו נתקע ולא מדברים על זה.

איך פותחים את הלוח בפועל ב-GitHub

  1. נכנסים ל-repository של הצוות, ולוחצים על הטאב Projects (ליד Code, Issues, Pull requests)
  2. לוחצים New project
  3. בוחרים בתבנית Board (זו תבנית הלוח עם עמודות To Do / In Progress / Done — מוכנה מראש)
  4. לוחצים Create — הלוח נפתח, בדרך כלל ריק או עם שלוש העמודות הבסיסיות
  5. מוסיפים את ה-Issues הקיימים ללוח: לוחצים + Add item בעמודת To Do, ומקלידים את שם ה-Issue — GitHub יציע להשלים אותו אוטומטית מתוך ה-Issues שפתחתם
  6. מכאן והלאה — כל חבר צוות גורר את הכרטיס שלו לעמודה המתאימה כשמתחיל/מסיים לעבוד עליו (גרירה רגילה עם העכבר)

חיבור אוטומטי בין Issue לכרטיס בלוח

אם פותחים Issue חדש ומוסיפים אותו ללוח (כפי שמתואר בשלב 5 מעלה) — הכרטיס בלוח מקושר לאותו Issue. זה אומר שאפשר לפתוח את הכרטיס בלוח ולערוך אותו, ואת השינוי תראו גם ב-Issue עצמו (ולהפך). אין צורך לתחזק שני מקומות נפרדים — זה אותו מידע, רק בשני תצוגות (רשימה מול לוח).

טעות נפוצה שכדאי להימנע ממנה

לא ליצור כרטיסים בלוח כטקסט חופשי בלי לקשר אותם ל-Issue אמיתי. כרטיס "תלוש" כזה לא ניתן להקצאה (assignee), לא מקבל label, ולא ניתן לעקוב אחריו בדרך שמבדק ובדק קוד — בפועל הוא רק טקסט יפה בלוח. תמיד ליצור Issue ראשון, ואז להוסיף אותו ללוח.

README.md לפרויקט צוותי

README הוא הדבר הראשון שכל מי שנכנס למאגר (כולל המורה בבדיקה) רואה. בפרויקט צוותי הוא צריך גם לשקף את חלוקת העבודה, לא רק מה הפרויקט עושה.

מוסכמות הודעת Commit בעבודת צוות

כשחמישה אנשים כותבים commits לאותו פרויקט, הודעה כמו "fix" או "update" לא עוזרת לאף אחד להבין מה קרה. מומלץ פורמט עקבי:

git commit -m "[server] add thread-per-client handling"
git commit -m "[security] implement password hashing with salt"
git commit -m "[ui] add login error message display"

התחילית בסוגריים ([server], [security]...) מאפשרת לכל אחד, בלי לפתוח את הקוד, לדעת מי תרם מה — ולמורה לראות בקלות אם כל חברי הצוות תרמו בפועל.

כשיש קונפליקט — טכני ובין-אישי

  • קונפליקט טכני (merge conflict): Git מסמן את שתי הגרסאות בקובץ — לא מנחשים, יושבים שני אחראי ה-branch יחד ומחליטים מה נשאר
  • קונפליקט בין-אישי: מישהו לא דחף קוד בזמן / כתב קוד שלא תואם להחלטה משותפת — זה לא "נושא Git", זה נושא לדיון צוותי פתוח, רצוי בנוכחות המורה אם זה חוסם התקדמות
