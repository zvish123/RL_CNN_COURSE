Magic Bytes — הבייטים שמזהים הכל

כל פורמט קובץ מתחיל בבייטים קבועים — ה-OS קורא אותם ומזהה את הסוג. לא הסיומת!

// HEX VIEWER — magic bytes highlighted
0000: FF D8 FF E0 00 10 4A 46 49 46 00 01 ...JFIF..
↑ JPEG signature
0000: 89 50 4E 47 0D 0A 1A 0A 00 00 00 0D .PNG....
↑ PNG signature
0000: 4D 5A 90 00 03 00 00 00 04 00 MZ......
↑ PE Executable (Windows EXE/DLL)
0000: 50 4B 03 04 14 00 00 00 08 00 CD 44 PK......
↑ ZIP / JAR / DOCX / APKX (כולם ZIP בפנים!)
# Python — קריאת magic bytes
with open("file.jpg", "rb") as f:
    magic = f.read(4)
    print(magic.hex())   # ffd8ffe0

# Linux — file command
$ file suspicious.exe
suspicious.exe: JPEG image data  ← שם מטעה!

# xxd / hexdump
$ xxd malware.pdf | head -2

🔴 מתקפה: Extension Spoofing

תוקף שומר malware.exe בשם vacation.jpg. User לוחץ, Windows מריץ EXE. file command מזהה את האמת.

🛡️ הגנה

בדוק תמיד magic bytes, לא סיומת. Email filters, AV — כולם קוראים signatures.

💡 Polyglot File

קובץ שתקף כשני פורמטים. GIFAR = GIF + JAR. נראה כ-GIF לנדפסר, רץ כ-Java.

// שאלת הבנה #1

זיהוי Magic Bytes

לפניך תחילת קובץ בhex. זהה את הסוג:

0000: 25 50 44 46 2D 31 2E 34 0A 25 E2 E3 %PDF-1.4.
א קובץ ZIP — PK signature
ב קובץ EXE — MZ header
ג קובץ PDF — 25 50 44 46 = "%PDF"

Hash Functions — נסה בעצמך

פונקציית hash מקבלת קובץ ומוציאה מחרוזת בגודל קבוע. שינוי בייט אחד = hash שונה לחלוטין.

MD5
128 bit
מהיר, שבור
SHA-1
160 bit
deprecated
SHA-256
256 bit
✅ מומלץ

⚠️ תכונות קריטיות

חד-כיווני — לא ניתן לחשב hash→מקור
Deterministic — אותו input = אותו output
Avalanche effect — שינוי בייט 1 = hash שונה
Collision resistant — קשה מאוד לשניים עם אותו hash

// LIVE HASH SIMULATOR

💡 נסה זאת

שנה רק אות אחת — ראה כמה ה-hash משתנה. הוסף רווח בסוף. זה ה-Avalanche Effect!

// שאלת הבנה #2

Hash Functions — מה נכון?

$ md5sum original.iso
a3f8c29b4e571d...  original.iso

$ md5sum downloaded.iso
b7d2e41a9c830f...  downloaded.iso   ← שונה!

מה ניתן להסיק?

א הקבצים זהים — MD5 לא תמיד מדויק
ב הקובץ שונה מהמקור — ייתכן שהורד קובץ פגום, זויף, או הוכנס malware
ג ה-MD5 לא יכול להשוות ISO — רק לקבצים קטנים
ד הבעיה היא ב-MD5 עצמו — צריך SHA-256
// השלם את הקוד

📝 בדיקת תקינות קובץ

כתוב Python שמחשב SHA-256 של קובץ ומשווה ל-hash ידוע:

import hashlib

def verify_file(filepath, expected_hash):
    h = hashlib.()
    with open(filepath, "") as f:
        for chunk in iter(lambda: f.read(4096), b""):
            h.update(chunk)
    actual = h.()
    return actual == expected_hash

⚠️ למה לקרוא ב-chunks ולא בבת אחת?

קבצי ISO, firmware, dumps — עשרות GB. טעינה לזיכרון שלמה = crash. chunk-by-chunk = עובד על כל גודל.

FIM — File Integrity Monitoring

FIM שומרת hash של כל קובץ בנקודת זמן ידועה. בבדיקה הבאה — כל שינוי מתגלה.

# AIDE — Advanced Intrusion Detection Env
aide --init      # יצירת baseline
aide --check     # השוואה לbaseline

# ידני ב-Python
# שלב 1: שמור snapshot
hashes = {f: sha256(f) for f in critical_files}

# שלב 2: בדוק מאוחר יותר
for f, h in hashes.items():
    if sha256(f) != h:
        alert(f"{f} שונה!")

🔴 מה מנטרים?

/etc/passwd, /etc/sudoers, /bin/, /usr/bin/, boot files, kernel modules. שינוי בלתי מורשה = indicator of compromise.

// FIM SIMULATOR — לחץ על קובץ לבדיקה

/etc/sudoers b7d241...3c92 ✓ תקין
/bin/ls c4e892...7a15 ✓ תקין

// משימה בזוגות

🛡️ בנה FIM פשוט ב-Python

בזוגות: כתבו סקריפט Python שמבצע File Integrity Monitoring.

  1. פונקציה get_hash(path) שמחזירה SHA-256 של קובץ ב-chunks
  2. פונקציה create_baseline(paths) שיוצרת dict של {path: hash} ושומרת ל-JSON
  3. פונקציה check_integrity(baseline_file) שמשווה ומדפיסה שינויים
  4. הריצו על תיקיית /etc (או תיקייה מקומית) — שנו קובץ ובדקו שה-FIM מזהה
  5. בונוס: הוסיפו timestamp לכל alert + שמירת לוג לקובץ

שחזור קבצים "נמחקים"

// מושג

File Carving — מחיקה ≠ מחיקה

כשמוחקים קובץ — ה-OS מסמן את המקום כ"פנוי". הנתונים עצמם נשארים עד שנכתב משהו חדש.

▶
// שלב 1
מחיקה — מה באמת קורה?
ה-OS מוחק רק את ה-directory entry ומסמן את ה-inode כ-free. הנתונים בבלוקים = לא נגעו. כמו למחוק תוכן עניינים של ספר — הפרקים עדיין שם.
▶
// שלב 2
איתור Signatures בדיסק גולמי
כלי כמו Foremost/Scalpel סורקים את הדיסק byte-by-byte ומחפשים magic bytes. מצאו FF D8 FF? סביר שיש JPEG. מחפשים גם ה-footer (end marker) של כל פורמט.
▶
// שלב 3
Header + Footer Carving
JPEG: header = FF D8 FF, footer = FF D9. PDF: header = %PDF, footer = %%EOF. החוקר חותך את המקטע ומשמר כקובץ. גם ללא footer — לפעמים מנחשים לפי גודל טיפוסי.
▶
// שלב 4
כלי File Carving
foremost -t jpg,pdf -i disk.img — Foremost. photorec disk.img — PhotoRec (GUI). binwalk -e firmware.bin — לחילוץ מ-firmware. Autopsy/Volatility — כלים מלאים.

🔴 חשיבות לסייבר

תוקף מחק ראיות? חוקר יכול לשחזר. Secure delete כותב אקראי מעל הנתונים (DoD 5220.22-M). SSD: wear leveling מסבך שחזור.

🔧 Foremost בפועל

foremost -t jpg,pdf,doc \
  -i /dev/sdb \
  -o /output/

💡 Slack Space

הדיסק מאורגן ב-clusters. קובץ קטן ממלא cluster שלם — הנתונים ה"נסתרים" שם = slack space. מקום נוסף לאחסון ראיות.

// שאלת הבנה #3

File Carving — תרחיש חקירה

חוקר פורנזי מקבל דיסק USB שהוחרם. תוקף טוען שמחק הכל. החוקר מריץ:

$ foremost -t jpg,pdf,zip -i usb_drive.img -o recovered/
Processing: usb_drive.img
|*****|
Finish. 47 files extracted.

מה מאפשר שחזור זה?

א ה-OS שמר גיבוי אוטומטי של הקבצים לפני המחיקה
ב USB drives לא ניתנים למחיקה אמיתית
ג מחיקה רק מסמנת מקום כ"פנוי" — הנתונים בבלוקים עדיין קיימים, Foremost מוצא אותם לפי magic bytes
ד Foremost מחזיר רק את שמות הקבצים, לא את התוכן

🏴‍☠️ File Forensics CTF

בקבוצות של 3–4: קבלתם disk image חשוד. בצעו חקירה מלאה.

  1. Signatures: הריצו file * על כל הקבצים. מצאו קבצים ששמם מטעה (EXE בשם PDF)
  2. Hashes: השוו sha256sum של קבצי מערכת לרשימה ידועה (NSRL). מה שונה?
  3. Carving: foremost -t jpg,pdf,exe -i image.dd -o out/ — מה שוחזר?
  4. Strings: strings suspicious.bin | grep -iE "http|pass|cmd"
  5. Metadata: exiftool *.jpg — יש GPS? timestamps חשודים?
  6. דוח: כתבו Incident Report קצר: מה מצאתם, timeline, המלצות
