למה בכלל Logger?

❌ בלי Logger
  • שגיאה קורית — איפה? מתי?
  • תלוי ב-print() שנשאר בקוד
  • אין היסטוריה של מה קרה
  • לא עובד ב-Production
  • המשתמש לא יכול לדווח בדיוק
✅ עם Logger
  • כל אירוע מתועד עם timestamp
  • שמור בקובץ אוטומטית
  • רמות — DEBUG עד ERROR
  • Production-ready מהיום הראשון
  • ניתן לנתח מה קרה בדיוק

בדקו את עצמכם

פרויקט הגמר שלכם רץ על מחשב של משתמש אחר ומתרסק לפעמים. לא ניתן לחבר debugger. איזו גישה הכי נכונה?
✅ Logger כותב לקובץ אוטומטית עם timestamp — גם כשאין גישה למחשב. print() נעלם מהמסך ואין היסטוריה. Debugger דורש גישה ישירה למחשב.

רמות לוג

כל רמה = מידת חשיבות. חמור יותר → ערך גבוה יותר

0 · DEBUG
מידע פיתוח — ערכים, מצבים פנימיים
log.debug(f"x = {x}")
1 · INFO
אירועים רגילים — המערכת עובדת כשורה
log.info("שרת עלה בהצלחה")
2 · WARNING
חריג אבל המערכת ממשיכה
log.warning("קובץ חסר, ברירת מחדל")
3 · ERROR
שגיאה — חלק מהמערכת נכשל
log.error("חיבור נכשל!")
💡 Development: min_level = DEBUG  |  Production: min_level = WARNING
⚡ שאלה 2

רמות לוג — מה הנכון?

משתמש מנסה להתחבר עם סיסמה שגויה פעמיים ברצף. בשלישייה — החשבון ננעל. איזו רמת לוג מתאימה לניסיון השגוי?
✅ WARNING — המערכת ממשיכה לפעול, אבל משהו חריג קורה שכדאי לשים לב אליו. ERROR יהיה מוגזם כי המערכת לא נכשלה. INFO לא מספיק דרמטי לאירוע ביטחוני.

class Logger: def __init__(self, filepath="app.log", to_screen=True, to_file=True, min_level=0): self._filepath = filepath self._to_screen = to_screen self._to_file = to_file self._min_level = min_level def debug(self, msg): self._log(0, msg) def info(self, msg): self._log(1, msg) def warning(self, msg): self._log(2, msg) def error(self, msg): self._log(3, msg) def _log(self, level, msg): # פרטי! if level >= self._min_level: # ... כתוב timestamp + הודעה
🔒 _log() פרטי — Encapsulation: המתודה הפנימית נסתרת מהמשתמש
⚡ שאלה 3

Encapsulation

בכיתת Logger, המתודה _log() מוגדרת עם קו תחתון. מה המשמעות?
✅ קו תחתון אחד = מוסכמה ב-Python שהמתודה פרטית (Private). זה חלק ממושג Encapsulation — מסתירים את הלוגיקה הפנימית. Python לא אוכף זאת בכוח, אבל זה signal למפתחים לא להשתמש בה ישירות.
📡 הרצאה

Context Manager — with Logger()

מה קורה אם בתוך הבלוק נזרקת Exception?
✅ __exit__ נקרא תמיד — זה הסוד של Context Manager! גם אם יש Exception, הקובץ ייסגר. זה מבטיח שאין "דליפת משאבים" (Resource Leak). ה-Exception עצמה עדיין תיזרק (כי __exit__ מחזיר False).

class Logger: # Facade def __init__(self, name="app"): self._logger = logging.getLogger(name) self._setup_handlers() def info(self, msg): self._logger.info(msg) def _setup_handlers(self): # כל המורכבות כאן — פנימי fh = logging.FileHandler("app.log") fmt = logging.Formatter("%(asctime)s %(message)s") fh.setFormatter(fmt) self._logger.addHandler(fh)
🎭 למה Facade?
  • ממשק פשוט: רק log.info()
  • המורכבות מוסתרת בפנים
  • קל להחליף ספרייה בעתיד
  • Design Pattern מוכר בתעשייה
💡 דוגמה מוכרת: requests.get(url) — Facade מעל HTTP
⚡ שאלה 5

Facade Pattern

Facade Pattern מציע: "ספק ממשק פשוט לתת-מערכת מורכבת". מה עיקר היתרון של זה בפרויקט גמר?
✅ אם מחר תרצו לעבור מ-logging המובנה ל-loguru או כל ספרייה אחרת — תשנו רק את ה-Facade (הכיתה), לא את כל הקוד שמשתמש ב-log.info(). זה עיקר עוצמת ה-Facade.
📡 הרצאה

Decorator — @log_calls

במקום להוסיף log בכל פונקציה — עוטפים אותה פעם אחת

decorator.py
import functools, logging log = logging.getLogger("app") def log_calls(func): @functools.wraps(func) def wrapper(*args, **kwargs): log.info(f"▶ {func.__name__}() נקרא") try: result = func(*args, **kwargs) log.info(f"✓ {func.__name__}() הסתיים") return result except Exception as e: log.error(f"✗ {func.__name__}(): {e}") raise return wrapper
שימוש — שורה אחת!
@log_calls # ← זהו! def load_data(filepath): # קוד רגיל... return data @log_calls def connect_db(host): # ...
פלט אוטומטי
▶ load_data() נקרא ✓ load_data() הסתיים ▶ connect_db() נקרא ✗ connect_db(): Connection refused
🎯 Separation of Concerns — לא נוגעים בקוד הפנימי של הפונקציה
⚡ שאלה 6

Decorator @log_calls

יש לכם 20 פונקציות בפרויקט ואתם רוצים לוג בכניסה וביציאה מכל פונקציה. בלי Decorator, כמה שורות לוג תצטרכו להוסיף לכל הפחות?
✅ לפחות 40 שורות — שורת log בכניסה ושורת log ביציאה לכל אחת מ-20 הפונקציות. עם @log_calls? רק 21 שורות: הגדרת ה-Decorator פעם אחת + @log_calls מעל כל פונקציה. כוח ה-DRY — Don't Repeat Yourself!

Design Patterns — סיכום

בפרויקט גמר, יצרנו class Logger שעוטפת את מודול logging ומספקת ממשק פשוט של log.info(), log.error(). איזה Design Pattern זה?
✅ Facade — מספק ממשק פשוט לתת-מערכת מורכבת. Decorator הוא @log_calls שמוסיף התנהגות. Singleton מבטיח מופע אחד. Observer מודיע על שינויים.
⚡ שאלה 8

שאלת בונוס — מתקדם

נניח שהפרויקט יוצר כמה מופעים של Logger: log1 = Logger("a.log") ו-log2 = Logger("b.log"). איזה Pattern נוסף נרצה לממש כדי לוודא שיש רק Logger אחד בכל הפרויקט?
✅ Singleton — מבטיח שנוצר מופע אחד בלבד של הכיתה. כל קריאה ל-Logger() מחזירה את אותו האובייקט. זה הנושא הבא שלנו!
