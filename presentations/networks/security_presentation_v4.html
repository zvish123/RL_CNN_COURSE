מערכת מאובטחת לא נשענת על טריק בודד. היא בנויה משכבות: הגנה על הסיסמאות, הגנה על התעבורה ברשת, זיהוי נכון של המשתמש, והגנה על מידע רגיש שנשמר במסד הנתונים.

שלב 7A
bcrypt — הגנה על סיסמאות
לא לשמור סיסמאות כטקסט גלוי — אלא כ-hash חד-כיווני שלא ניתן לפענח

🎯 המטרה

גם אם קובץ ה-DB דולף, הסיסמאות לא קריאות ישירות. שומרים hash — לא הסיסמה עצמה.

❓ למה זה נדרש?

  • מסד נתונים עלול להיחשף
  • סיסמה היא מידע שאסור לשחזר
  • בדיקת login — מול hash, לא טקסט

🔬 איך bcrypt עובד — הסבר מורחב

bcrypt הוא אלגוריתם hashing חד-כיווני. הוא לא מצפין — הוא ממיר את הסיסמה לייצוג בלתי-הפיך.

קלטסיסמה: "1234"
←
Saltמחרוזת אקראית ייחודית
←
Hashingאלגוריתם bcrypt (12 rounds)
←
פלט$2b$12$abc...xyz (60 תווים)
1
Salt אקראי bcrypt מייצר מחרוזת אקראית (salt) לכל סיסמה. כך שתי משתמשים עם אותה סיסמה מקבלים hash שונה לחלוטין — מה שמנטרל מתקפות rainbow table.
2
Cost factor — מכוון איטי bcrypt מבצע את החישוב 2cost פעמים (ברירת מחדל: 12 rounds = 4,096 איטרציות). זה לוקח ~100ms למחשב רגיל. לתוקף שמנסה מיליוני סיסמאות — זה הופך brute-force לבלתי-כדאי.
3
בדיקת login — השוואה ולא פענוח כשמשתמש מתחבר, bcrypt לוקח את הסיסמה שהוזנה + את ה-salt מה-hash השמור, מחשב מחדש, ומשווה. אין פענוח — יש hash → hash.
4
חד-כיווני לחלוטין אין דרך מתמטית לשחזר את הסיסמה המקורית מה-hash. זו תכונה מובנית — ולא באג. 
import bcrypt

# הרשמה — יצירת hash מהסיסמה
password = "1234"
hashed = bcrypt.hashpw(password.encode("utf-8"), bcrypt.gensalt())
# hashed נשמר במסד הנתונים (לא "1234"!)
# לדוגמה: b'$2b$12$EixZaYVK1fsbw1ZfbX3OXe.PkAbm...'

# התחברות — בדיקת התאמה (ולא פענוח)
entered_password = "1234"
is_valid = bcrypt.checkpw(entered_password.encode("utf-8"), hashed)
# is_valid == True אם הסיסמה נכונה

שלב 7B
TLS — הגנה על התעבורה
הצפנת ערוץ התקשורת בין הלקוח לשרת — כדי שאף מאזין לא יוכל לקרוא את המידע

🎯 המטרה

להצפין את המידע בזמן המעבר ברשת: login, הודעות, tokens וכל מידע אחר.

❓ למה זה נדרש?

  • הודעות ברשת יכולות להיצפות (Man in the Middle)
  • גם JWT וגם password עוברים בתעבורה
  • ללא TLS, מאזין לרשת יקרא הכול

🔬 איך TLS עובד — הסבר מורחב

TLS הוא פרוטוקול שמקים ערוץ מאובטח בין הלקוח לשרת. הוא משלב שני סוגי הצפנה: אסימטרית (לחילופי מפתחות) וסימטרית (לתעבורה עצמה).

לקוחClient Hello
←
שרתCertificate + Public Key
←
לקוחמאמת cert + מחשב Session Key
←
שניהםתעבורה מוצפנת בסימטרי
1
TLS Handshake — לחיצת יד הלקוח שולח Client Hello עם רשימת אלגוריתמים שתומך בהם. השרת בוחר אלגוריתם ושולח בחזרה Server Hello.
2
Certificate — תעודת זהות של השרת השרת שולח Certificate חתום על ידי רשות מהימנה (CA). הלקוח מאמת שאכן מדובר בשרת הנכון ולא בתוקף שמתחזה.
3
Key Exchange — חילוף מפתחות (הצפנה אסימטרית) הלקוח מייצר Session Key, מצפין אותו עם ה-Public Key של השרת ושולח. רק השרת יכול לפענח עם ה-Private Key שלו. כעת שניהם יודעים את Session Key — בלי לשלוח אותו בגלוי.
4
Symmetric Encryption — הצפנה סימטרית לתעבורה מכאן והלאה, כל התעבורה מוצפנת עם ה-Session Key הסימטרי (למשל AES). הצפנה סימטרית הרבה יותר מהירה — לכן משתמשים בה לנתונים בפועל. 
Client ──── ClientHello ──────────────────────► Server
             (TLS version, cipher suites)

Client ◄─── ServerHello + Certificate ───────── Server
             (chosen cipher, public key)

Client ──── ClientKeyExchange ────────────────► Server
             (Session Key, encrypted with Server's Public Key)

Client ◄════ Encrypted Application Data ══════► Server
             (all traffic now encrypted with Session Key)
🧠 שאלת הבנה
מדוע TLS משתמש גם בהצפנה אסימטרית (RSA/Diffie-Hellman) וגם בהצפנה סימטרית (AES), ולא רק בסוג אחד?

🎯 המטרה

לאחר login — לזהות את המשתמש בכל בקשה עתידית בלי לשלוח שוב סיסמה.

❓ למה זה נדרש?

  • להפריד בין login לבין פעולות בהמשך
  • לנהל זהות משתמש ב-API
  • לבדוק תוקף התחברות ו-expiration

🔬 איך JWT עובד — הסבר מורחב

JWT = JSON Web Token. הוא מורכב משלושה חלקים מופרדים בנקודה: Header.Payload.Signature

Header{"alg":"HS256","typ":"JWT"}
.
Payload{"username":"alice","exp":...}
.
SignatureHMAC(header+payload, SECRET_KEY)
1
Header מכיל מידע על סוג ה-token ואלגוריתם החתימה. מקודד ב-Base64 (לא מוצפן — ניתן לקריאה).
2
Payload — המידע מכיל claims: מי המשתמש (username), מתי פג תוקף (exp), הרשאות וכו'. גם זה מקודד ב-Base64 ולא מוצפן — אל תשים בו סיסמאות!
3
Signature — החתימה השרת מחשב HMAC על Header+Payload עם SECRET_KEY. אם מישהו ישנה את ה-Payload, החתימה תיפסל — השרת יזהה את הזיוף.
4
Stateless — השרת לא שומר sessions כל המידע הנדרש נמצא בתוך ה-token. השרת רק מאמת את החתימה ובודק תוקף — אין צורך בשאילתת DB לכל בקשה.
⚠️ JWT לא מוצפן — הוא חתום. כל אחד יכול לקרוא את ה-Payload. לכן הוא חייב לנסוע מעל TLS, ואסור לשמור בו מידע רגיש.
import jwt, datetime

# יצירת token בעת login מוצלח
token = jwt.encode(
    {
        "username": "alice",
        "exp": datetime.datetime.utcnow() + datetime.timedelta(hours=1)
    },
    SECRET_KEY,
    algorithm="HS256"
)
# token נשלח ללקוח — הוא ישמור אותו ויצרף לכל בקשה

# אימות token בכל בקשה מוגנת (בשרת)
try:
    payload = jwt.decode(token, SECRET_KEY, algorithms=["HS256"])
    username = payload["username"]   # זהות המשתמש
except jwt.ExpiredSignatureError:
    print("Token פג תוקפו")
except jwt.InvalidTokenError:
    print("Token לא תקין — אולי זויף")
🧠 שאלת הבנה
תוקף שיירט JWT token יכול לקרוא את תוכן ה-Payload (Username, תוקף וכו'). מה הוא לא יכול לעשות?
שלב 9
Fernet — הצפנת מידע רגיש
הגנה על שדות רגישים שנשמרים במסד הנתונים — הצפנה הפיכה שניתן לשחזר בעת הצורך

🎯 המטרה

להגן על שדות רגישים (private note, email וכו') כך שגם אם ה-DB נחשף — המידע לא קריא ישירות.

❓ למה זה נדרש?

  • DB דלף → מידע רגיש לא קריא
  • bcrypt לא מתאים — אנחנו צריכים לקרוא את המידע בחזרה
  • הצפנה סימטרית: מהירה ומאובטחת

🔬 איך Fernet עובד — הסבר מורחב

Fernet היא ספרייה להצפנה סימטרית — אותו מפתח משמש להצפנה ולפענוח. היא משתמשת ב-AES-128-CBC + HMAC לאימות שלמות.

טקסט גלוי"my private note"
←
Key + IV256-bit key, IV אקראי
←
AES-CBCהצפנת בלוקים
←
HMACחתימה לאימות
←
TokenBase64 מוצפן
1
המפתח — 256 ביט (32 בייטים) Fernet מייצר מפתח אקראי 256-bit. המפתח הוא הסוד האמיתי — מי שמחזיק אותו יכול לפענח הכול. שומרים אותו מחוץ לקוד ב-environment variable.
2
IV — Initialization Vector אקראי Fernet מוסיף IV אקראי לכל הצפנה. כמו ה-salt ב-bcrypt — מבטיח שאותו טקסט מוצפן יניב תוצאה שונה בכל פעם.
3
AES-128-CBC — ההצפנה עצמה המידע מחולק לבלוקים של 128 ביט ומוצפן בשרשרת (CBC = Cipher Block Chaining). כל בלוק תלוי בבלוק שלפניו.
4
HMAC-SHA256 — אימות שלמות Fernet מחשב HMAC על הנתונים המוצפנים. בעת פענוח, מאמת שהנתונים לא שונו — מגן מפני tampering.
5
Timestamp — תוקף אופציונלי ה-token כולל timestamp של יצירה. ניתן להגדיר TTL (time-to-live) כך שה-token יפוג לאחר זמן מסוים.
🆚 bcrypt vs Fernet: bcrypt = חד-כיווני (לסיסמאות). Fernet = דו-כיווני (למידע שצריך לשחזר). לעולם אל תצפין סיסמאות עם Fernet!
from cryptography.fernet import Fernet
import os

# יצירת מפתח — פעם אחת, שמור ב-env variable
key = Fernet.generate_key()
# key = b'abc123...' (32 bytes, base64url encoded)
fernet = Fernet(key)

# הצפנה — לפני שמירה במסד
plaintext = "my private note"
encrypted = fernet.encrypt(plaintext.encode("utf-8"))
# encrypted = b'gAAAAABh...' — זה מה ששמור בDB

# פענוח — בעת קריאה מה-DB
decrypted = fernet.decrypt(encrypted).decode("utf-8")
# decrypted == "my private note"

# שימוש נכון — מפתח מ-environment
KEY = os.environ.get("FERNET_KEY")
fernet = Fernet(KEY)

מבחן הבנה
שאלות הבנה — כלליות
שאלות שמחברות בין כל המנגנונים — לחצו על שאלה לחשיפת תשובה
תוקף שמאזין לרשת יוכל לגנוב את ה-JWT token. מכיוון שה-token מזהה את המשתמש, התוקף יוכל להתחזות אליו עד שפג תוקפו. JWT לא מחליף TLS — הוא חייב לנסוע בתוך ערוץ מוצפן.
bcrypt הוא hash חד-כיווני — לא ניתן לשחזר את הטקסט המקורי. לסיסמה זה מושלם כי אנחנו רק צריכים לבדוק התאמה. אבל private note צריך להיות קריא בחזרה — לכן משתמשים ב-Fernet שמאפשר פענוח.
לדוגמה: כתובת email, מספר טלפון, תוכן הודעות פרטיות, מספר ת"ז. כל שדה שהוא מידע רגיש אישי, שצריך להיות קריא בעתיד (לכן לא hash), ושנרצה להגן עליו גם אם ה-DB דולף.
סימטרית (למשל AES, Fernet): אותו מפתח מצפין ומפענח. מהיר אבל מחייב שניהם להחזיק מפתח משותף. אסימטרית (למשל RSA): זוג מפתחות Public+Private. Public מצפין, Private מפענח. בטוח לחילופי מפתח אבל איטי. TLS משתמש בשניהם.
הגנה בשכבות — אם שכבת אבטחה אחת נפרצת, שאר השכבות ממשיכות להגן. לדוגמה: אם TLS מתפשר, עדיין bcrypt מגן על הסיסמאות ו-Fernet על המידע הרגיש. אין נקודת כשל אחת שמפילה את הכול.
סיכום
מה חשוב לזכור
ארבעה מנגנונים, ארבע בעיות שונות, הגנה אחת משולבת
  • 🔑 bcrypt — hash חד-כיווני לסיסמאות. לא ניתן לפענח. Salt + Cost factor.
  • 🌐 TLS — מצפין את ערוץ התקשורת. Handshake ← Session Key ← AES.
  • 🪪 JWT — token חתום לזיהוי משתמש. Header.Payload.Signature. חייב על TLS.
  • 🗄️ Fernet — AES-128-CBC + HMAC לשדות רגישים. דו-כיווני. מפתח = הסוד.
שאלות יישום לתרגול
יישום 1

כתבו במילים שלכם מה יקרה אם המערכת תשתמש ב-JWT אבל לא ב-TLS.

יישום 2

הסבירו למה bcrypt מתאים לסיסמאות אבל לא מתאים ל-private note.

יישום 3

הציעו שדה אחד נוסף שצריך להצפין עם Fernet והסבירו למה.
