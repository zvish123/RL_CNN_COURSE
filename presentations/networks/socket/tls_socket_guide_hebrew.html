שימוש ב־TLS בקשר שרת–לקוח עם סוקטים

במדריך זה נלמד מהו TLS, למה משתמשים בו, איך יוצרים תעודה לצורך עבודה מעשית, ואיך משלבים TLS בקוד פייתון המבוסס על socket.

מה נלמד?

  • ההבדל בין תקשורת רגילה לתקשורת מאובטחת
  • מה תפקידם של תעודה ומפתח פרטי
  • איך TLS מתלבש על סוקט TCP קיים
  • איך כותבים שרת ולקוח מאובטחים בפייתון

תוצרים צפויים

  • שרת TLS בסיסי
  • לקוח TLS בסיסי
  • הבנה של שלבי ההתחברות
  • יכולת להסביר טעויות אבטחה נפוצות

1. למה בכלל צריך TLS?

כאשר לקוח ושרת מתקשרים באמצעות TCP רגיל, הנתונים עלולים לעבור ברשת באופן גלוי. מי שמאזין לתעבורה עלול לקרוא הודעות, סיסמאות או מידע רגיש אחר.

לקוח TCP רגיל שרת

TLS מוסיף שכבת אבטחה מעל TCP. אחרי יצירת החיבור מתבצע תהליך הזדהות והסכמה על מפתחות, ורק לאחר מכן נשלחים הנתונים כשהם מוצפנים.

לקוח TLS TCP TLS שרת
רעיון מרכזי: TLS לא מחליף את הסוקט, אלא עוטף אותו ומגן על המידע שעובר דרכו.

2. מטרות TLS

מטרה הסבר
סודיות המידע מוצפן כך שמאזין ברשת לא יוכל לקרוא אותו.
שלמות ניתן לזהות אם המידע שונה בדרך.
אימות הלקוח יכול לוודא מול מי הוא מדבר, באמצעות תעודה דיגיטלית.
חשוב: TLS מגן על תוכן התקשורת, אבל לא מסתיר את כתובת ה־IP או את מספר הפורט.

3. מהי תעודה דיגיטלית?

כדי שהלקוח יוכל להאמין שהוא מדבר עם השרת הנכון, השרת מציג תעודה דיגיטלית. התעודה כוללת מידע מזהה ומפתח ציבורי. בנוסף, לשרת יש מפתח פרטי שמתאים לתעודה.

cert.pem

קובץ התעודה של השרת. ניתן לשלוח ללקוח לצורך אימות.

key.pem

המפתח הפרטי של השרת. אסור לחשוף אותו לאחרים.

כלל זהב: את המפתח הפרטי שומרים בסוד. אם הוא דולף, הזהות של השרת בסכנה.

4. איך יוצרים תעודה לצורך תרגול?

לצורך עבודה בכיתה או במעבדה אפשר ליצור תעודה עצמית (Self-Signed Certificate). היא מתאימה לתרגול, אך אינה מתאימה למערכת אמיתית באינטרנט. 

openssl req -x509 -newkey rsa:2048 -keyout key.pem -out cert.pem -days 365 -nodes

הפקודה יוצרת שני קבצים:

  • cert.pem – תעודת השרת
  • key.pem – המפתח הפרטי
אם OpenSSL לא מותקן במחשב, אפשר להתקין אותו או לעבוד דרך WSL / Linux.

5. הרעיון המעשי בקוד פייתון

  1. יוצרים סוקט TCP רגיל.
  2. מגדירים אובייקט הקשר של TLS באמצעות ssl.SSLContext.
  3. טוענים לשרת את התעודה והמפתח הפרטי.
  4. עוטפים את הסוקט באמצעות wrap_socket.
  5. מכאן והלאה עובדים עם הסוקט המאובטח.

6. קוד שרת TLS בפייתון

import socket
import ssl

HOST = '0.0.0.0'
PORT = 8443

# יצירת סוקט TCP רגיל
server_socket = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
server_socket.bind((HOST, PORT))
server_socket.listen(5)

# יצירת הקשר TLS לשרת
context = ssl.SSLContext(ssl.PROTOCOL_TLS_SERVER)
context.load_cert_chain(certfile='cert.pem', keyfile='key.pem')

print(f'Server listening on {HOST}:{PORT}')

while True:
    client_socket, addr = server_socket.accept()
    print('New connection from', addr)

    # עטיפת הסוקט ב-TLS
    secure_socket = context.wrap_socket(client_socket, server_side=True)

    data = secure_socket.recv(1024)
    print('Received:', data.decode())

    secure_socket.send(b'Hello secure client!')
    secure_socket.close()

הסבר קצר

  • ssl.PROTOCOL_TLS_SERVER – מגדיר שההקשר מיועד לשרת.
  • load_cert_chain – טוען את התעודה והמפתח הפרטי.
  • wrap_socket – הופך את סוקט ה־TCP לחיבור TLS.

7. קוד לקוח TLS בפייתון

import socket
import ssl

HOST = '127.0.0.1'
PORT = 8443

# יצירת הקשר TLS ללקוח
context = ssl.create_default_context()

# לצורכי תרגול בלבד - לא מתאים למערכת אמיתית
context.check_hostname = False
context.verify_mode = ssl.CERT_NONE

with socket.create_connection((HOST, PORT)) as sock:
    with context.wrap_socket(sock, server_hostname=HOST) as secure_sock:
        secure_sock.send(b'Hello server')
        data = secure_sock.recv(1024)
        print('Received:', data.decode())
אזהרה מקצועית: ההגדרות CERT_NONE ו־check_hostname = False נועדו לניסוי בלבד. במערכת אמיתית צריך לאמת את התעודה של השרת.

8. איך עובדים נכון במערכת אמיתית?

במערכת אמיתית נרצה שהלקוח יוודא שהוא מדבר עם השרת האמיתי.

  • שימוש ב־CERT_REQUIRED
  • בדיקת שם המארח (hostname)
  • עבודה עם תעודה חתומה על ידי גוף אמין או CA ארגוני
context = ssl.create_default_context(cafile='cert.pem')
with socket.create_connection((HOST, PORT)) as sock:
    with context.wrap_socket(sock, server_hostname=HOST) as secure_sock:
        secure_sock.send(b'Hello secure server')

בדוגמה זו הלקוח סומך על התעודה הנתונה באמצעות cafile, ולכן אין צורך לבטל אימות.

9. שלבי העבודה בקשר TLS

  1. הלקוח יוצר חיבור TCP לשרת.
  2. השרת שולח תעודה ללקוח.
  3. הלקוח בודק את התעודה.
  4. הצדדים מסכימים על מפתחות הצפנה.
  5. מכאן התקשורת מוצפנת.
כל השלבים האלו מתרחשים לפני שהאפליקציה מתחילה לשלוח את ההודעות הרגילות שלה.

10. טעויות נפוצות

טעות למה זו בעיה?
שימוש ב־CERT_NONE בפרודקשן הלקוח לא באמת מאמת את זהות השרת.
שמירת key.pem בתיקייה חשופה המפתח הפרטי עלול לדלוף.
מחשבה ש־TLS מסתיר IP ופורט TLS מצפין את התוכן, לא את פרטי הרשת הבסיסיים.
שכחת עטיפת הסוקט ב־TLS החיבור נשאר TCP רגיל וללא הצפנה.

בנו תוכנת צ'אט קטנה בין לקוח לשרת שבה כל התקשורת נעשית מעל TLS. דרישות מינימום:

  • השרת מקבל כמה הודעות מאותו לקוח
  • הלקוח יכול להקליד מספר הודעות

  • הסבירו בכתב מהי נקודת החולשה אם לא בודקים תעודה

14. סיכום

TLS הוא מנגנון קריטי בהגנה על תקשורת בין לקוח לשרת. הוא אינו מחליף סוקטים, אלא מוסיף להם שכבת אבטחה. כדי להשתמש בו נכון צריך להבין גם את התאוריה וגם את המימוש: תעודה, מפתח פרטי, אימות, והעטפת הסוקט בקוד.

תלמיד שמבין את המדריך הזה אמור לדעת להסביר מה TLS עושה, איך יוצרים סביבת ניסוי, ואיך משלבים TLS בקשר שרת–לקוח בפייתון.
