לוגיקת שרת בסיסית — JSON & List

ניהול משתמשים, קובץ JSON, רשימת לקוחות מחוברים

מושגי יסוד
📡
Socket + recv
ה-socket מקבל מחרוזת טקסט מהלקוח. recv(1024) מחזיר bytes שיש לפענח.
📂
users.json
שמירת משתמשים (שם + סיסמה) כקובץ JSON על הדיסק.
📋
clients [ ]
רשימה גלובלית של לקוחות מחוברים כרגע — socket + שם משתמש.
🔀
פרוטוקול action||
כל הודעה מתחילה בפעולה ואחריה פרמטרים: register||zvi||1234
פרוטוקול הפקודות
פקודה פורמט פעולה בשרת
register||user||pass רישום בדיקה אם קיים ב-JSON → הוספה אם לא → שמירה
login||user||pass התחברות אימות מול JSON → הוספת socket לרשימת clients
message||text הודעה קבלת הודעה → אישור "Message received"
logout התנתקות הסרה מרשימת clients → סגירת החיבור
קוד עזר — שלד handle_client
server.py
def handle_client(conn, addr):
    while True:
        data = conn.recv(1024).decode()
        parts = data.split("||")
        action = parts[0]

        if action == "register":
            do_register(parts[1], parts[2])
        elif action == "login":
            do_login(conn, parts[1], parts[2])
        elif action == "message":
            conn.send(b"Message received")
        elif action == "logout":
            do_logout(conn); break

תקשורת דו-כיוונית — Threads & Broadcast

Threads בלקוח, הפצת הודעות לכולם, נעילה למניעת Race Condition

הבעיה ופתרונה
🚫
הבעיה
recv() חוסם — אם הלקוח מחכה לקלט, הוא לא יכול לקבל הודעות נכנסות.
🔁
Thread ראשי
input() + שליחה לשרת. ממתין לקלט המשתמש.
👂
Thread האזנה
לולאה אינסופית עם recv() — מדפיס הודעות ברגע שמגיעות.
📢
broadcast()
שולח הודעה לכל הלקוחות המחוברים, חוץ מהשולח עצמו.
🔒
threading.Lock
מונע Race Condition כאשר מספר Threads ניגשים ל-clients[] ול-JSON במקביל.
קוד עזר — broadcast + Lock
server.py
clients_lock = threading.Lock()

def broadcast(data_dict, exclude_sock):
    with clients_lock:
        for client in clients:
            if client["socket"] != exclude_sock:
                try:
                    client["socket"].send(
                        json.dumps(data_dict).encode()
                    )
                except:
                    pass
client.py — Thread האזנה
def listen_from_server(sock):
    while True:
        msg = sock.recv(1024).decode()
        print(f"\n[הודעה] {msg}")

t = threading.Thread(target=listen_from_server, args=(sock,))
t.daemon = True
t.start()
⚠️ ה-Lock נדרש גם כשקוראים את הרשימה וגם כשכותבים אליה. גישה בלי Lock עלולה לגרום לתוצאות בלתי צפויות.
✏️   משימה — שלב 2
  • 1הוסיפו Thread האזנה בלקוח עם daemon=True
  • 2ממשו broadcast(data, exclude_sock) בשרת עם Lock
  • 3קראו ל-broadcast מתוך handle_client כשמתקבלת הודעת message
  • 4הגנו על כל גישה לרשימת clients ולקובץ JSON עם threading.Lock()
  • 5בדקו: משתמש א' שולח → משתמש ב' מקבל ללא שביצע פעולה

פרוטוקול תקשורת — JSON Protocol & Framing

TCP Stream, מפריד \n, Buffer, json.loads

הבעיה — TCP Stream

TCP הוא זרם רציף — אין "גבול" בין הודעות. שתי הודעות שנשלחות מהר עלולות להגיע כחתיכה אחת אצל הנמען.

שולח: הודעה 1
+
שולח: הודעה 2
→
recv מקבל: הודעה1הודעה2

הפתרון — Delimited Framing: נוסיף \n בסוף כל הודעה. הנמען קורא עד שהוא רואה \n — ואז הוא יודע שהגיע הודעה שלמה.

פורמט ההודעה החדש
פורמט — JSON + newline
# כל הודעה: JSON object + תו \n
{"action": "login", "username": "zvi", "password": "1234"}\n
{"action": "message", "text": "שלום!"}\n
client.py — שליחה
msg = {"action": "login", "username": "zvi", "password": "1234"}
sock.send((json.dumps(msg) + "\n").encode())
server.py — קבלה עם Buffer
buffer = ""

while True:
    chunk = conn.recv(4096).decode()
    buffer += chunk

    while "\n" in buffer:
        line, buffer = buffer.split("\n", 1)
        data = json.loads(line.strip())   # dict מלא
        handle_message(data)
⚠️ recv(4096) יכול להחזיר חצי הודעה או הודעה וחצי! ה-Buffer מבטיח שנעבד רק הודעות שלמות.
✏️   משימה — שלב 3
  • 1שנו את הלקוח: כל שליחה תהיה JSON + "\n"
  • 2שנו את השרת: קריאה לתוך buffer עד "\n", פיענוח עם json.loads(line.strip())
  • 3ודאו ש-broadcast גם שולח JSON + "\n"
  • 4בדקו edge case: שלחו 3 הודעות רצופות מהר — האם כולן מתקבלות בנפרד?

בסיס נתונים — SQLAlchemy & SQLite

החלפת users.json ב-ORM מקצועי: מודל, session, שאילתות

למה לעבור ל-DB?
⚛️
עסקאות אטומיות
ה-DB מנהל נעילה פנימית — אין צורך ב-Lock על קובץ.
⚡
חיפוש מהיר
filter_by() עם אינדקס — גם מיליון שורות במילישניות.
🧩
Python Objects
עובדים עם objects מוכרים — לא JSON ידני ולא SQL גולמי.
💾
עמידות
הנתונים נשמרים גם אחרי הפעלה מחדש של השרת.
קוד עזר — הגדרת מודל
server.py — setup
from sqlalchemy import create_engine, Column, Integer, String
from sqlalchemy.orm import declarative_base, sessionmaker

Base = declarative_base()

class User(Base):
    __tablename__ = "users"
    id       = Column(Integer, primary_key=True)
    username = Column(String, unique=True)
    password = Column(String)

engine  = create_engine("sqlite:///chat.db")
Base.metadata.create_all(engine)
Session = sessionmaker(bind=engine)
server.py — Register & Login
def do_register(username, password):
    session = Session()
    existing = session.query(User).filter_by(username=username).first()
    if not existing:
        session.add(User(username=username, password=password))
        session.commit()
        return "registered"
    return "exists"

def do_login(username, password):
    session = Session()
    user = session.query(User).filter_by(
        username=username, password=password
    ).first()
    return user is not None
⚠️ הסירו את ה-Lock שהגנתם על קובץ JSON — ה-DB מנהל נעילה בעצמו. שמרו את ה-Lock על רשימת ה-clients!

  • 1הגדירו מודל User עם שדות: id, username, password
  • 2החליפו את כל קריאות users.json ב-session.add() ו-filter_by()
  • 3הסירו את ה-Lock על ה-JSON (שמרו Lock רק על clients[])
  • 4הוסיפו הדפסות [DB] לכל פעולת DB
