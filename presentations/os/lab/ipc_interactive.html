למה IPC? Thread vs Process

🧵

Thread

חי בתוך אותו process. חולק זיכרון אוטומטית עם כל ה-threads האחרים. תקשורת = פשוטה, אבל מסוכנת (Race Conditions).

🏠

Process

כל process חי בבועה מבודדת. OS נותן לו זיכרון נפרד לחלוטין. מאובטח יותר — אבל לא יכול לגעת בזיכרון של process אחר.

// מושג

Pipe — צינור נתונים

🔧 אנלוגיה: אינסטגרם DM — process אחד שולח, השני מקבל. חד-כיווני, מאגן, בלוקינג.

Process A
writer
Pipe Buffer (kernel)
Process B
reader
from multiprocessing import Process, Pipe

def sender(conn):
    conn.send("שלום מ-Process A! 👋")
    conn.close()

def receiver(conn):
    msg = conn.recv()  # חוסם עד קבלה
    print(f"קיבלתי: {msg}")

parent_conn, child_conn = Pipe()
p1 = Process(target=sender,   args=(child_conn,))
p2 = Process(target=receiver, args=(parent_conn,))
p1.start(); p2.start()
p1.join();  p2.join()

✅ יתרונות Pipe

פשוט, מהיר, מובנה ב-OS. מאגן אוטומטי. בלוקינג = סנכרון חינם.

❌ מגבלות

חד-כיווני (Pipe רגיל). רק בין processes קשורים (parent/child). אין persistence.

💡 duplex=True

Pipe(duplex=True) — תקשורת דו-כיוונית. כל קצה יכול לשלוח ולקבל.

// מושג

Named Pipe — FIFO

Named Pipe = Pipe עם שם בפסיסטם — כל process יכול להתחבר, גם ללא קשר משפחתי (parent/child).

import os, time

FIFO = "/tmp/my_pipe"

# יוצר Named Pipe בפסיסטם
if not os.path.exists(FIFO):
    os.mkfifo(FIFO)

## --- Process A: כותב ---
with open(FIFO, 'w') as f:
    f.write("secret_data\n")

## --- Process B (נפרד): קורא ---
with open(FIFO, 'r') as f:
    data = f.readline()
    print(f"קיבלתי: {data}")

os.unlink(FIFO)  # ניקוי

🔐 קשר לסייבר

Named Pipes ב-Windows ידועים כווקטור תקיפה — privilege escalation דרך impersonation של Named Pipe. (CVE-2021-1732)

Pipe — מה יקרה?

parent_conn, child_conn = Pipe()

def child(conn):
    conn.send(42)
    conn.send(99)

p = Process(target=child, args=(child_conn,))
p.start()

result = parent_conn.recv()
print(result)   # מה יודפס?
p.join()
א שגיאה — לא ניתן לשלוח שני ערכים ב-Pipe
ב 99 — recv() מקבל את ההודעה האחרונה
ג 42 — Pipe הוא FIFO, recv() מקבל את הראשונה
ד (42, 99) — recv() מקבל הכל בבת אחת

בנה תקשורת Pipe דו-כיוונית

from multiprocessing import Process, Pipe

def process_b(conn):

parent, child = Pipe()

parent.send("מה שלומך?")

multiprocessing.Queue

כמו queue.Queue שאתם מכירים — אבל עובד בין processes נפרדים, לא רק threads. הסריאליזציה נעשית אוטומטית ב-pickle.

from multiprocessing import Process, Queue

def producer(q):
    for i in range(5):
        q.put({"id": i, "data": f"packet_{i}"})
        print(f"[P] שלח packet_{i}")
    q.put(None)  # סימן עצירה

def consumer(q):
    while True:
        item = q.get()
        if item is None: break
        print(f"[C] עיבד: {item['data']}")

q = Queue(maxsize=3)
Process(target=producer, args=(q,)).start()
Process(target=consumer, args=(q,)).start()

Queue vs Pipe

Queue — כמה producers/consumers. Pipe — רק 2 קצוות. Queue גמיש יותר.

⚠️ pickle

כל מה שנשלח ב-Queue עובר סריאליזציה. אובייקטים שלא ניתנים ל-pickle יגרמו לשגיאה.

🔐 סייבר

Deserialization of untrusted data = וקטור תקיפה נפוץ. לעולם אל תעשה pickle.loads() על נתון לא מאומת!

📨 Message Pipeline

בזוגות: בנו pipeline של 3 processes — כל אחד מעבד ומעביר הלאה.

  1. Process A: מייצר רשימת מספרים 1–10, שולח דרך Pipe לB

  4. בונוס: שנה כך שB גם מסנן מספרים זוגיים בלבד לפני שליחה לC

Shared Memory — Value & Array

הדרך המהירה ביותר לשיתוף נתונים בין processes — אבל דורשת סנכרון ידני. ללא Lock = Race Condition בדיוק כמו ב-threads!

from multiprocessing import Process, Value, Array, Lock

def increment(counter, lock):
    for _ in range(1000):
        with lock:
            counter.value += 1  # חייב lock!

counter = Value('i', 0)  # integer משותף
lock    = Lock()

procs = [Process(target=increment, args=(counter, lock))
         for _ in range(4)]

for p in procs: p.start()
for p in procs: p.join()

print(counter.value)  # צפוי: 4000
      
# Array — מערך משותף
arr = Array('d', [1.0, 2.0, 3.0])
# 'd' = double, 'i' = int, 'c' = char

Type Codes

'i' = int   'd' = double
'f' = float   'c' = char
'b' = signed byte

⚠️ בלי Lock = בעיה

הסר את ה-lock מהקוד — הרץ. תקבל פחות מ-4000. זה Race Condition בין processes.

SharedMemory (Python 3.8+)

לנתונים גדולים: multiprocessing.shared_memory.SharedMemory — מאפשר שיתוף numpy arrays!

מה יקרה בלי Lock?

def bad_increment(counter):
    for _ in range(10000):
        counter.value += 1  # ← ללא lock

counter = Value('i', 0)
procs = [Process(target=bad_increment, args=(counter,))

# ... start, join ...
print(counter.value)  # צפוי 30000 — מה יקרה?

ג שגיאה — Python לא מאפשר Value ללא Lock
ד מספר אקראי — יכול להיות גם יותר מ-30000

Signals — אינטרופטים לתהליכים

📱 אנלוגיה: התראת פוש — ה-OS "דופק על הדלת" של process בכל רגע. Process יכול להחליט מה לעשות.

SIGTERM (15) — בקשה לסגירה מסודרת
SIGKILL (9) — הרג מיידי (לא ניתן לתפוס!)
SIGINT (2) — Ctrl+C מהמשתמש
SIGUSR1/2 — signal מותאם אישית
SIGCHLD — child process סיים
import signal, time, os

def handle_sigusr1(signum, frame):
    print("📡 קיבלתי SIGUSR1!")

# רישום handler
signal.signal(signal.SIGUSR1, handle_sigusr1)

print(f"PID: {os.getpid()}")
print("ממתין לsignal...")
signal.pause()  # ישן עד signal
# מהטרמינל: kill -USR1 <PID>

Signal Handler

פונקציה שרצה כשה-OS שולח signal. מקבל (signum, frame). חייב להיות מהיר!

⚠️ SIGKILL

SIGKILL אי-אפשר לתפוס, לבטל, או להתעלם ממנו. OS הורג מיידית — ללא cleanup.

🔐 Signals בסייבר

Signal injection, process hijacking דרך signals, TOCTOU דרך SIGALRM — ווקטורי תקיפה אמיתיים.

מנגנוני סנכרון — Lock · Mutex · Semaphore

🔒

Lock = Mutex

mutual exclusion — רק process אחד בכל פעם. Mutex הוא הרעיון, Lock הוא המימוש.

lock = Lock()
with lock:
    counter.value += 1  # רק אחד כאן
📢

Event

דגל go/stop משותף. Process A מחכה ב-wait(), Process B מאותת ב-set().

ev = Event()
ev.wait()  # Process A: ממתין
ev.set()   # Process B: go!
🚦

Semaphore — connection pool

מגביל ל-N processes בו-זמנית. מונה פנימי: acquire() מקטין, release() מגדיל. כשמונה=0 — חוסם.

db_sem = Semaphore(3)  # רק 3 ב-DB

def query_db(wid, sem):
    print(f"[W{wid}] ממתין...")
    with sem:              # acquire()
        print(f"[W{wid}] 🟢 מחובר")
        time.sleep(random.uniform(1,3))
    # release() — פינה לאחר

# 6 workers, מקסימום 3 בו-זמנית
for i in range(6):
    Process(target=query_db,
            args=(i, db_sem)).start()
פלט: W0/W1/W2 נכנסים → W3/W4/W5 חוסמים → כשW0 יוצא W3 נכנס

⚠️ מתי מה?

Lock/Mutex = critical section, רק 1
Semaphore(N) = resource pool, עד N
RLock = Mutex שאותו thread יכול לנעול שוב
Event = תיאום "התחל עכשיו"

// מושג

Mutex vs RLock — מה ההבדל?

🔑

Mutex = Lock()

Mutex (Mutual Exclusion) הוא הרעיון: רק אחד בכל פעם. ב-Python, Lock() הוא Mutex. בעיה: אם thread קורא לאותו Lock פעמיים — Deadlock!

lock = Lock()

def outer():
    with lock:      # acquire #1
        inner()     # קורא inner...

def inner():
    with lock:      # acquire #2
        ...         # 💀 DEADLOCK!
ויזואליזציה — Race Condition ללא Mutex: 
# ללא Mutex — counter = 5:
# T1 קורא  → 5
# T2 קורא  → 5   ← אותו ערך!
# T1 כותב  → 6
# T2 כותב  → 6   ← אבד +1!

# עם Mutex:
# T1: נועל→קורא 5→כותב 6→משחרר
# T2: נועל→קורא 6→כותב 7→משחרר ✓
🔄

RLock — Reentrant Mutex

Reentrant = אותו thread יכול לנעול שוב ושוב ללא deadlock. חייב לשחרר אותו מספר פעמים שנעל.

import threading

rlock = threading.RLock()

def outer():
    with rlock:      # acquire #1
        print("outer")
        inner()       # לא deadlock!

def inner():
    with rlock:      # acquire #2 ✓
        print("inner")
    # release #2
# release #1

threading.Thread(target=outer).start()

📊 השוואה מהירה

מנגנון N בו-זמנית Reentrant
Lock / Mutex 1 ✗
RLock 1 ✓
Semaphore(N) N ✗

🔐 סייבר: TOCTOU — תקיפה שמנצלת את החלון בין בדיקת תנאי לביצועו, לפני שה-Mutex נרכש.

conn.send() שולח אובייקט Python דרך ה-Pipe Pipe ▶
conn.send(obj)

שולח אובייקט Python כלשהו דרך ה-Pipe. האובייקט עובר סריאליזציה אוטומטית ב-pickle, כלומר ניתן לשלוח list, dict, מחלקות, וכו'.

// פרמטרים

// דוגמה
conn.send("hello") conn.send([1, 2, 3]) conn.send({"key": "value"})
⚠️ send() עם אובייקט שלא ניתן ל-pickle יזרוק PicklingError.
conn.recv() מקבל אובייקט מה-Pipe — חוסם עד שמגיע נתון Pipe ▶
conn.recv()

מקבל ומחזיר אובייקט שנשלח דרך ה-Pipe. אם אין נתון ממתין — חוסם (blocking) עד שיגיע. אם הצד השני סגר — זורק EOFError.

// פרמטרים
— — אין פרמטרים. תמיד מקבל את האובייקט הבא בתור (FIFO).
↩ מחזיר:האובייקט שנשלח — כפי שנשלח, אחרי deserialization מ-pickle.
// דוגמה
try: msg = conn.recv() # חוסם עד קבלה except EOFError: # הצד השני סגר את ה-Pipe print("Pipe נסגר")
conn.poll() בדיקה non-blocking האם יש נתון ממתין ב-Pipe Pipe ▶
conn.poll(timeout=0.0)

בודק האם יש נתון ממתין ב-Pipe מבלי לחסום. שימושי כשרוצים לבדוק ולהמשיך בעבודה אחרת אם אין נתון.

// פרמטרים
timeout float 0.0 (ברירת מחדל) = non-blocking. None = חוסם עד שיגיע נתון. מספר = ממתין עד N שניות.
↩ מחזיר:True אם יש נתון ממתין, False אם לא.
// דוגמה
if conn.poll(timeout=0.5): # ממתין 0.5 שנ msg = conn.recv() print(f"קיבלתי: {msg}") else: print("אין הודעה עדיין")
os.mkfifo() יוצר Named Pipe (FIFO) בפסיסטם Pipe ▶
os.mkfifo(path, mode=0o666)

יוצר Named Pipe — קובץ מיוחד בפסיסטם שמתפקד כ-Pipe. כל process שיודע את ה-path יכול להתחבר, גם אם אינו קשור (parent/child).

// פרמטרים
path str הנתיב בפסיסטם לקובץ ה-FIFO. לדוגמה "/tmp/my_pipe". חייב להיות נתיב ייחודי.
mode int הרשאות הקובץ באוקטל. 0o666 = קריאה וכתיבה לכולם (subject to umask).
↩ מחזיר:None. יוצר את הקובץ בפסיסטם.
// דוגמה
os.mkfifo("/tmp/ipc_pipe") # פתיחה חוסמת עד שהצד השני נפתח! with open("/tmp/ipc_pipe", 'w') as f: f.write("data\n") os.unlink("/tmp/ipc_pipe") # ניקוי
🔐 סייבר: Named Pipes ב-Windows ידועים כווקטור privilege escalation. תמיד בדוק הרשאות (CVE-2021-1732).
Queue() יוצר תור הודעות process-safe לתקשורת בין processes Queue ▶
Queue(maxsize=0)

יוצר תור FIFO שניתן לשיתוף בין processes. בניגוד ל-queue.Queue הרגיל — עובד דרך Pipe פנימי עם lock ו-semaphore, כך שבטוח לשימוש בין processes שונים.

// פרמטרים
maxsize int גודל מקסימלי של התור. 0 (ברירת מחדל) = גודל בלתי מוגבל. אם התור מלא — put() חוסם.
↩ מחזיר:אובייקט Queue מוכן לשימוש.
// דוגמה
q = Queue(maxsize=10) # מקסימום 10 פריטים q2 = Queue() # ללא מגבלה
q.put() מכניס פריט לתור — חוסם אם התור מלא Queue ▶
q.put(item, block=True, timeout=None)

מכניס פריט לסוף התור. האובייקט עובר pickle אוטומטית לפני שמועבר בין processes.

// פרמטרים
item any הפריט להכניס. חייב להיות picklable.
block bool True (ברירת מחדל) = חסום אם התור מלא. False = זרוק Full מיד.
timeout float None = חסום ללא מגבלה. מספר = ממתין N שניות, אחר כך זורק queue.Full.
// דוגמה
q.put("task_1") q.put({"id": 1}, timeout=3) # ממתין 3 שנ q.put_nowait("urgent") # = put(block=False)
q.get() מוציא ומחזיר פריט מהתחלת התור — חוסם אם ריק Queue ▶
q.get(block=True, timeout=None)

מוציא את הפריט הראשון מהתור (FIFO). אם התור ריק — חוסם עד שמגיע פריט.

// פרמטרים
block bool True = חסום אם ריק. False = זרוק queue.Empty מיד.
timeout float None = חסום ללא מגבלה. מספר = ממתין N שניות, אחר כך queue.Empty.
↩ מחזיר:הפריט שהוצא מהתור, אחרי deserialization.
// דוגמה
item = q.get() # חוסם item = q.get(timeout=5) # ממתין 5 שנ item = q.get_nowait() # = get(block=False)
חובה לקרוא ל-task_done() אחרי עיבוד אם משתמשים ב-q.join()!
q.task_done() מודיע שהפריט שהוצא מהתור עובד — חובה עם q.join() Queue ▶
q.task_done()

מודיע לתור שפריט שהוצא ב-get() עובד בהצלחה. כל קריאה ל-get() חייבת להיות מלווה בקריאה ל-task_done(). נדרש כדי ש-q.join() ידע מתי לשחרר.

// דוגמה
while True: item = q.get() process(item) q.task_done() # חובה!
⚠️ קריאה ל-task_done() יותר פעמים מאשר get() תזרוק ValueError.
Value() יוצר משתנה סקלרי בזיכרון משותף בין processes Shared Mem ▶
Value(typecode_or_type, *args, lock=True)

יוצר משתנה בודד (int, float, וכו') בזיכרון שמשותף לכל ה-processes. גישה לערך דרך .value. כברירת מחדל מוגן ב-Lock פנימי.

// פרמטרים
typecode str/type 'i'=int, 'd'=double, 'f'=float, 'b'=byte, 'c'=char. או ctypes type.
*args any ערך התחלתי. לדוגמה Value('i', 0) יוצר int עם ערך 0.
lock bool True = יוצר Lock פנימי אוטומטי. False = ללא הגנה (מסוכן!). אפשר לתת Lock קיים.
// דוגמה
counter = Value('i', 0) # int = 0 pi = Value('d', 3.14) # double with counter.get_lock(): counter.value += 1 # גישה בטוחה
⚠️ גם עם lock=True, פעולות כמו += אינן אטומיות! תמיד עטוף ב-with counter.get_lock().
Array() יוצר מערך בזיכרון משותף בין processes Shared Mem ▶
Array(typecode_or_type, size_or_initializer, lock=True)

יוצר מערך של ערכים בזיכרון משותף. מהיר מאוד — הנתונים ישבים בזיכרון אחד שכולם רואים, ללא העתקות.

// פרמטרים
typecode str סוג האיברים: 'i'=int, 'd'=double, 'f'=float, 'c'=char.
size_or_initializer int/list מספר = גודל המערך (0 התחלתי). רשימה = ערכי אתחול ומגדיר את הגודל.
// דוגמה
arr = Array('i', [1, 2, 3, 4]) arr = Array('d', 10) # 10 doubles, = 0.0 arr[0] = 42 # גישה רגילה print(list(arr)) # [42, 0, 0, ...]
signal.signal() רושם handler — פונקציה שתרוץ כשמגיע signal Signal ▶
signal.signal(signalnum, handler)

מגדיר פונקציה שתיקרא אוטומטית כשה-OS שולח signal מסוים ל-process. הפונקציה מקבלת שני ארגומנטים: מספר ה-signal ו-frame (context ריצה).

// פרמטרים
signalnum int/signal ה-signal לטפל בו: signal.SIGTERM, signal.SIGUSR1, signal.SIGINT, וכו'. לא ניתן לתפוס SIGKILL או SIGSTOP.
handler callable פונקציה שמקבלת (signum, frame). או signal.SIG_DFL (ברירת מחדל) או signal.SIG_IGN (התעלמות).
// דוגמה
import signal def handle_term(signum, frame): print("מקבל SIGTERM — מתנקה...") cleanup() exit(0) signal.signal(signal.SIGTERM, handle_term) signal.signal(signal.SIGINT, signal.SIG_IGN) # מתעלם מCtrl+C
⚠️ Handler חייב להיות קצר ומהיר. אסור לקרוא ל-print(), lock(), ופונקציות blocking בתוכו — זה לא thread-safe!
os.kill() שולח signal לprocess לפי PID Signal ▶
os.kill(pid, sig)

שולח signal ל-process לפי מספר הזיהוי שלו (PID). למרות השם, לא תמיד "הורג" — תלוי ב-signal שנשלח.

// פרמטרים
pid int מזהה ה-process. os.getpid() = ה-PID שלי. p.pid = ה-PID של Process אובייקט.
sig int/signal ה-signal לשלוח. signal.SIGTERM=15, signal.SIGKILL=9, signal.SIGUSR1=10.
// דוגמה
import os, signal p = Process(target=worker) p.start() os.kill(p.pid, signal.SIGUSR1) # שלח SIGUSR1 os.kill(p.pid, signal.SIGTERM) # בקש סגירה
Lock() יוצר נעילה בין processes — mutual exclusion Sync ▶
multiprocessing.Lock()

יוצר Lock שעובד בין processes (לא רק threads!). כל process שרוצה להיכנס ל-critical section חייב לרכוש את ה-Lock קודם. רק process אחד יכול להחזיק את ה-Lock בכל רגע.

// שיטות עיקריות
acquire(block, timeout) — רכוש את ה-Lock. חוסם עד שזמין. עם timeout לא יחכה לנצח.
release() — שחרר את ה-Lock. חובה לשחרר! אחרת deadlock.
// דוגמה
lock = Lock() with lock: # acquire + release אוטומטי counter.value += 1 # ידני: lock.acquire() try: counter.value += 1 finally: lock.release() # תמיד ישתחרר!
💡 תמיד העדף with lock: על פני acquire/release ידני — מבטיח שחרור גם במקרה של שגיאה.
Semaphore() מגביל גישה ל-N processes בו-זמנית Sync ▶
Semaphore(value=1)

כמו Lock, אבל מאפשר ל-N processes לגשת למשאב בו-זמנית. שומר מונה פנימי — acquire() מקטין ב-1, release() מגדיל ב-1. כשהמונה = 0, acquire() חוסם עד שמישהו קורא release().

// פרמטרים
value int ערך התחלתי של המונה = מספר processes שיכולים לגשת במקביל. 1 = זהה ל-Lock. 0 = חסום מיד.
// שיטות
acquire(block, timeout) bool מקטין מונה ב-1. אם מונה=0 וblock=True — חוסם. מחזיר True אם הצליח.
release() — מגדיל מונה ב-1 ומעיר process ממתין אחד.
// דוגמה מלאה — connection pool
from multiprocessing import Process, Semaphore import time, random db_sem = Semaphore(3) # מגביל ל-3 חיבורי DB def query_db(worker_id, sem): print(f"[W{worker_id}] ממתין לחיבור...") with sem: # acquire() print(f"[W{worker_id}] 🟢 מחובר!") time.sleep(random.uniform(1, 3)) # מדמה query print(f"[W{worker_id}] סיים") # release() אוטומטי — פינה מקום לאחר # 6 processes, אבל רק 3 יחד ב-DB procs = [Process(target=query_db, args=(i, db_sem)) for i in range(6)] for p in procs: p.start() for p in procs: p.join()
// פלט
# כולם מנסים, רק 3 נכנסים בו-זמנית: [W0] 🟢 מחובר! [W1] 🟢 מחובר! [W2] 🟢 מחובר! # sem=0, W3/4/5 חוסמים [W0] סיים [W3] 🟢 מחובר! # W0 שוחרר → W3 נכנס
💡 שימוש קלאסי: connection pool לDB, הגבלת בקשות API, ניהול worker pool.
⚠️ בניגוד ל-Lock, process שלא עשה acquire() יכול לקרוא release() — אם צריך "ownership" אמיתי השתמש ב-Mutex.
Event() דגל סנכרון — process אחד "מאותת", השאר ממתינים Sync ▶
multiprocessing.Event()

מנגנון סנכרון פשוט: Event הוא דגל (True/False) שמשותף לכל ה-processes. Process שממתין ב-wait() יעצור עד שמישהו יקרא ל-set().

// שיטות עיקריות
set() — מגדיר את הדגל ל-True ומעיר את כל ה-processes שמחכים ב-wait().
wait(timeout) — ממתין עד שהדגל = True. אם כבר True — חוזר מיד. timeout = מספר שניות מקסימום.
clear() — מאפס את הדגל ל-False. processes שיקראו wait() יחכו שוב.
is_set() bool בודק האם הדגל מוגדר. True = set() נקרא ולא clear() אחריו.
// דוגמה
ev = Event() def worker(ev): ev.wait() # ממתין ל-"go" start_work() def manager(ev): prepare() ev.set() # "go!" לכולם
Mutex / RLock() נעילה בלעדית עם ownership — רק מי שנעל יכול לשחרר Sync ▶
threading.RLock()  // Reentrant Lock

Mutex (Mutual Exclusion) הוא רעיון, לא מחלקה ספציפית ב-Python. Lock() רגיל הוא Mutex פשוט — רק process/thread אחד מחזיק אותו. RLock() הוא Mutex עם שני שיפורים: (1) Ownership — רק מי שרכש יכול לשחרר. (2) Reentrant — אותו thread יכול לרכוש שוב ושוב ללא deadlock.

// Mutex vs Lock vs Semaphore
Lock() = Mutex Mutex פשוט. רק 1 בכל פעם. שים לב: ב-Python, thread אחר יכול לשחרר Lock שרכש thread אחר — זה לא ownership אמיתי!
RLock() Reentrant Mutex עם ownership. רק thread שרכש יכול לשחרר. אותו thread יכול לרכוש N פעמים — חייב לשחרר N פעמים.
Semaphore(1) ≈ Mutex פונקציונלית זהה ל-Lock, אבל ללא ownership. כל thread יכול לקרוא release() — זה מסוכן לעיתים.
// דוגמה — RLock (Reentrant Mutex)
import threading rlock = threading.RLock() def outer(): with rlock: # acquire #1 print("outer נועל") inner() # קורא inner שגם נועל def inner(): with rlock: # acquire #2 — לא deadlock! print("inner נועל בתוך outer") # release #2 כאן # release #1 כאן # עם Lock() רגיל — deadlock! # עם RLock() — עובד תקין threading.Thread(target=outer).start()
// דוגמה — Lock() כ-Mutex עם multiprocessing
from multiprocessing import Process, Lock, Value # Mutex לגישה בטוחה ל-counter משותף mutex = Lock() counter = Value('i', 0) def safe_increment(mutex, counter, n): for _ in range(n): with mutex: # Mutex — רק אחד בכל פעם counter.value += 1 procs = [Process(target=safe_increment, args=(mutex, counter, 1000)) for _ in range(4)] for p in procs: p.start() for p in procs: p.join() print(counter.value) # תמיד 4000 ✓
// השוואה ויזואלית: מה קורה בלי Mutex
# ללא Mutex — Race Condition: # T1 קורא counter = 5 # T2 קורא counter = 5 ← אותו ערך! # T1 כותב counter = 6 # T2 כותב counter = 6 ← אבד increment אחד! # עם Mutex: # T1 נועל → קורא 5 → כותב 6 → משחרר # T2 נועל → קורא 6 → כותב 7 → משחרר ✓
🔐 סייבר: Mutex vulnerabilities — TOCTOU (Time Of Check To Time Of Use) היא מתקפה שמנצלת את החלון בין בדיקת תנאי לביצועו, לפני שה-Mutex נרכש. תמיד בדוק ורכוש בתוך אותה פעולה אטומית.
Process() יוצר process חדש — תהליך עצמאי עם זיכרון נפרד Process ▶
Process(target=None, args=(), kwargs={}, name=None, daemon=None)

יוצר process חדש שירוץ במקביל. לאחר יצירה — חייב לקרוא ל-start(). הprocess יחיה בזיכרון נפרד לחלוטין מה-parent.

// פרמטרים
target callable הפונקציה שתרוץ בprocess החדש. שם הפונקציה ללא סוגריים.
args tuple ארגומנטים positional לפונקציה. חייב להיות tuple — גם לארגומנט יחיד: (val,)

// דוגמה
p = Process(target=worker, args=(q,)) p.start() # מתחיל לרוץ p.join() # ממתין לסיום p.terminate() # הורג (SIGTERM) p.kill() # הורג (SIGKILL) print(p.pid) # מזהה ה-process
