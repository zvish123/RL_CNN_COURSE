3.1 — דרישות פונקציונליות

רשימה מצומצמת של פיצ'רים. לוחצים על כל שורה כדי להרחיב ולראות פירוט מלא: הסבר כללי, קלט, תהליך, פלט וטיפול בשגיאות.

FR-01אימות משתמש (Login) להרחבה ▾לסגירה ▴
הסבר כללי

משתמש מזדהה בשם משתמש וסיסמה כתנאי לכל פעולה אחרת במערכת. ההזדהות מייצרת session token שמלווה את כל הבקשות הבאות.

קלט

שם משתמש (מחרוזת), סיסמה (מחרוזת), כתובת IP של הלקוח (נאספת אוטומטית ע"י השרת).

תהליך
  • השרת מחפש את המשתמש בטבלת המשתמשים
  • מחשב hash על הסיסמה שהתקבלה ומשווה ל-hash השמור
  • בהצלחה — מנפיק session token ושומר אותו עם זמן תפוגה
  • בכל מקרה (הצלחה/כישלון) — נרשם אירוע ב-Logger
פלט

בהצלחה: session token + הודעת אישור. בכישלון: הודעת שגיאה גנרית (ללא ציון אם הבעיה היא שם המשתמש או הסיסמה).

טיפול בשגיאות
  • שם משתמש לא קיים → אותה הודעת שגיאה כמו סיסמה שגויה
  • ניסיונות כושלים חוזרים → מועבר ל-AnomalyDetector (ראו דף אנומליות)
  • שדה ריק/חסר → שגיאת קלט מפורשת לפני שליחה לשרת
FR-02העלאת קובץ לכספת להרחבה ▾לסגירה ▴
הסבר כללי

משתמש מאומת בוחר קובץ במחשבו ושולח אותו לשמירה מוצפנת בצד השרת, תחת תיקייה אישית.

קלט

session token תקף, תוכן הקובץ (bytes), שם קובץ, גודל קובץ.

תהליך
  • אימות תקינות ה-session token
  • בדיקת גודל קובץ מול מגבלה מוגדרת מראש
  • הצפנת תוכן הקובץ באמצעות CryptoService
  • כתיבה לדיסק בתיקיית המשתמש בלבד (חסימת path traversal)
פלט

אישור הצלחה עם שם הקובץ שנשמר, או הודעת שגיאה מתאימה.

טיפול בשגיאות
  • קובץ גדול מהמגבלה → דחייה לפני תחילת ההעברה
  • נתיב שם קובץ חשוד (מכיל ../) → דחייה ורישום אנומליה
  • החיבור נופל בזמן העברה → מחיקת קובץ חלקי, לא לשמור שאריות
FR-03הורדת קובץ מהכספת להרחבה ▾לסגירה ▴
הסבר כללי

משתמש מאומת מבקש לקבל קובץ ששמור בכספת שלו, והמערכת שולחת אותו בחזרה מפוענח.

קלט

session token תקף, שם/מזהה הקובץ המבוקש.

תהליך
  • בדיקת הרשאת המשתמש לקובץ הספציפי
  • קריאת הקובץ המוצפן מהדיסק
  • פענוח ושליחה ללקוח בהודעת פרוטוקול מתאימה
פלט

תוכן הקובץ (מפוענח) בצד הלקוח, או הודעת שגיאה.

טיפול בשגיאות
  • קובץ לא קיים → הודעת "קובץ לא נמצא" (לא לחשוף מבנה תיקיות)
  • בקשה לקובץ של משתמש אחר → דחייה + רישום privilege violation
FR-04חיבור מרובה לקוחות בו-זמנית להרחבה ▾לסגירה ▴
הסבר כללי

השרת מטפל במספר לקוחות מחוברים במקביל, ללא תלות הדדית — תקלה בלקוח אחד לא משפיעה על האחרים.

קלט

חיבור socket נכנס חדש.

תהליך
  • קבלת חיבור TCP חדש ב-VaultServer
  • פתיחת thread נפרד עבור הלקוח (ClientHandler)
  • הוספת הלקוח לרשימת הלקוחות המחוברים
פלט

חיבור פעיל ורשום, מוכן לקבל הודעות מהלקוח.

טיפול בשגיאות
  • לקוח מתנתק באמצע פעולה → סגירת ה-thread שלו בלבד, ניקוי משאבים, השרת ממשיך לרוץ
  • חיבורים רבים מאוד מ-IP יחיד → flag כ-anomaly (connection flooding)
FR-05זיהוי והתראה על אנומליה להרחבה ▾לסגירה ▴
הסבר כללי

כל אירוע במערכת (login, גישה לקובץ, חבילת רשת) נבדק מול חוקים מוגדרים מראש; אירוע שעובר סף מסוים מסומן כאנומליה ומוצג למנהל.

קלט

רצף אירועים מה-Logger (timestamp, IP, משתמש, סוג פעולה).

תהליך
  • ספירת אירועים בחלון זמן נגלל (sliding window) לפי IP/משתמש
  • השוואה ל-threshold המוגדר לאותו סוג אירוע
  • במקרה חריגה — יצירת רשומת התראה + קביעת רמת חמורה (severity)
פלט

רשומת התראה בלוג + הצגה בזמן אמת בדשבורד המנהל.

טיפול בשגיאות
  • אירוע לא תקין/חסר שדות → מתעלם ונרשם כשגיאת פורמט, לא קורס את ה-Detector
FR-06דשבורד ניטור למנהל להרחבה ▾לסגירה ▴
הסבר כללי

מסך נפרד למשתמש מנהל, המציג משתמשים מחוברים כעת ורשימת התראות אנומליה מתעדכנת.

קלט

session token עם הרשאת admin.

תהליך
  • בדיקת הרשאת admin לפני הצגת המסך
  • שליפת רשימת לקוחות מחוברים מה-VaultServer
  • שליפת התראות אחרונות מה-Logger/EventStore
פלט

טבלת משתמשים מחוברים + טבלת התראות עם רמת חמורה.

טיפול בשגיאות
  • משתמש לא-admin שמנסה לגשת → דחייה + רישום privilege violation
הרחבה לצוותים מתקדמים: ניתן (ואף רצוי) להציג חלק מהדרישות הפונקציונליות בפורמט תרחיש שימוש (use case) — בכל נקודת הסתעפות לפרט את כל האפשרויות העומדות בפני המשתמש/המערכת, כולל מקרים שליליים (שגיאות, קלט לא תקין), ולא רק את "מסלול האושר".

3.2 — דרישות של ממשקים חיצוניים

3.2.1 ממשקי משתמש (GUI)

שני ממשקים נפרדים: ממשק משתמש קצה (login, רשימת קבצים, העלאה/הורדה) וממשק מנהל (דשבורד ניטור). פירוט מלא ומוקאפים — בדף דרישות UI/UX.

3.2.2 ממשקי חומרה

אין דרישת חומרה מיוחדת מעבר לכרטיס רשת תקני. אם צוות בוחר לשלב רכיב מערכת הפעלה נוסף (מצלמה/מיקרופון/קובץ מקומי) — יש לתעד זאת כאן בנפרד.

3.2.3 ממשקי תוכנה

המערכת מחולקת לשתי תוכנות נפרדות שמתקשרות ביניהן: תוכנת שרת (VaultServer) ותוכנת לקוח (VaultClient). שתיהן בפייתון, ומתקשרות אך ורק באמצעות פרוטוקול ההודעות המוגדר (ProtocolMessage) — לא דרך גישה משותפת לזיכרון או לקובץ.

3.2.4 ממשקי תקשורת

תקשורת TCP בין שרת ללקוחות, פורט קבוע מוגדר מראש. כל ההודעות עוברות בפרוטוקול אפליקטיבי מותאם (JSON/Header+Body) מעל ה-socket, עם הצפנה על השכבה האפליקטיבית (לא TLS חיצוני).

3.3 — דרישות לא פונקציונליות

3.3.1 ביצועים

זמן תגובה

פעולת login/הורדת קובץ קטן מגיבה למשתמש תוך 1-2 שניות בתנאי רשת תקינים.

3.3.2 מהימנות

חוסן לתקלות

נפילת לקוח אחד (ניתוק לא תקין) לא גורמת לקריסת השרת או להשפעה על לקוחות אחרים.

3.3.3 זמינות

תמיכה בעומס

השרת תומך בלפחות 5 לקוחות מחוברים בו-זמנית בלי האטה נראית לעין.

3.3.4 אבטחה

הצפנה ובקרה

סיסמאות נשמרות hash+salt בלבד; תעבורה רגישה מוצפנת; כל פעולה רגישה נרשמת ב-log לצורך ביקורת.

3.3.5 תחזוקתיות

קוד ניתן להרחבה

הוספת סוג אנומליה חדש או סוג הודעה חדש בפרוטוקול לא מצריכה שינוי במחלקות קיימות, רק הוספת קוד.

3.3.6 ניידות

פלטפורמה

המערכת רצה על כל מחשב עם Python 3 מותקן (Windows/Linux/macOS), בלי תלות בספריות ספציפיות-מערכת הפעלה.

3.4 — דרישות בסיס נתונים

נושאפירוט
אילו נתונים יישמרומשתמשים (username, password_hash, salt, הרשאות), מטא-דאטה של קבצים (שם, בעלים, גודל, תאריך), לוג אירועים (timestamp, IP, משתמש, סוג פעולה), רשומות אנומליה והתראות
פורמט אחסוןקובץ JSON/SQLite למשתמשים ולוגים (לבחירת הצוות); קבצי הכספת עצמם נשמרים מוצפנים בדיסק בתיקייה ייעודית לכל משתמש
יכולות אחסון נדרשותמספיק לסביבת בדיקה/הדגמה בכיתה — אין צורך בקנה-מידה תעשייתי; להעריך לפי גודל קבצי הדמו (סדר גודל MB בודדים עד עשרות MB)
רמת אבטחה נדרשתסיסמאות לעולם לא בטקסט גלוי; קבצים רגישים מוצפנים at-rest; לוגים נגישים רק למשתמש admin

3.5 — דרישות נוספות

כל דרישה שמתגלה במהלך הפיתוח ולא משויכת בבירור לאחד מהסעיפים לעיל — לדוגמה: קובץ קונפיגורציה חיצוני לערכי ה-threshold של האנומליות, או צורך בכלי עזר לניתוח לוגים — מתועדת כאן, עם הסבר קצר מדוע היא נדרשת.
