👥 זרימת עבודה בצוות

הבעיה: איך עובדים יחד?

תרחיש:

אתם צוות של 4 תלמידים שפיתחים אפליקציית מזג אויר:

  • אליס - עובדת על ממשק המשתמש
  • בוב - מפתח API לקבלת נתוני מזג אויר
  • צ'רלי - עובד על מסד הנתונים
  • דיינה - כותבת בדיקות

בעיות אפשריות:

  • שני אנשים עורכים את אותו הקובץ
  • קוד של מישהו מקלקל את העבודה של אחרים
  • איך לדעת מי שינה מה ומתי?
  • איך לחזור לגרסה קודמת שעבדה?

🌐 Repository מרוחק

Local Repo
מחשב אליס
↕
Remote Repo
GitHub/GitLab
↕
Local Repo
מחשב בוב

פקודות עיקריות:

חיבור מחשב מקומי ל GITHUB בענן

git remote add origin https://github.com/username/repository-name.git

העתקת הפרויקט מהשרת למחשב המקומי

git clone https://github.com/team/weather-app.git

העלאת השינויים לשרת המרוחק

git push origin main

הורדת עדכונים מהשרת המרוחק

git pull origin main

תרגיל מעשי:

צרו repository חדש ב-GitHub בשם "team-project" והזמינו את חברי הצוות כ-collaborators

🌳 ניהול ענפים (Branches)

למה צריך ענפים?

  • עבודה על תכונות שונות במקביל
  • בדיקת רעיונות חדשים מבלי להשפיע על הקוד הראשי
  • אפשרות לחזור לגרסה יציבה

פקודות ענפים:

יצירת ענף חדש בשם feature-ui

git branch feature-ui

מעבר לענף החדש

git checkout feature-ui

יצירה ומעבר לענף חדש בפקודה אחת

git checkout -b feature-api

הצגת כל הענפים (מקומיים ומרוחקים)

git branch -a

🔧 דוגמה מעשית - עבודה בענפים

תרחיש: אפליקציית To-Do

אליס עובדת על עיצוב החדש, בוב מוסיף תכונת התראות

אליס (ענף design):

git checkout -b design
git add styles.css
git commit -m "עיצוב חדש לרשימת משימות"

בוב (ענף notifications):

git checkout -b notifications
git add notifications.js
git commit -m "הוספת תכונת התראות"

שאלה להבנה:

מה יקרה אם שני המפתחים יעבדו על אותו קובץ באותו זמן?

A. Git ימחק את העבודה של אחד מהם
B. יווצר קונפליקט שנצטרך לפתור ידנית
C. Git ישלב את השינויים אוטומטית

🔀 מיזוג ענפים (Merge)

main branch
←
feature branch
→
merged main

תהליך המיזוג:

מעבר לענף הראשי

git checkout main

עדכון הענף הראשי

git pull origin main

מיזוג ענף התכונה לענף הראשי

git merge feature-ui

💡 טיפ מקצועי:

תמיד עדכנו את הענף הראשי לפני מיזוג כדי למנוע קונפליקטים מיותרים

⚔️ פתרון התנגשויות

מתי מתרחשות התנגשויות?

כאשר שני מפתחים משנים את אותו קובץ באותן השורות

גרסת אליס:

function calculateTotal() {
    return price * 1.17; // מע"מ 17%
}

גרסת בוב:

function calculateTotal() {
    return price * 1.18; // מע"מ 18%
}

כך נראה קונפליקט ב-Git:

<<<<<<< HEAD
return price * 1.17; // מע"מ 17%
=======
return price * 1.18; // מע"מ 18%
>>>>>>> feature-branch

🛠️ שלבי פתרון התנגשות

זיהוי קונפליקט
→
עריכת הקובץ
→
בדיקת הקוד
→
Commit

פקודות לפתרון קונפליקט:

בדיקת מצב הקונפליקט

git status

סימון הקובץ כפתור

git add conflicted-file.js

שמירת הפתרון

git commit -m "פתרון קונפליקט במחשבון מע\"מ"

תרגיל:

צרו קונפליקט מכוון (שני מפתחים עורכים את אותה השורה) ופתרו אותו

📥 Pull Requests

מה זה Pull Request?

בקשה לסקור ולמזג שינויים מענף אחד לענף אחר

עבודה על ענף
→
יצירת PR
→
סקירת קוד
→
מיזוג

יתרונות של Pull Requests:

  • סקירת קוד לפני מיזוג
  • דיון על השינויים
  • בדיקות אוטומטיות
  • תיעוד השינויים

תהליך יצירת PR:

  1. דחיפת הענף לשרת המרוחק
  2. יצירת Pull Request דרך GitHub/GitLab
  3. הוספת תיאור מפורט
  4. המתנה לאישור מחברי הצוות

🏆 תרגיל צוותי

פרויקט: מחשבון מתקדם

הצוות שלכם יפתח מחשבון עם התכונות הבאות:

אליס - UI

  • עיצוב הממשק
  • כפתורי המספרים
  • תצוגת התוצאות

בוב - Logic

  • פונקציות חשבון בסיסי
  • חשבון מתקדם
  • זיכרון המחשבון

משימות:

  1. צרו repository משותף
  2. כל אחד יעבוד בענף נפרד
  3. בצעו commits קטנים ותכופים
  4. צרו Pull Requests לסקירה
  5. פתרו קונפליקטים במידת הצורך

⭐ שיטות עבודה מומלצות

הודעות Commit איכותיות:

git commit -m "הוסף תכונת חישוב אחוזים למחשבון"

✅ תיאור ברור ומדויק

git commit -m "תיקון"

❌ לא מספיק מידע

מבנה הודעת commit טובה:

  • התחילו בפועל (הוסף, תקן, עדכן)
  • תיארו מה השתנה, לא איך
  • השתמשו בעברית או אנגלית באופן עקבי
  • הגבילו ל-50 תווים בשורה הראשונה

תדירות Commits:

  • בצעו commit בכל פעם שמסיימים משימה קטנה
  • לא חובה שהקוד יהיה מושלם
  • עדיף הרבה commits קטנים מ-commit אחד גדול

🚨 פתרון בעיות נפוצות

בעיה: "שכחתי לעשות pull לפני push"

git pull origin main --rebase

פתרון: שלב את השינויים המקומיים עם השרת

בעיה: "רוצה לבטל את ה-commit האחרון"

git reset --soft HEAD~1

ביטול commit אבל השמירה על השינויים

בעיה: "הקוד שלי נעלם!"

git stash list
git stash pop

בדיקה והחזרת שינויים שנשמרו זמנית

Git שומר כמעט הכל - בדרך כלל אפשר לשחזר מידע "אבוד"

📝 בחן את עצמך

שאלה 1: איך תתחילו פרויקט חדש בצוות?

A. יוצרים repository ב-GitHub ואז clone לכולם
B. כל אחד יוצר repository משלו
C. עובדים עם Google Drive משותף

שאלה 2: מתי כדאי ליצור ענף חדש?

A. כשמתחילים לעבוד על תכונה חדשה
B. רק בסוף הפרויקט
C. אף פעם, עובדים רק על main

שאלה 3: מה עושים כשיש קונפליקט?

A. מוחקים את הקובץ ומתחילים מחדש
B. עורכים את הקובץ, מסירים את סימני הקונפליקט וממשיכים
C. מחכים שמישהו אחר יפתור

תרגיל סופי - חידון מהיר:

איזו פקודה תשתמשו בכל מצב?

רוצים לראות את כל הענפים:
git branch -a
רוצים להעלות ענף חדש לשרת:
git push -u origin branch-name
רוצים לבטל שינויים שלא נשמרו:
git checkout -- .
