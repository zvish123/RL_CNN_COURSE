תזכורת

מה זה class ומה זה object?

למדתם את זה בשיעור OOP - רגע תזכורת לפני שקופצים לפרויקט

class

תבנית שמתארת אילו data attributes ואילו methods יהיו לאובייקט. ה-class עצמו אינו אובייקט - הוא ה"מתאר".

object (instance)

מופע קונקרטי שנוצר מה-class בזיכרון. בפרויקט שלכם: השרת שרץ עכשיו הוא instance אחד של NetworkServer.

אז מה חדש בשלב הזה?

בשיעור למדנו לכתוב class בודד. כעת אנחנו עוברים שלב: לתכנן כמה מחלקות שעובדות בצוות - וזו בדיוק ההגדרה של ארכיטקטורת מערכת.

עיקרון 1

אחריות יחידה (Single Responsibility)

לכל מחלקה יש סיבה אחת להשתנות. אם תוכלו לתאר מחלקה רק במילה אחת או משפט קצר אחד - היא מתוכננת טוב.

✗ מחלקה אחת שעושה הכול

class VaultSystem:
  # מקבלת חיבורים
  # מצפינה סיסמאות
  # שומרת קבצים
  # מנתחת תעבורה חשודה
  # מציגה Dashboard

שינוי קטן בהצפנה = סיכון לשבור את ה-GUI. אי אפשר לבדוק (test) חלק אחד בלי כל השאר.

✓ חמש מחלקות, חמש אחריות

class NetworkServer: ...
class SecurityManager: ...
class StorageManager: ...
class PacketAnalyzer: ...
class DashboardUI: ...

כל מוביל בצוות יכול לעבוד על המחלקה שלו במקביל, ולבדוק אותה בנפרד.

בדיקת הבנה

אחריות אחת או שתיים?

מחלקה ששומרת קבצים מוצפנים לדיסק וגם שולחת התראת מייל כשמתגלה anomaly - זו אחריות יחידה?

כן, שתי הפעולות קשורות לאבטחה
לא - אחסון קבצים והתראות הן שתי אחריויות נפרדות, כדאי לפצל ל-StorageManager ול-AlertManager

טיפ: אם תיאור המחלקה כולל את המילה "וגם" - זה לרוב סימן שכדאי לפצל.

עיקרון 2

הפרדה בין שכבות (Layered Architecture)

קבוצות של מחלקות שמטפלות בתחום אחד, ולא "יודעות" יותר ממה שהן צריכות על שכבות אחרות.

תקשורת

Network Layer

קבלת חיבורים, ניהול threads, פרוטוקול הודעות.

אבטחה

Security Layer

הצפנה, אימות, ניתוח חשד.

אחסון

Storage Layer

שמירת קבצים, לוגים, נתוני משתמשים.

ממשק

UI Layer

Dashboard, התראות למשתמש.

למה זה חשוב?

אם ה-DashboardUI "יודע" איך עובד ה-hashing, כל שינוי באלגוריתם ההצפנה יחייב לגעת גם בקוד הממשק. שכבות מבודדות = שינוי בשכבה אחת לא מתפשט.

דוגמה

NetworkServer ↔ ClientHandler

קשר טיפוסי בין מחלקות: מחלקה אחת מחזיקה / יוצרת אובייקטים של מחלקה אחרת.

class NetworkServer:
  def __init__(self):
    self.clients = [] # רשימת ClientHandler

  def accept_connection(self, sock):
    handler = ClientHandler(sock) # יוצר אובייקט חדש
    self.clients.append(handler)
    handler.start() # מריץ ב-thread נפרד

מה קרה כאן?

  • NetworkServer לא "יודע" איך מצפינים סיסמה - הוא רק מנהל חיבורים.
  • ClientHandler אחד נוצר לכל לקוח מחובר - זה composition: ה-server "מחזיק" הרבה handlers.
  • כשלקוח מתנתק, ה-handler שלו נמחק - חיי ה-handler קשורים לחיי החיבור.
דוגמה

Protocol ↔ ProtocolMessage

לעיתים מחלקה אחת היא "מנהל כללים" ומחלקה שנייה היא "יחידת מידע" שעוברת בין הצדדים.

ProtocolMessage - data class

class ProtocolMessage:
  def __init__(self, msg_type, body):
    self.msg_type = msg_type
    self.body = body

מחלקה "דלה" שרק מארגנת מידע - LOGIN, FILE_REQUEST וכו'.

Protocol - מנהל כללים

class Protocol:
  def encode(self, msg: ProtocolMessage):
    ...
  def decode(self, raw_bytes):
    ... # מחזיר ProtocolMessage

Protocol "משתמש" ב-ProtocolMessage בלי לדעת מי שולח/מקבל אותה - זה dependency, לא composition.

קשרים בין מחלקות

ארבעה סוגי קשר שכדאי להכיר

סוג קשרמשמעותדוגמה מהפרויקט
Dependencyמחלקה A "משתמשת" במחלקה B דרך פרמטר/החזרה - בלי להחזיק אותה לאורך זמןProtocol משתמש ב-ProtocolMessage
Associationקשר רגיל ומתמשך: A מחזיקה reference ל-BClientHandler מחזיק SecurityManager
AggregationA "מכילה" Bים, אך הם יכולים להתקיים גם בלעדיהStorageManager מכיל קבצים שנשארים גם אם המנהל נמחק
CompositionA "בעלים" של B - כש-A נמחקת, B נמחק איתהNetworkServer ↔ ClientHandler (כשהשרת נסגר, כל ה-handlers נסגרים)

סוג הקשר משפיע על סדר היצירה וההריסה בקוד שלכם: אם ClientHandler הוא composition בתוך NetworkServer, סגירת השרת חייבת לסגור את כל ה-handlers (אחרת תישארו עם threads "תלויים" שלא נסגרים נכון).

איך קוראים תרשים

תיבת UML למחלקה בודדת

SecurityManager
- users: dict
- sessions: dict
+ hash_password(pw)
+ verify(user, pw)
+ create_session(user)

תיבת UML מחולקת לשלושה חלקים:

  • שם המחלקה - שורה עליונה
  • attributes - שורה אמצעית (- = פרטי, + = ציבורי)
  • methods - שורה תחתונה, אילו פעולות אפשר לבקש מהאובייקט

חוקיים תמיד: לציין type ככל שאפשר (dict, list, str) - זה מאלץ אתכם להחליט מבנה נתונים לפני שכותבים קוד.

חצים בין תיבות

──> (חץ פתוח) = dependency ──> (חץ מלא) = association ◇── = aggregation ◆── = composition ▷── (חץ משולש) = inheritance

God Object

מחלקה אחת ("VaultSystem") שעושה הכול. סימן אזהרה: קובץ אחד מכיל 5 classes כי "כך קל יותר".

סיכום שלב 3

מה תיקחו מכאן

  • אחריות יחידה - אם מחלקה צריכה "וגם" בתיאור שלה, כדאי לפצל.
  • שכבות - תקשורת, אבטחה, אחסון, ממשק. שכבה לא צריכה לדעת פרטי מימוש של שכבה אחרת.
  • UML - תיבה = מחלקה (שם / attributes / methods), חץ = קשר. סוג החץ אומר לנו מה קורה כשהאובייקטים נמחקים.
  • בשלב הבא תכתבו את שכבת התקשורת הבסיסית (TCP echo server) על בסיס התרשים שתכננתם כאן.

הצלחה בבניית הצוות - ותזכרו: כל אחד כותב קוד, אבל כולם צריכים להבין את כל המערכת (כי בהגנה האישית נשאלים על הכול).
