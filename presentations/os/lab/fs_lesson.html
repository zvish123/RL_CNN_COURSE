התיקיות

היררכיה, root, ומסלולים

// מושג

עץ הקבצים — הכל מתחיל מ-/

ב-Linux הכל מתחיל מ-/ (root). אין "C:\\" — רק עץ אחד, מוחלט.

/  ← root directory
├── bin/        ← binary executables (ls, cp, bash)
├── boot/       ← kernel + bootloader
├── dev/        ← device files (sda, null, random)
├── etc/        ← config files (passwd, hosts, ssh/)
├── home/
│   ├── alice/
│   │   ├── .bashrc    ← hidden (dot file)
│   │   ├── Documents/
│   │   └── Downloads/
│   └── bob/
├── proc/       ← virtual FS — process info
├── root/       ← home dir of root user
├── tmp/        ← temporary (deleted on reboot)
├── usr/
│   ├── bin/       ← user programs
│   └── lib/       ← shared libraries
└── var/
    └── log/       ← system logs ← חשוב לסייבר!

🔐 קשר לסייבר

/etc/passwd, /etc/shadow — קבצי משתמשים. /var/log/ — לוגים. /tmp/ — מקום נפוץ להנחת backdoors. /proc/ — מידע חי על processes.

מסלול מוחלט

מתחיל מ-/. מדויק תמיד.
/home/alice/docs/secret.txt

מסלול יחסי

יחסי לתיקייה הנוכחית.
../docs/secret.txt
./script.sh

Windows vs Linux

Windows: C:\Users\
Linux: /home/
Linux — הכל קובץ, כולל devices!

הסתרת קבצים

קובץ שמתחיל ב-. מוסתר ב-ls. חשוף עם ls -la. טכניקה ידועה להסתרת malware.

// שאלת הבנה #1

מסלולים — מה נכון?

המשתמש נמצא ב-/home/alice/Documents. הוא כותב:

$ cat ../Downloads/report.txt
א שגיאה — מסלול יחסי לא עובד עם cat
ב פותח את /home/alice/Downloads/report.txt
ג פותח את /home/Downloads/report.txt
ד פותח את /Downloads/report.txt
// השלם את הפקודה

🔍 מצא קבצים מוסתרים

אנליסט סייבר רוצה לראות כל הקבצים בתיקייה, כולל מוסתרים, עם הרשאות:

$ ls -
# פלט:
drwxr-xr-x  alice  alice   4096 Jan 15 .ssh/
-rw-------  alice  alice    256 Jan 12 .bash_history
-rw-r--r--  alice  alice   1234 Jan 10 notes.txt

💡 סייבר — מה לחפש?

.bash_history — היסטוריית פקודות. .ssh/ — מפתחות SSH. קבצי dot שלא מוכרים — חשודים! תמיד בצע ls -la בחקירה.

rwx · chmod · chown · SUID

// מושג

הרשאות — rwx

כל קובץ ב-Linux יש לו 3 קבוצות הרשאות × 3 סוגי הרשאה:

-rwxr-xr--  alice  staff  4096  file.txt
 ↑   Owner Group Others
 └ סוג קובץ: - = regular, d = dir, l = symlink
r
read
קריאה
= 4
w
write
כתיבה
= 2
x
execute
הרצה
= 1
# chmod — שינוי הרשאות
chmod 755 script.sh   # rwxr-xr-x
chmod 644 config.txt  # rw-r--r--
chmod +x script.sh    # הוסף execute לכולם
chmod g-w file.txt    # הסר write מGroup

# chown — שינוי בעלות
chown alice:staff file.txt
chown -R root /etc/ssh  # רקורסיבי

⚠️ SUID bit

קובץ עם SUID רץ עם הרשאות ה-Owner, לא המריץ. chmod 4755 = rwsr-xr-x. SUID על /bin/bash = privilege escalation!

💡 755 = הכי נפוץ

7=rwx (owner), 5=r-x (group), 5=r-x (others). Script שרוצים שכולם יריצו.

umask

מסכה שמורידה הרשאות ברירת מחדל. umask 022 → קבצים חדשים = 644, תיקיות = 755.

// פעילות אינטראקטיבית

🔐 מחשבון הרשאות

לחץ על הביטים כדי להדליק/לכבות הרשאות וראה את ייצוג האוקטל:

chmod 644 file.txt
-rw-r--r--
Owner: read+write | Group: read | Others: read

🔐 בדיקה: מה מסוכן?

נסה 777 — כולם יכולים לכתוב ולהריץ. נסה 600 — רק Owner. מה נכון ל-/etc/shadow?

// שאלת הבנה #2

הרשאות — תרחיש סייבר

אנליסט מוצא קובץ עם ההרשאות הבאות:

-rwsr-xr-x  root  root  /usr/bin/suspicious

מה מיוחד ומסוכן בהרשאות אלה?

א הקובץ שייך ל-root — זה רגיל ובסדר
ב ה-group ו-others לא יכולים לכתוב — בעיה
ג SUID bit פעיל (s במקום x) — כל משתמש שמריץ אותו מקבל הרשאות root
ד אין כאן שום בעיה, זה קובץ רגיל של המערכת
// משימה בזוגות

🔐 Permission Audit

בזוגות: בדקו את ה-permissions הבאים וסמנו אילו מהם מסוכנים ולמה.

  1. הריצו find / -perm -4000 2>/dev/null — מה מוצאים? למה זה חשוב?
  2. בדקו ls -la /etc/shadow — מה ההרשאות? למה הגיוני?
  3. מצאו קובץ עם chmod 777 בסיסטם — האם זה בסדר?
  4. בונוס: הסבירו מה עושה find / -writable -not -user $(whoami) 2>/dev/null

מה מסתתר מאחורי שם הקובץ

// מושג

Inode — תעודת הזהות של הקובץ

שם הקובץ הוא רק מדבקה. המידע האמיתי נמצא ב-inode — מבנה נתונים בדיסק.

// INODE #274823
file typeregular file (-)
permissionsrw-r--r-- (644)
owner UID1001 (alice)
group GID1001 (staff)
size4,096 bytes
hard links1
atime2024-01-15 09:23:11
mtime2024-01-10 14:05:33
ctime2024-01-10 14:05:33
data blocks→ [block 8192, 8193, ...]
$ stat secret.txt
  File: secret.txt
  Size: 4096
  Inode: 274823
  Access: 2024-01-15 09:23:11  # atime
  Modify: 2024-01-10 14:05:33  # mtime
  Change: 2024-01-10 14:05:33  # ctime

⏰ שלושת ה-timestamps

atime = last Access (קריאה)
mtime = last Modify (שינוי תוכן)
ctime = last Change (שינוי inode/metadata)

🔐 Timestamps בסייבר

תוקף עשוי לשנות timestamps עם touch -t. אבל ctime לא ניתן לשנות בקלות! השוואת timestamps = כלי forensics חשוב.

Hard Link vs Symlink

Hard link: שם נוסף לאותו inode. מחיקת אחד לא מוחקת הנתונים.
Symlink: קיצור דרך — קובץ שמצביע למסלול.

ls -i — מספר inode

ls -i file.txt מראה את מספר ה-inode. שני שמות עם אותו מספר = hard links לאותו קובץ.

// שאלת הבנה #3 — Forensics

חקירת Timestamps

חוקר מוצא קובץ עם הנתונים הבאים:

$ stat malware.exe
  Modify: 2020-03-15 08:00:00   # mtime — "ישן"
  Change: 2024-01-14 23:47:12   # ctime — אתמול!
  Access: 2024-01-14 23:47:15   # atime — אתמול!

מה מסקנת החוקר?

א הקובץ ישן ולא מסוכן — mtime משנת 2020
ב הקובץ הועבר/שונה אתמול — תוקף כנראה זייף את mtime כדי להסוות פעילות
ג ctime וatime לא רלוונטיים לחקירה
ד הקובץ נגעה בו רק קריאה, לא שינוי — בסדר
// סוגי קבצים

כל דבר ב-Linux הוא קובץ

לחץ על כל כרטיס לפרטים:

-
Regular File
קובץ רגיל — טקסט, binary, image. ls -la מראה - בהתחלה.
d
Directory
תיקייה — מכילה שמות קבצים וה-inode שלהם. גם היא קובץ!
l
Symlink
Symbolic link — קיצור דרך. מצביע למסלול אחר. יכול להצביע לקובץ שלא קיים.
b
Block Device
מכשיר בלוק — דיסק, USB. /dev/sda. גישה בבלוקים, עם buffer.
c
Char Device
מכשיר תווים — מקלדת, terminal. /dev/null, /dev/random.
s
Socket
Unix Domain Socket — תקשורת בין processes. /var/run/docker.sock.
p
Named Pipe
FIFO — IPC בין processes. ראינו בשיעור הקודם! גם הוא קובץ.
🔐
Cyber Note
/dev/null — מחיקת פלט. תוקפים משתמשים ב-2>/dev/null להסתיר שגיאות.

💡 ls -la — הסימן הראשון בשורה = סוג הקובץ

04
ניטור
ואבחון

find · stat · lsof · auditd

// כלי ניטור

כלי Forensics בסיסיים

find — חיפוש מתקדם

# קבצים שהשתנו ב-24 שעות אחרונות
find / -mtime -1 2>/dev/null

# קבצים עם SUID
find / -perm -4000 -type f 2>/dev/null

# קבצים ששייכים לאף אחד (orphaned)
find / -nouser 2>/dev/null

# קבצים גדולים מ-100MB
find / -size +100M -type f 2>/dev/null

# חיפוש לפי שם עם wildcard
find /tmp -name "*.sh" -type f

stat — מטא-דאטה מלא

$ stat /etc/passwd
  Inode: 1048577
  Access: 2024-01-15 09:00
  Modify: 2024-01-10 14:30
  Change: 2024-01-10 14:30

lsof — קבצים פתוחים

# מי פותח את הקובץ הזה?
lsof /var/log/auth.log

# כל הקבצים של process
lsof -p 1234

# network connections
lsof -i TCP

# קבצים שנמחקו אבל עדיין פתוחים
lsof | grep deleted

🔐 auditd — ניטור מתמשך

# עקוב אחר גישה לקובץ
auditctl -w /etc/passwd -p rwxa

# בדוק לוג auditd
ausearch -f /etc/passwd

# כל פעולת write על /etc/
auditctl -w /etc/ -p w -k etc_changes
// פעילות — התאם פקודה לתרחיש

🎯 איזו פקודה?

לחץ על תרחיש, אחר כך על הפקודה המתאימה:

// תרחישים
מצא כל קובץ שהשתנה אתמול
מי משתמש בקובץ /etc/passwd עכשיו
בדוק timestamps של קובץ חשוד
מצא קבצים עם SUID שעלולים לסכן
// פקודות
lsof /etc/passwd
find / -perm -4000
find / -mtime -1
stat suspicious.file
// השלם את הפקודה

🔍 כתוב פקודת find

חוקר רוצה למצוא כל הקבצים ב-/tmp עם סיומת .sh שהשתנו בשעה האחרונה:

find 
     -name 
     -mmin 
     -type f

💡 פרמטרים שימושיים של find

-mtime -1 = שינוי ב-24שע | -mmin -60 = שינוי בשעה | -name "*.sh" = שמות | -type f/d/l = סוג

// שאלת הבנה #4

lsof | grep deleted — למה חשוב?

חוקר מריץ lsof | grep deleted ומוצא:

malware  4523  root  3r  REG  8,1  0  /tmp/payload (deleted)

מה משמעות הממצא?

א הקובץ נמחק — אין מה לדאוג, הוא לא קיים יותר
ב Process 4523 מנסה לפתוח קובץ שלא קיים — שגיאה
ג הקובץ "נמחק" מהפסיסטם אבל עדיין בזיכרון — הנתונים ניתנים לשחזור ואפשר לקרוא אותם ב-/proc
ד זה תקין — OS מנקה קבצים ישנים אוטומטית
// משימת סיום קבוצתית

🕵️ CTF — File System Investigation

בקבוצות של 3: "פרצו" לשרת (VM/sandbox) ובצעו חקירת file system.

  1. מיפוי: הריצו ls -la /, ls -la /home/, ls -la /tmp/ — תעדו מה מוזר
  2. הרשאות: find / -perm -4000 2>/dev/null — האם יש SUID חשוד?
  3. Timestamps: find / -mtime -1 2>/dev/null — מה השתנה לאחרונה?
  4. קבצים פתוחים: lsof | grep -E "deleted|/tmp" — ממצאים?
  5. דוח: כתבו 3 משפטים — מה מצאתם, מה חשוד, מה המלצתכם
